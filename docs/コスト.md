## 1. GCS 料金体系

| 項目 | 単価 (USD) | 単価 (JPY, @¥150) | GCS分類 |
|------|-----------|-------------------|---------|
| **Class A** (list / put) | $0.05 / 10K ops | ¥7.5 / 10K ops | `list_blobs`, `upload` |
| **Class B** (get / metadata) | $0.004 / 10K ops | ¥0.6 / 10K ops | `download`, `exists`, `reload` |
| **ストレージ** (Standard) | $0.020 / GB / 月 | ¥3.0 / GB / 月 | 保存データ量 |
| **ネットワーク下り** (GCS → インターネット) | $0.12 / GB | ¥18 / GB | `load()` のレスポンスサイズ |

### storage.py 内の GCS 操作マッピング

|-------------------|---------|-------|-----------|

## 6. 実施済みコスト削減施策

### ── 第1回最適化 (2026-04-13) ──

キャッシュ強化・ローカル Parquet レイヤー導入・exists 最適化を中心とした基盤改善。

### 6.1 storage.py キャッシュ TTL の大幅延長

| キャッシュ | 変更前 | 変更後 | 効果 |
|-----------|--------|--------|------|
| blob list メモリ (過去年) | 12h | **7日** | list (Class A) 大幅削減 |
| blob list メモリ (当年) | 5min | **30min** | list (Class A) 6倍削減 |
| blob list ディスク (過去年) | 12h | **7日** | 再起動後も有効 |
| blob list ディスク (当年) | 5min | **30min** | 再起動後も有効 |
| メモリ L1 (load結果) | 1h | **2h** | GCS get 削減 |
| メモリ L1 最大エントリ | 8,000 | **12,000** | eviction 削減 |
| ディスク L2 (過去年) | 48h | **7日** | GCS get 大幅削減 |
| ディスク L2 (当年) | 12h | **24h** | GCS get 削減 |
| ディスク L2 書込閾値 | 週2回アクセス | **週1回** | 初回アクセスで即キャッシュ |

### 6.12 is_horse_fresh() の GCS load 完全廃止 — NEW

**問題**: `is_horse_fresh()` がセッションキャッシュにない馬について `storage.load("horse_result", hid)` を
呼び出し、**毎回 GCS から horse_result のフル JSON (平均 20KB) をダウンロード**していた。
1レース18頭 × 12レース/日 = 216回/日、週次バッチでは数万回に達し、
**264 GiB / 月の Download APAC (¥3,149)** の主要因となっていた。

**修正内容**:
- `is_horse_fresh()` から `storage.load()` を**完全に削除**
- blob list cache のタイムスタンプのみで鮮度判定 (GCS コスト ¥0)
- 過去年 (当年未満) のレース → 存在すれば無条件スキップ
- 当年レース → `horse_freshness_days` (7日) 以内ならスキップ

**ファイル**: `scraper/run.py`

### 6.13 scrape_race_all() のスキップ時 GCS load 廃止 — NEW

**問題**: `is_horse_fresh` が True でスキップ判定後、再度 `storage.load("horse_result", hid)` で
**フル JSON をダウンロード**して `horses` dict に格納していた。
同様に `horse_pedigree_5gen` も `storage.load()` で存在チェック + ダウンロードしていた。

**修正内容**:
- horse_result: スキップ時は `hdata = None` (GCS load しない)
- horse_pedigree_5gen: `storage.load()` → `storage.exists()` に変更 (blob list cache 判定)
- 種牡馬の 5gen チェックも同様に `exists()` に変更

| horse_pedigree_5gen チェック | `storage.load()` (GCS get + egress) | `storage.exists()` (blob list cache) | **¥0** |

### 6.14 prefetch_horse_timestamps() の生年範囲網羅 — NEW

**問題**: `prefetch_horse_timestamps("2024")` は馬ID先頭4桁が "2024" の馬のみプリフェッチ。

例: 2024年レース → 2015〜2022年生まれの馬の blob list を取得。
