# 馬券推奨レイヤと強化学習 — 設計議論

## 1. スコープの宣言

本ドキュメントは次の **境界** を明確にする。

| レイヤ | 役割 | 主な手法（想定） |
|--------|------|-------------------|
| **確率モデル（Stage6 まで）** | レース直前情報のみで **勝率・連帯率・複勝率**（`p_win`, `p_top2`, `p_top3`）を推定 | 従来の **教師あり学習**（GBDT / ランキング / NN 等）。`docs/modeling/` の母表・split・OOF 方針が主戦場。 |
| **馬券推奨レイヤ（本書）** | **オッズ**と**モデル確率**と**予測印**を統合し、**バランスの良い期待値**に基づき **推奨券種・組合せ・印の結びつけ** を出力 | **強化学習（RL）** による **戦略（ポリシー）学習** を第一候補とする。ここを「もう一段の勾配ブースティング」とみなさない。 |

- **期待値計算**（`model_prob × オッズ` からのエッジ、safe_ev 等）は推奨レイヤの **構成要素** であるが、**それ単体で終わらせない**。印との一貫性・券種間トレードオフ・資金制約下の系列意思決定を **RL で統合**する想定。

関連: `betting_backtest_dataset_spec.md`（データ契約・`safe_ev`）、`pre_race_feature_whitelist.md`（LayerA/LayerB）、`staged_probability_pipeline.md`（Stage7）。

## 2. なぜ推奨部分を「機械学習の延長」と切り分けるか

1. **目的関数が異なる**  
   確率モデルは **校正された確度**（LogLoss / Brier / ECE）が主。馬券レイヤは **長期リターン・ドローダウン・的中体験（印の説得力）** など **複数目的と制約** のバランスになる。

2. **行動空間が組合せ爆発する**  
   馬単・三連系は離散行動が巨大。**単純な argmax EV** は分散が極端になりやすく、**資金制約・同レース内相関**を同時に扱いにくい。

3. **観測は市場に依存する（LayerB）**  
   オッズは **対抗的・非定常**。確率モデル学習時に除外した市場情報が、ここでは **状態の中心** になる。

4. **予測印は「人間可読な中間表現」**  
   印は UI/説明責任のための **シンボル**。確率ベクトルと **一貫した写像** を学習で持つより、**ポリシーが印と券を同時に選ぶ**方が、商品として自然なことが多い。

## 3. 強化学習で捉える MDP の骨格（概念）

### 3.1 エージェントと環境

- **エージェント**: 推奨エンジン（1 ユーザーまたは 1 ポリシー単位でも可）。
- **環境**: **開催日・レース列**（バックテストでは履歴リプレイ、本番ではストリーミング）。各ステップで「そのレースの市場スナップショット＋確率テーブル」が観測される。

### 3.2 状態（State）の典型成分

- **モデル側**: `p_win`, `p_top2`, `p_top3`、校正後確率、不確実性 proxy、ランキングスコア、（任意）多段 stage の中間出力。
- **市場側**: `win_odds`, `place_odds_*`, 組合せオッズの要約、人気順、オッズ変化率（利用可能なら）。
- **文脈**: `venue_code`, `surface`, `field_size`, 重賞フラグ、残高・当日累積損益（系列 MDP にする場合）。
- **印候補**: ルールまたは別ヘッドで生成した **印ラベル候補**（◎○▲△など）を multi-hot または embedding で入れる。

### 3.3 行動（Action）

現実的には **階層化** が必須。

1. **レベル1**: 券種選択（単勝のみ / 複勝まで / 馬連を含む 等）— 少数離散。
2. **レベル2**: 各券種内で **Top-K 組** または **金額付き離散バケット**（100円×{0,1,2,5}）。
3. **レベル3（印）**: 「今回の推奨セットに含める印の集合」と「券の対応」— **同一ポリシー出力**にまとめると一貫性が出やすい。

**行動空間の縮小**が学習成功率を支配する。初期は **contextual bandit（1 レース = 1 ステップ）** に近い形に寄せるのが現実的。

### 3.4 報酬（Reward）

単純に「そのレースの払戻 − 投資」だけだと分散が殺す。候補:

- **短期**: `safe_ev` 加重の実現エッジ（`betting_backtest_dataset_spec.md` と整合）。
- **リスク調整**: `return − λ * drawdown_penalty` や `Sharpe-like`（レース単位ではノイズ大 → **セッション単位**で集約）。
- **印一貫性ペナルティ**: 印が「買い推奨」と矛盾する場合に負報酬（ルールベースでも可）。
- **スパース対策**: 的中時だけでなく **「買わない」** に小さな報酬を与え過学習を防ぐ設計は慎重に（何も買わない退化）。

## 4. 「バランスの良い期待値」と RL の関係

- **期待値**は **行動のスコアリング関数** として使うのが自然（Q や advantage の形で `f(p, odds, stake)`）。
- **バランス**は **報酬設計と制約** で表現する例:
  - 高オッズ偏重を `safe_ev` の **buffer** で抑える（既存の安全域思想を RL の報酬に埋め込む）。
  - 券種ごとに **上限 stake** や **Kelly 風クリップ** を環境側の制約として入れる。
- **RL は期待値を「置き換え」ではなく「方策の学習信号として統合」**するイメージがよい。  
  すなわち **ルール EV フィルタ → 残候補に対して方策勾配** の二段でもよい。

## 5. 予測印との結びつけ（商品設計）

### 5.1 印の役割

- **説明**: ユーザーが「なぜこの券か」を理解するための圧縮表現。
- **制約**: 印の種類数は少ない → **方策の出力次元を削る**のに使える。

### 5.2 実装パターン

| パターン | 内容 | メリット / デメリット |
|----------|------|------------------------|
| **A. 印を状態のみ** | 別モジュールが印候補を生成し、RL は券と金額のみ選択 | 印と券のズレが出やすい → ペナルティで調整 |
| **B. 印を行動の一部** | 行動 = (印セット, 券種, 組) の joint | 一貫性は高いが学習難易度上昇 |
| **C. 階層 RL** | 高レベルが「印ストーリー」、低レベルが具体券 | 解釈性と探索の分離 |

推奨: **初期は A + ペナルティ**、データが溜まったら **B へ移行**。

## 6. 学習パラダイムの比較（深い議論の核）

### 6.1 オンライン RL

- 実データで探索するコストが極大（金銭・評判）。**本番ではまず採用しない**。

### 6.2 オフライン RL（Batch RL）

- ログ: `(state, action, reward, next_state)` を **バックテスト or シャドウ運用**から蓄積。
- **Conservative Q-Learning (CQL)** 系や **IQL** など、**学習ポリシーとログ生成ポリシーのずれ**に強い手法が候補。
- 課題: **オッズは反事実**（別の券を買っていたらオッズが動いた）— 環境は **確定オッズでの反事実は近似**に留める宣言が必要。

### 6.3 Contextual Bandits

- 1 レース = 1 決定、**次状態は独立近似**とみなす。
- 行動を「候補券の有限集合」に制限すれば **LinUCB / Thompson** から始められる。
- **RL への橋渡し**として最優先でプロトタイプしやすい。

### 6.4 模倣学習 + 微調整

- 人間または **safe_ev ルール** の行動を **BC（Behavior Cloning）** で真似し、その後 **小さなオフライン RL** で改善。
- **安全性**: ルールが「教師ポリシー」の下限になる。

## 7. バックテスト環境としての MDP

- `betting_backtest_dataset` が **遷移の simulator** になる想定。
- **1 ステップ = 1 レース**（または 1 開催日）で、報酬は `payout_table` と `stake` から計算。
- **状態遷移**: 主に **残高** と **累積指標** が変化（レース特徴は次サンプルから独立に見せる近似も可）。

### 7.1 評価で必ず分ける指標

| 種別 | 指標例 | 注意 |
|------|--------|------|
| **確率品質** | LogLoss, ECE | Stage6 の責務。RL で改善したくない（混同禁止）。 |
| **戦略品質** | ROI, maxDD, avg_safe_ev, selection_rate | RL の責務。test 期間のみ報告。 |

## 8. リスクと対策（設計レビュー用チェックリスト）

- [ ] **データリーク**: RL の学習に **確率モデルの test 期間の予測**を混ぜていないか。  
- [ ] **オッズ因果**: 反事実評価の仮定を文書化したか。  
- [ ] **探索の安全**: 本番は **確率ポリシー + 閾値** で行動をクリップするか。  
- [ ] **非定常**: モデル・オッズ分布のドリフト検知と **ポリシー再学習トリガー**。  
- [ ] **説明責任**: 印・券・期待値の **トレースログ** を同一 `run_id` で保存。  
- [ ] **規制・倫理**: 自動投票の可否は製品ポリシーで分離（技術文書では参照のみ）。

## 9. 実装ロードマップ（提案）

1. **Phase 0（非 RL）**  
   `safe_ev` + 閾値 + 券種別ルールで **ベースライン推奨** と **印の機械的マッピング**。ログ形式を RL 用に整える。
2. **Phase 1（Contextual Bandit）**  
   候補券を有限個に量子化し、ログから **方策改善**（LinUCB / ニューラル bandit）。
3. **Phase 2（Offline RL）**  
   Q 関数または **制約付き方策最適化**で、セッション報酬を導入。
4. **Phase 3（階層・印 joint）**  
   データと安全性担保が揃ってから。

## 11. まとめ

- **勝率/連帯/複勝の学習**は従来の教師あり ML で進める。  
- **馬券推奨**は **オッズ + 確率 + 印 + 資金制約** を束ねた **意思決定問題** として **RL（まず bandit / offline RL）** で進めるのが筋が良い。  
- **期待値は中核だが終点ではない**；**バランス**は報酬・制約・階層行動で表現する。  
- 既存の `safe_ev`・バックテストデータ仕様と **ログ契約を揃える**ことが、研究と製品の両方で成功確率を上げる。
