Pre-race Probability — Multi-stage ML Pipeline

機械学習パイプライン 全体図

レース直前の T-10分 時点情報のみで、各馬の 勝率 / 連対率 / 複勝率 を出し、馬券最適化までつなげる 多段階 ML パイプラインを、データ契約・ステージ設計・テクニック・運用ルールまで一枚に統合したリファレンスです。

Stage0 → Stage7 / 7 段階 LayerA (能力) / LayerB (市場) BaselineV1 / ExtendedV1 Train/Valid/Test（2020–23 / 24 / 25） Purged Group Time Series Split OOF Target Encoding LightGBM × CatBoost ensemble Polars-first / fold materialize

01 目的とゴール

到達ゴール

  • レース直前情報のみで p_win / p_top2 / p_top3 を出す
  • 予測確率を用いて safe_ev ベースの馬券最適化を行う
  • 確率信頼性 (LogLoss / Brier / ECE) を ROI より先に評価する

設計の思想

  • 単段ベースラインと多段階パイプラインの両建て
  • 特徴量を LayerA (能力) と LayerB (市場) に厳格分離
  • 母表は BaselineV1 と ExtendedV1 を同時生成し、差分で寄与を測る
  • OOF・Purged CV・version 管理を前提に、Kaggle 風 tabular 手法を投入

02 全体アーキテクチャ

多段階パイプラインは Stage0〜Stage7 の 7 段で構成。 Stage1 で馬単体の latent state を作り、Stage2/3 で位置取り・ペース、 Stage4 で当日馬場、Stage5 で血統文脈適合を経由して、Stage6 が最終確率を出力します。 Stage7 で初めて市場価格と統合して馬券判断に使います。

多段階化の意義

中間状態 (位置取り・ペース・当日バイアス・血統文脈) を明示的にモデル化することで、 ① 失敗原因を分解しやすい / ② 当日更新できる情報と事前固定情報を切り分けられる / ③ 血統を「その日の馬場特性」に応じて使い分けられる、という3つの利点があります。

03 LayerA / LayerB の分離

LayerA 市場非依存の能力・状態モデル。単勝/複勝/人気/連系オッズ を 絶対に 入れない。
LayerB 市場統合・馬券最適化。LayerA の馬単位確率に、直前オッズ・連系オッズを追加。

分離理由

市場を確率推定器に混ぜると、モデルが市場をなぞるだけになり edge を出せない。 市場は意思決定器側でのみ使い、確率推定器は純粋に能力 / 状態を当てに行く構造とします。

04 BaselineV1 / ExtendedV1 の二段構え

BaselineV1 — 安定基準

coverage と保守性を優先。最初の比較対象を固定。

race_shutuba_flat race_shutuba_past_flat race_index_flat horse bundle 集約

追加特徴量の寄与は常にこの差分で評価。

ExtendedV1 — 直前情報強化

BaselineV1 + 直前状態・馬場 proxy。

race_oikiri_flat race_paddock_flat race_trainer_comment_flat cushion / moisture venue_era_fit seasonal_surface_fit

STAGE 1

主な latent block

S2 Stage2 — Positioning / 追走難度

STAGE 2

「いつもの位置を取れるか」を予測

既存コード pipeline/tracking_difficulty.py を活用。

主な入力

  • 直近走の通過順・脚質ばらつき

S3 Stage3 — Pace Scenario

STAGE 3

レース全体の前半流速・消耗度・上がり勝負化を推定

既存コード pipeline/pace_predictor.py を活用。

入力

  • Stage2 の front_run_probability 集約

出力

派生

S4 Stage4 — Day Track Bias

STAGE 4

「今日はどんな馬が走りやすい日か」を二層 (prior / posterior) で推定

4A: 事前 day prior

  • venue / venue_era / surface / distance
  • track_condition / weather / cushion / moisture / season

4B: 同日更新 day posterior

  • 同日先行レースの上位馬特徴 / 脚質偏り / 内外枠バイアス
  • 上がり偏重 or 前残り / 血統クラスタ偏り

出力

day_bias_front day_bias_closing day_bias_inner day_bias_outer day_bias_stamina day_bias_burst day_bias_mud day_bias_speed

ブレンド方針

先行レース 0 → prior のみ / 1-2 本 → prior 強め + posterior 弱め / 3 本以上 → posterior 重み増。

S5 Stage5 — Bloodline × 当日文脈適合

STAGE 5

血統を「静的距離適性」だけでなく「当日 day-bias との相性」で再評価

既存コード: research/bloodline_distance.py / bloodline_vector.py / course_bloodline.py / sire_factor_*.py / note_aptitude_5gen.py / myostatin.py / race_quality_model.py

5A: 静的血統適性

sire_distance_fit dam_sire_distance_fit surface_fit mud_fit speed_fit stamina_fit burst_fit

5B: 当日文脈適合

bloodline_day_fit_speed   = sire_speed_fit  * day_bias_speed
bloodline_day_fit_stamina = sire_stamina_fit* day_bias_stamina
bloodline_day_fit_mud     = mud_fit        * day_bias_mud
bloodline_day_fit_burst   = burst_fit      * day_bias_burst

出力

contextual_bloodline_fit contextual_bloodline_speed_fit contextual_bloodline_stamina_fit contextual_bloodline_mud_fit contextual_bloodline_burst_fit same_day_sire_cluster_bias

S6 Stage6 — Final Probability

STAGE 6

Stage1-5 を統合し p_win / p_top2 / p_top3 を出す

入力ブロック

  • 基礎母表 (Stage0)
  • Stage1 latent state
  • Stage2 positioning outputs
  • Stage3 pace outputs
  • Stage4 day bias outputs
  • Stage5 contextual bloodline outputs
  • optional: LayerB market features (※意思決定層側に渡すのが原則)

モデル構成

Modelターゲット

ModelBp_top2
ModelCp_top3
ModelD (optional)race-wise ranking score

アンサンブル / 校正

LightGBM CatBoost OOF blending temperature / isotonic

出力

p_win_raw / p_win_calibrated p_top2_raw / p_top2_calibrated p_top3_raw / p_top3_calibrated rank_by_p_win

S7 Stage7 — Market 統合 & 馬券最適化

STAGE 7

初めて市場価格と結合し、safe_ev ベースで馬券判断

  • horse_probability_table + horse_market_snapshot + pair_market_snapshot + payout_table を統合
  • Stage6 までで市場非依存モデル完成 / Stage7 でのみ市場統合 を厳守
  • 高オッズ帯ほど buffer を厚くして safe_ev > 0 を最低条件に
  • 推奨印・券種・資金配分の戦略層は教師あり ML の延長ではなく 強化学習（方策） を想定。議論は betting_policy_reinforcement_learning.html、要約 HTML は betting_rl_policy。

05b Train / Valid / Test 分割と分布把握

競馬の行データは 同一レース内で強く相関し、かつ 時間方向の因果が厳しいため、 通常のランダム split はリークと楽観バイアスの温床になる。 本プロジェクトでは data/meta/modeling/dataset_split_manifest.json を 機械上の正とし、詳細はマークダウン docs/html/modeling/train_valid_test_split_strategy.html に集約する。

一次プロトコル（確率モデル本線）

役割期間用途
train2020–2023学習本体・fold 内 target encoding / 集約特徴の学習
valid2024特徴・モデル・ハイパラ・キャリブレーション fit（test を見ない）
test2025一度きりの最終汎化報告・セグメント補助評価

分割の単位

race_id 単位で train / valid / test を完全分離する。 行単位ランダム split は禁止。キャリブレーションは OOF または valid のみに fit し、 test では fit しない（advanced_modeling_techniques.html と一致）。

開発時の CV（主戦場）

Purged + embargo

  • group_key = race_id / time_key = race_date
  • 初期値: purge_window_days = 7、embargo_days = 3
  • OOF target encoding・中間 stage 出力も同一制約に従う

Rolling-origin folds

マニフェストの rolling_origin_folds（2020 起点で valid を半年ずつスライド）で ハイパラと特徴の安定性を見る。一次 valid が 2024 のみでも、開発中の分散推定に使う。

補助評価・ドリフト

  • 京都改修後（venue_era フィルタ）— 現行コース条件への追従
  • クッション観測あり / パドック有無 — ExtendedV1 の依存度と欠損バイアス
  • Adversarial validation — train(2020–2023) vs valid(2024) と train(2020–2023) vs test(2025) を用途別に実施し分布差を監視（advanced_modeling_techniques.html §3.2 と一致）

他ドキュメントの split との関係

race_performance_rating_system.html の 「2020–2024 履歴 vs 2025 test」は レースパフォーマンス特徴モジュール単体のベンチ向けであり、 上記の一次プロトコル（2020–2023 / 2024 / 2025）と目的が異なる。実験ログに protocol 名を必ず残す。

EDA サマリ（母集団の安定性）

13 モデル戦略・アンサンブル

単体モデル推奨順

  1. LightGBM baseline
  2. CatBoost baseline
  3. 2 者の OOF blending
  4. (option) XGBoost / 線形 baseline

初期 blend 案

blend_p_win  = 0.5 * lgbm + 0.5 * catboost
blend_p_top2 = weighted average
blend_p_top3 = weighted average

stacker は OOF 予測のみで学習

rank + probability の二層化

  • ModelA (確率) を主、ModelB (race 内 ranking) を tie-break / 安定化補助に

多段階パイプラインの学習ルール

  • 各 stage の出力は OOF で生成して次段に渡す
  • 次段学習時に train fold へ valid fold の stage 出力を混ぜない
  • stacking 時も OOF only を守る

    pl.col("speed_max").rank("ordinal").over("race_id").alias("speed_rank_in_race"),
    (
        (pl.col("speed_max") - pl.col("speed_max").mean().over("race_id"))
        / (pl.col("speed_max").std().over("race_id") + 1e-6)
    ).alias("speed_z_in_race"),
])

18 safe_ev と policy 管理

1. 生期待値

raw_ev = predicted_prob * odds - 1

2. 校正後期待値

calibrated_ev = calibrated_prob * odds - 1

3. 安全域込み

safe_ev = calibrated_ev - buffer

初期ルール

ベット候補は safe_ev > 0 を最低条件。 buffer は固定値ではなく odds bucket × uncertainty_proxy で可変。 高オッズ帯ほど buffer を厚く。

19 リーク防止チェック

必須ルール

  • 結果確定後テーブルを特徴量母表に入れない
  • LayerA に当日オッズを入れない
  • target encoding / 集計は split 後・fold 内で作る
  • finish_position を説明変数に戻さない
  • 同一 race_id を split 跨いで分けない
  • 推論時点に存在しない future master / encoded mapping を読まない
  • ordered / target encoding は purged_group_time_series_split 前提
  • calibration は train ではなく valid / OOF 予測のみで fit

学習前チェック

  • 除外テーブルが混入していないか
  • LayerA に odds / popularity が入っていないか
  • race_oikiri_flat が 1 頭 1 行に集約されているか
  • sparse テーブルに presence_flag が付与されているか
  • 未来日付の履歴が混ざっていないか
  • split 前に target encoding を作っていないか
  • BaselineV1 / ExtendedV1 を同時保存しているか

やらないこと

split 前 TE / test を見ながら encoding 作成 / market を LayerA 投入 / 高次交差を無制限増設 / ROI のみでモデル採択。
