Pre-race Probability — Multi-stage ML Pipeline

機械学習パイプライン 全体図

レース直前の T-10分 時点情報のみを使い、各馬の 勝率 / 連対率 / 複勝率 を推定します。 Stage0 から Stage7 までの多段階の処理の流れと、データ定義・テクニック・運用ルールをまとめたリファレンスです。

Stage0 → Stage7 / 7 段階 LayerA (能力) / LayerB (市場) BaselineV1 / ExtendedV1 Train/Valid/Test（2020–23 / 24 / 25） Purged Group Time Series Split OOF Target Encoding LightGBM × CatBoost ensemble Polars-first / fold materialize

01 目的とゴール

達成したいこと

  • レース直前情報のみで p_win / p_top2 / p_top3 を推定する
  • 推定した確率を使って safe_ev ベースの馬券最適化を行う
  • 確率の精度（LogLoss / Brier / ECE）を ROI より先に評価する

設計の考え方

  • シンプルな単段ベースラインと多段階パイプラインを並行して管理する
  • 特徴量（予測に使う数値）を LayerA（能力） と LayerB（市場） に厳格に分離する
  • 母表は BaselineV1 と ExtendedV1 を同時に生成し、差分で各特徴の寄与を測る
  • 交差検証での予測（OOF）・Purged CV・バージョン管理を前提とし、表形式データ向けの機械学習手法を活用する

02 全体アーキテクチャ

処理の流れは Stage0〜Stage7 の 8 段で構成されています。 Stage1 で馬の潜在的な能力状態を作り、Stage2・3 で位置取りとペースを推定します。 Stage4 で当日の馬場バイアス、Stage5 で血統と当日文脈の適合度を算出します。 Stage6 がこれらすべてを統合して最終確率を出力し、Stage7 で初めて市場オッズと組み合わせて馬券判断に使います。

なぜ多段階にするのか

位置取り・ペース・当日バイアス・血統文脈を個別にモデル化することで、次の3つの利点があります。

  • 予測が外れたとき、どの段階が原因かを特定しやすい
  • 当日更新できる情報（馬場・ペース）と事前固定の情報を明確に分けられる
  • 血統の評価を「その日の馬場特性」に合わせて調整できる

03 LayerA / LayerB の分離

レイヤー目的使用情報禁止事項
LayerA 能力・状態モデル（確率推定） 戦績・血統・調教・馬場状態（非市場情報） 単勝/複勝/人気/連系オッズ は絶対に使わない
LayerB 市場統合・馬券最適化 LayerA の確率 ＋ 直前オッズ・連系オッズ 確率推定には使わない（馬券判断層のみ）

なぜ分離するのか

市場オッズを確率推定に混ぜると、モデルが「市場をそのまま映す」だけになり、予測の優位性（edge）が生まれません。 市場情報は馬券判断の層（LayerB）でのみ使い、確率推定は純粋に馬の能力・状態を根拠にします。

04 BaselineV1 / ExtendedV1 の二段構え

バージョン方針主な入力テーブル備考
BaselineV1 カバレッジと保守性を優先した比較基準 race_shutuba_flat, race_shutuba_past_flat, race_index_flat, horse bundle 集約 追加特徴量の寄与は常にこの差分で評価する
ExtendedV1 直前状態・馬場の代理変数を強化 race_oikiri_flat, race_paddock_flat, race_trainer_comment_flat, cushion / moisture, venue_era_fit, seasonal_surface_fit BaselineV1 を基盤とし直前情報を追加

S4 Stage4 — 当日馬場バイアスの推定

STAGE 4

「今日はどんな馬が走りやすいか」を事前情報と当日実績の2層で推定する

4A: 事前推定（当日レース前）

  • venue / venue_era / surface / distance
  • track_condition / weather / cushion / moisture / season

4B: 当日更新（同日先行レース後）

  • 同日先行レースの上位馬特徴 / 脚質偏り / 内外枠バイアス
  • 上がり偏重か前残りか / 血統クラスタの偏り

出力

ブレンド方針

先行レースが 0 本 → 事前推定のみ使用。 1〜2 本 → 事前推定を主に、当日実績を補助的に使用。 3 本以上 → 当日実績の重みを増やす。

S7 Stage7 — 市場統合と馬券最適化

STAGE 7

初めて市場オッズと組み合わせ、safe_ev を根拠に馬券判断を行う

  • 推奨印・券種・資金配分の戦略層は、教師あり ML の延長ではなく 強化学習（方策） を想定。詳細は betting_policy_reinforcement_learning.html、要約 HTML は betting_rl_policy。

05b Train / Valid / Test の分割方針と分布把握

競馬のデータは 同一レース内で強く相関しており、時間方向の因果も厳しく存在します。 通常のランダム分割を使うと、情報リークと過楽観バイアスの原因になります。 本プロジェクトでは data/meta/modeling/dataset_split_manifest.json を 唯一の分割定義とし、詳細は docs/html/modeling/train_valid_test_split_strategy.html に集約します。

一次プロトコル（確率モデル本線）

役割期間用途
train2020–2023学習本体・fold 内 target encoding / 集約特徴量（予測に使う数値）の学習
valid2024特徴・モデル・モデルの設定値（ハイパーパラメータ）・確率の補正（キャリブレーション）の fit（test は見ない）
test2025一度きりの最終汎化報告・セグメント補助評価

分割の単位

race_id 単位で train / valid / test を完全分離します。 行単位のランダム分割は禁止です。確率の補正（キャリブレーション）は交差検証での予測（OOF）または valid のみに fit し、 test には適用しません（advanced_modeling_techniques.html と一致）。

開発時の交差検証（CV）

Purged + embargo

  • group_key = race_id / time_key = race_date
  • 初期値: purge_window_days = 7、embargo_days = 3
  • 交差検証での予測（OOF）target encoding・中間 stage 出力も同じ制約に従う

Rolling-origin folds

マニフェストの rolling_origin_folds（2020 起点で valid を半年ずつスライド）を使って、 モデルの設定値（ハイパーパラメータ）と特徴量の安定性を確認します。一次 valid が 2024 のみでも、開発中の分散推定に使えます。

補助評価・分布ドリフト

  • 京都改修後（venue_era フィルタ）— 現行コース条件への追従確認
  • クッション観測あり / パドック有無 — ExtendedV1 の依存度と欠損バイアスの確認
  • Adversarial validation（敵対的検証）— train(2020–2023) vs valid(2024) と train(2020–2023) vs test(2025) を用途別に実施し、分布差を監視（advanced_modeling_techniques.html §3.2 と一致）

他ドキュメントの分割との関係

race_performance_rating_system.html の 「2020–2024 履歴 vs 2025 test」は レースパフォーマンス特徴モジュール単体の検証向けです。 上記の一次プロトコル（2020–2023 / 2024 / 2025）とは目的が異なります。実験ログには必ずプロトコル名を記録してください。

09 カテゴリ変数のエンコーディング（数値化）方法

Tier分類手法注意事項
Tier 1 常設・低リスク frequency encoding count encoding rare label grouping label encoding リークリスクが低く常時使用可能
Tier 2 OOF 必須 target encoding CatBoost ordered TE expanding mean group target encoding fold 内 train 側のみで計算。test は全 train 統計量のみ使用
Tier 3 比較実験 target std / skew WoE 風 cross feature TE 差分評価が必要。常設しない

主な対象カテゴリと交差

jockey_name trainer_name sire dam_sire venue_surface_distance_key jockey × venue trainer × surface × distance sire × surface venue_era × surface bracket × venue_surface_distance

交差検証での予測（OOF）target encoding の鉄則

  • 全データの平均を直接使うことは禁止
  • fold ごとに train 側だけでカテゴリ平均を計算し、valid に適用する
  • test には全 train の統計量のみを使う
  • 希少カテゴリは global mean に縮小（shrink）する
  • 未知カテゴリは train fold の global mean にフォールバックする

10 数値特徴量の前処理

変換

  • log / clipping / winsorization
  • rank-gauss / quantile transform
  • 裾の重い列 (career_runs / days_since_last / training_time_4f / oikiri_best_lap)

レース内相対化（最重要）

  • rank / percentile / z-score
  • gap to best / gap to median
  • 対象: speed_*, body_weight, oikiri_best_evaluation, paddock_score

11 時系列を考慮した交差検証 / 敵対的検証

年次の train / valid / test の割り当てとデータ分布の整理は §5b Train/Valid/Test の分割方針と分布把握 を参照してください。

Purged Group Time Series Split

  • split unit = race_id
  • 時間キー = race_date
  • purge_window_days + embargo_days
  • OOF target encoding もこの分割に従う

Adversarial Validation（敵対的検証）

  • train(2020-2023) vs valid(2024) と train(2020-2023) vs test(2025) の用途別分布差を確認する
  • venue_era / cushion / paddock の分布ドリフトを検知する
  • 出力: adversarial_auc / ドリフトが大きい特徴量のリスト / ドリフトセグメント一覧

分割ルールのまとめ

2020-2023 train / 2024 valid / 2025 test
rolling-origin fold (race_id 単位)
行単位ランダム split: 禁止
train fold 内でのみ集計・encoding

12 確率の補正（キャリブレーション）

確率モデルにおいて最も重要な工程です。コンペ上位を目指すより実運用を重視するなら、後段での確率補正が効果的です。

項目内容
候補手法 temperature scaling isotonic regression Platt scaling
fit 先 交差検証での予測（OOF）または valid の予測結果にのみ fit する。test には fit しない
個別比較 p_win / p_top2 / p_top3 それぞれで個別に比較する
評価指標 LogLoss Brier ECE

  2. CatBoost ベースライン

18 safe_ev の定義とベット方針の管理

1. 生期待値

raw_ev = predicted_prob * odds - 1

2. 確率補正後の期待値

calibrated_ev = calibrated_prob * odds - 1

3. 安全域を差し引いた期待値

safe_ev = calibrated_ev - buffer

基本ルール

ベット候補の最低条件は safe_ev > 0 です。 buffer は固定値ではなく、オッズ帯 × 不確実性の代理変数 で変動します。 高オッズ帯ほど buffer を厚くします。

20 実装フェーズ

PHASE 1
単段ベースラインの確立
Stage0 + Stage1 + Stage6 を構築し、BaselineV1 で安定した比較基準を作る。
PHASE 2
位置取り・ペースの追加
Stage2 / Stage3 を導入し、差分で改善幅を測定する。
PHASE 3
当日バイアス・血統文脈の追加
Stage4 / Stage5 を追加して、当日の馬場・文脈に適合したモデルへ発展させる。
PHASE 4
safe_ev による馬券最適化
Stage7 を実装。pair_probability_input から資金配分の最適化へつなげる。

技術の導入順序

  1. frequency / count encoding
  2. 交差検証での予測（OOF）target encoding
  3. CatBoost-style ordered encoding
  4. purged group time series split
  5. 確率の補正（キャリブレーション）
  6. 位置取り・ペース推定の補助モデル
  7. 当日バイアス・血統文脈の補助モデル
  8. LightGBM + CatBoost 交差検証での予測（OOF）ensemble
  9. safe_ev と方針探索
