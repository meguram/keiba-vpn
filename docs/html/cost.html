6.2 storage.load() にローカル Parquet レイヤーを追加

新しい優先順位: メモリ L1 → ディスク L2 → ローカル Parquet → GCS
  • src/scraper/local_tables.py を新設
  • storage.load() に L3 として Parquet テーブルからの読み込みを追加
  • 全 API エンドポイント・全研究モジュール・全パイプライン が自動的にローカル Parquet を利用
  • エクスポート済みの過去年レースデータは GCS コスト完全ゼロ に

6.3 exists() の最適化

  • blob list がキャッシュにない場合、個別 exists_gcs() (Class B) の前に batch_list_blobs() (Class A × 1) を実行
  • 1年分の全キーを一括取得し、以降の同年 exists チェックはキャッシュから応答

6.4 トラック速度計算のローカル化

  • _compute_track_speed() がローカル Parquet から直接読み込むように変更
  • GCS フォールバックは Parquet が存在しない場合のみ
  • ¥1.44/回 → ¥0/回

6.5 成長曲線 race_index のローカル化

  • _load_race_index_local_batch() で Parquet から一括取得
  • GCS にフォールバックするのは Parquet にない年のみ
  • ¥0.001/回 → ¥0/回 (大半)

6.12 is_horse_fresh() の GCS load 完全廃止 — NEW

問題: is_horse_fresh() がセッションキャッシュにない馬について storage.load("horse_result", hid) を
呼び出し、毎回 GCS から horse_result のフル JSON (平均 20KB) をダウンロードしていた。
1レース18頭 × 12レース/日 = 216回/日、週次バッチでは数万回に達し、
264 GiB / 月の Download APAC (¥3,149) の主要因となっていた。

修正内容:
- is_horse_fresh() から storage.load() を完全に削除
- blob list cache のタイムスタンプのみで鮮度判定 (GCS コスト ¥0)
- 過去年 (当年未満) のレース → 存在すれば無条件スキップ
- 当年レース → horse_freshness_days (7日) 以内ならスキップ

ファイル: src/scraper/run.py

6.13 scrape_race_all() のスキップ時 GCS load 廃止 — NEW

問題: is_horse_fresh が True でスキップ判定後、再度 storage.load("horse_result", hid) で
フル JSON をダウンロードして horses dict に格納していた。
同様に horse_pedigree_5gen も storage.load() で存在チェック + ダウンロードしていた。

修正内容:
- horse_result: スキップ時は hdata = None (GCS load しない)
- horse_pedigree_5gen: storage.load() → storage.exists() に変更 (blob list cache 判定)
- 種牡馬の 5gen チェックも同様に exists() に変更

horse_result スキップ時 storage.load() (GCS get + egress) None ¥0

6.14 prefetch_horse_timestamps() の生年範囲網羅 — NEW

問題: prefetch_horse_timestamps("2024") は馬ID先頭4桁が "2024" の馬のみプリフェッチ。
実際のレースには多様な生年の馬が出走するため、カバー率が不足。

修正: レース開催年から逆算して生年 -9〜-2 の範囲を一括プリフェッチ。
例: 2024年レース → 2015〜2022年生まれの馬の blob list を取得。

8. GCP SKU 内訳確認の手順

実績コスト ¥6,026 の内訳を確認するには:

  1. GCP Console → 請求先アカウント
  2. レポート → 「SKU」でグループ化
  3. 各SKUの料金を確認:
    - Standard Storage - Class A Operations (list/put)
    - Standard Storage - Class B Operations (get)
    - Standard Storage (データ保存)
    - Download Worldwide Destinations (ネットワーク下り)
    - その他のサービス (Compute Engine 等)

4月の急増 (前月比 +157.7%) は、テーブルエクスポート (全年全カテゴリ) や
大規模な再スクレイピング作業が 4月に集中したことが原因と考えられる。
第3回最適化後は、馬データの GCS egress が排除されるため、翌月から大幅減が見込まれる。
