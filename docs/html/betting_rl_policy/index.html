1. 境界 — 何が ML で、何が RL か

レイヤ出力手法の位置づけ
確率モデル
Stage6 まで		 p_win, p_top2, p_top3 等 GBDT / ランキング / NN など 教師あり学習。split・OOF・校正は既存 docs/modeling/ に従う。
馬券推奨
本ページの主題		 券種・組合せ・投資額・予測印との対応 強化学習（ポリシー）。オッズは状態の中心。期待値はスコアリングの核だが 終点ではない。

2. なぜ推奨を RL に寄せるか

目的が異なる

確率層は校正・対数損失。推奨層は 長期リターン・ドローダウン・印の説得力など複数目標と制約のバランス。

行動空間が巨大

単純 argmax EV は分散が尖りやすい。資金・券種相関を同時に扱うには方策表現が有利。

市場は非定常・対抗的

LayerB のオッズは確率学習から切り離してきた情報であり、状態遷移の一部として RL に入れるのが自然。

3. MDP の骨格（概念）

flowchart LR subgraph obs[状態] P[p_calibrated] U[uncertainty] O[odds snapshot] C[印候補] X[venue/surface/field] end subgraph act[行動] T[券種・組] S[金額バケット] M[印の束ね方] end subgraph rew[報酬] R[safe_ev 加重エッジ] D[DD ペナルティ] I[印一貫性] end obs --> act --> rew

行動の階層化

  • レベル1：券種の離散選択（少数）。
  • レベル2：各券種で Top-K 組 ＋ 金額の離散バケット。
  • レベル3：印を 行動の一部にするか 状態＋ペナルティにするか（初期は後者が学習しやすい）。

4. 期待値と「バランス」

  • 期待値は Q / advantage の構成要素として組み込む（safe_ev・buffer は betting_backtest_dataset_spec.md と整合）。
  • バランスは報酬合成（リスク項・印ペナルティ）と 環境側の stake 制約で表現する。
  • 実務的には ルール EV で候補を刈り込み → 残りを方策で配分 でもよい（二段ガード）。

5. 予測印との結びつけ

パターン概要コメント
A印は別モジュールが生成し、RL は券＋金額のみ導入容易。印と券のズレは報酬ペナルティで抑える。
B行動 = (印セット, 券, 組) の joint一貫性は高いが探索空間が膨らむ。
C階層 RL：上位が「印ストーリー」、下位が具体券説明性と探索の分離。データが溜まってから。

6. 学習パラダイムの比較

  • オンライン RL：本番での探索コストが高すぎるため、原則採用しない。
  • Contextual Bandit：1 レース ≈ 1 ステップ。候補券を有限化すれば 最初の本番的プロトタイプに最適。
  • Offline RL：ログ (s,a,r,s') から学習。CQL / IQL 等で 行動分布のずれに注意。オッズの反事実性は仮定として文書化。
  • BC + 微調整：safe_ev ルールや人間方策を教師にし、小さなオフライン RL で改善。

7. バックテスト環境

horse_probability_table と horse_market_snapshot と payout_table が simulator の入力契約になる（既存仕様を MDP に写像）。

指標の分離：LogLoss / ECE は確率モデルの責務。ROI・DD・selection_rate は推奨ポリシーの責務。test 期間の扱いは manifest と一致させる。

8. リスクチェック（要約）

  • RL 学習に 確率モデルの test 予測を混ぜない。
  • 反事実オッズの近似範囲を明記する。
  • 本番は方策 + ハードクリップ（最大投資・券種禁止リスト等）。
  • 印・券・期待値を同一 run_id でトレース可能にする。

9. ロードマップ（提案）

  1. Phase 0：safe_ev + 閾値 + 印のルールマッピング（ログスキーマを RL 用に整備）。
  2. Phase 1：Contextual Bandit（有限候補）。
  3. Phase 2：Offline RL（セッション報酬）。
  4. Phase 3：階層化・印 joint 行動。
