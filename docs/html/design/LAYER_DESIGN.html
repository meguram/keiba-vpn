レイヤー構成・設計仕様書

keiba-vpn の全レイヤー・モジュール構成、データフロー、主要設計パターンをまとめた技術リファレンスです。 2026-05

システム概要

keiba-vpn は JRA 競馬予測プラットフォームです。スクレイピング（Webからの自動データ収集）・ストレージ・ML 予測実行・Web API を垂直に統合し、以下の 5 つの責務層で構成されています。

L5 API 層 — FastAPI Web Server
HTTP のAPIのアドレス、レース詳細 UI、血統分析 UI、ベッティング最適化。バックグラウンドワーカーを内包。
L4 予測処理・キャッシュ層 — Inference Services
tracking_difficulty / finish_order / final_odds / pace の 4 モデルサービス。InferenceCacheMixin で Google Cloud Storage（GCS） ← → メモリ LRU の二段キャッシュを実現。
L3 パイプライン層 — Feature & ML Pipeline
特徴量（予測に使う数値）エンジニアリング（Parquet フィーチャーストア）・GBDT（木を積み上げた機械学習モデル）学習・MLflow（モデルの保管・バージョン管理ツール）管理。
L2 ストレージ層 — HybridStorage + SQLite
GCS（正典）→ ディスク L2 キャッシュ → メモリ LRU の三段構成。血統 DB は SQLite (bloodline.db) で別管理。
L1 スクレイピング層 — Scraper
netkeiba / SmartRC / JRA HTTP クライアント、HTML パーサー、ジョブキュー、並列ワーカー。

L1 スクレイピング層

src/scraper/ — netkeiba・SmartRC・JRA からのデータ取得

HTTP クライアント client.py

  • netkeiba プレミアム会員ログイン
  • レート制限 (throttle min/max)
  • 再試行・Bot 検出回避
  • 同時接続数の上限制御

HTML パーサー parsers.py

  • BeautifulSoup ベース
  • race_shutuba / race_result / horse_result
  • 出馬表・レース結果・追い切り
  • ページ構造のバージョン管理

ジョブキュー job_queue.py

  • 優先度付きキュー
  • smart_skip（データの新鮮度チェック）
  • 強制上書きポリシー
  • ロックファイルで複数プロセス間を協調

ワーカー queue_worker.py

  • SCRAPE_QUEUE_PARALLEL=6（既定）
  • stagger_sec でバースト防止
  • STAGGER スタート（デフォルト 0.25s）
  • ヘルス JSON を data/queue/ に書き出し

自動スクレイピング auto_scrape.py

  • 前日夕方（18:00 JST）に Cron で起動
  • 出走表取得 → 事前計算まで一括実行
  • tracking_difficulty と final_odds を事前計算

ScraperRunner run.py

  • HybridStorage を保持するオーケストレーター
  • 複数のスクレイプジョブをディスパッチ
  • storage アクセスを集約する（DI 的役割）

L1 メモリ LRU

Python dict + タイムスタンプ

  • 上限: LOAD_CACHE_MAX_ENTRIES（8,000）
  • データの有効期限（TTL）: LOAD_CACHE_TTL_SEC（3,600 s）
  • 上限超過時: 古い 10% を削除
  • VPS: 1,500 エントリ / 1,800 s

L2 ディスク

data/cache/ 以下の JSON/Parquet

  • 週次アクセス回数が閾値を超えると書き込み
  • DISK_L2_MIN_WEEKLY_ACCESSES（通常 2、VPS 3）
  • データの有効期限（TTL）: 2 日（デフォルト）
  • アクセス統計は .meta.json に保存

L2b SQLite 血統 DB

data/bloodline/bloodline.db + src/pipeline/data/bloodline_sqlite.py

大容量 Parquet をプロセス内で全件展開するとメモリが不足するため、血統関連テーブルを SQLite に集約しています。インデックス付きクエリで必要な行だけ取得します。

主な削減効果: horse_pedigree_cats.parquet（846 MB をフルロード）→ horse_ids フィルタクエリ（数 MB / リクエスト）に置き換え。 race_records.parquet（235 MB）→ race_results テーブルから必要なカラムだけ SELECT。

スキーマのバージョン管理

各テーブルはソース Parquet のカラム名リストを MD5 ハッシュして _schema_meta に保存します。
次回マイグレーション時にハッシュが一致すればスキップし、変更があればそのテーブルのみ DROP して再構築します。

src/pipeline/ — 特徴量エンジニアリング・学習・MLflow 管理

フィーチャーストアの設計

Parquet 列指向ストレージで特徴量（予測に使う数値）ブロックを管理します。ブロック単位で年次パーティションを区切ります。

ブロック名結合キー内容

Layer A データセット（レース前特徴量）

設計方針: Layer A はオッズを一切含まない特徴量（予測に使う数値）のセットです。 本番予測実行でも同じ特徴量を使うことで、情報漏洩を構造的に防ぎます。
除外するカラムは config/settings.yaml の feature_policy.odds_excluded_columns で明示的に定義します。

前日夜の事前計算フロー

auto_scrape.run_raceday_eve_for_date()
  └─ 出走表スクレイプ (race_shutuba)
  └─ precompute_tracking_for_race_ids(race_ids, storage)   # tracking_difficulty
  └─ precompute_final_odds_for_race_ids(race_ids, storage) # final_odds
      └─ InferenceCacheMixin.get_or_compute() → GCS 保存
18:00 JST の Cron で前日夜に全レース分を事前計算し、GCS に保存します。 翌朝以降の API リクエストはキャッシュから即時返却（2 ms 以内）。 初回アクセスは ~2s、2 回目以降は ~0.3s 程度です。

ModelSpec カタログ (src/pipeline/mlflow/catalog.py)

全モデルの単一の信頼源（Single Source of Truth）です。モデルキーごとに、フレーバー・Serve URI・キャッシュ設定・ローカルフォールバックを定義します。

フォールバックの仕組み:
MLflow Serve URI が未設定 → ローカルの Booster ファイル（models/）を直接ロード →
Booster もない → ヒューリスティック推定（モデルなしで動作）
VPS では MLflow Docker を起動しないため、ローカル Booster またはヒューリスティックで運用します。

デプロイ構成

本番（VPS 2 GB）

  • uvicorn × 2 workers、KEIBA_PROFILE=vps
  • MLflow Serve なし → ローカル Booster を使用
  • bloodline.db を GCS から自動ダウンロード
  • Cron（18:00 JST）で前日夜に事前計算
  • Docker Compose は使わず直接起動

開発（大メモリ環境）

  • uvicorn × 4 workers
  • Docker Compose で MLflow Serve を起動
  • bloodline 先読み有効（267 MB）
  • MLflow Tracking UI: http://localhost:5000
  • Nginx リバースプロキシ（オプション）

Docker Compose のサービス構成

mlflow/server/docker-compose.yml:
  mlflow:5000            # tracking server (SQLite backend, artifact store: /mlflow/data)
  mlflow-serve-tracking:5001    # tracking_difficulty モデル実行サービス (pyfunc)
  mlflow-serve-finish-order:5002
  mlflow-serve-final-odds:5003
  mlflow-serve-keiba-lgbm:5010
  nginx:80/443           # リバースプロキシ + Basic Auth (htpasswd)
  keiba-api:8000         # FastAPI (data/ + models/ + config/ をマウント)
