推論サービス設計

keiba-vpn の推論層は src/pipeline/inference/ 配下の 3 つのサービスと MLflow ランタイム (src/pipeline/mlflow/) から構成されます。 このページでは各サービスの役割・キャッシュフロー・API エンドポイントを整理します。

共通キャッシュ構造 — InferenceCacheMixin

全サービスは src/pipeline/mlflow/inference_cache.py の InferenceCacheMixin を 継承し、HybridStorage（GCS + ローカル L2 + メモリ LRU）に推論 JSON をキャッシュします。 

class _TrackingDifficultyCache(InferenceCacheMixin):
    model_key = "tracking_difficulty"
    cache_version = 6   # バージョン変更でキャッシュ自動無効化

キャッシュフロー（read-through）

get_or_compute(storage, race_id)
  ├─ load_cached()  →  HybridStorage.load(cache_category, race_id)
  │     ├─ メモリ LRU
  │     ├─ ローカル disk L2
  │     └─ GCS
  ├─ キャッシュヒット + version 一致 + TTL 内  →  返却
  └─ キャッシュミス  →  build_*_response()  →  save_cached()  →  返却

TTL とキャッシュ有効化

設定環境変数既定値

追走難度 TTL KEIBA_TRACKING_DIFFICULTY_CACHE_TTL_SEC 86400 秒（24h）

追走難度サービス — tracking_difficulty_service.py

ACTIVE TrackingDifficultyService

枠順・隣接馬の傾向・前走通過順から「いつもの位置を取れるか（追走難度）」と 位置取りフロー（前半〜最終コーナーの各馬想定順位）を推定します。

主要関数

関数役割
load_race_bundle() race_shutuba + horse_result を並行ロード。pre_race_only=True で事後リーク防止
build_tracking_difficulty_response() レース全体の追走難度・ペース・位置フローを組み立てる（キャッシュ非管理）
get_or_compute() キャッシュ優先で結果を返す（APIから呼ぶ入口）
precompute_tracking_for_race_ids() 前日 18 時トリガーでバッチ事前計算

レスポンスの主な項目

pace_prediction position_flow entries (tracking_difficulty per horse) field_prev_stats _compute_meta

最終オッズ予測サービス — final_odds_service.py

ACTIVE FinalOddsService

発走前の出馬表・調教・指数・過去走から単勝・複勝オッズを推定します。 LightGBM 3 ヘッド（win / place_min / place_max）モデルを使用し、 models/final_odds_bundle.json またはMLflow Registry からロードします。

主要関数

関数役割
_build_race_data_from_storage() storage から出馬表・指数・馬過去走を並行収集
build_final_odds_response() 特徴量構築 → predict_final_odds() 呼出 → JSON 整形
get_or_compute() キャッシュ優先エントリポイント
precompute_final_odds_for_race_ids() 前日バッチ一括事前計算

レスポンスの主な項目

entries[].horse_number entries[].predicted_win_odds entries[].predicted_place_odds_min entries[].predicted_place_odds_max elapsed_sec

着順予測サービス — finish_order_service.py

PLANNED FinishOrderService（スタブ）

次期本番モデルの着順・複勝圏確率予測。現在は InferenceCacheMixin の サブクラス宣言のみ実装済みで、推論ロジックは未実装です。 実装時は tracking_difficulty_service.py と同パターンで build_finish_order_response() を追加します。 

class FinishOrderCache(InferenceCacheMixin):
    model_key = "finish_order"
    cache_version = 1

MLflow バックエンド解決フロー

src/pipeline/mlflow/runtime.py の infer_backend_label() が 推論時にどのバックエンドを使うかを決定します。 

def infer_backend_label(key: str) -> str:
    client = get_serve_client(key)
    if client is not None and client.is_available():
        return "mlflow_serve"      # 外部 MLflow モデルサーバ（推奨）
    if load_lightgbm_booster(key) is not None:
        return "local_mlflow"      # ローカル Registry または models/ ファイル
    return "heuristic"             # ルールベースフォールバック

ヘルスチェック

全モデルのバックエンド状態は GET /api/inference/health で確認できます。 MlflowServeClient.is_available() が /health エンドポイントに HTTP GET して 200 かどうかで判定します（60 秒キャッシュ）。
