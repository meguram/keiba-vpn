スクレイパーアーキテクチャ

JRA 開催スケジュールに連動した自動スクレイピングの全体設計をまとめたページです。 タスク種別・実行タイミング・取得対象・ストレージ構成を説明します。

概要

スクレイピングは src/scraper/auto_scrape.py のタスクを cron から起動することで実行されます。 データは GCS をプライマリストレージとし、HybridStorage（GCS + ローカル disk L2 + メモリ LRU）を経由して保存・読み込みします。

データソース

  • netkeiba (db.netkeiba.com) — メイン。レース・馬情報
  • SmartRC — 追加能力指標（cr_value・ten1f 等）
  • JRA 公式 — 馬場クッション値・含水率

レートリミット

サイト最小間隔日次上限
netkeiba1.0 s5,000 req
SmartRC2.0 s1,000 req

タスク一覧とスケジュール

タスク名 実行タイミング 対象日 主な取得データ
raceday-eve 前日 18:00 開催前日 race_shutuba / race_shutuba_past / race_oikiri、追走難度事前計算
raceday-runner 07:30 起動（常駐） 開催日 各 R 発走 −15 分に race_shutuba / race_odds / smartrc / jra_baba_live、AI 予測トリガー
raceday-result-runner 07:30 起動（常駐） 開催日 各 R 発走 +15 分に race_result（速報）
raceday-evening 17:30 開催日 race_result（確定）/ 確定オッズ / SmartRC 最終値
weekly-update 毎週金曜 17:30 毎週金曜 先週分 race_result / horse_result / race_index / race_oikiri
daily-race-lists 毎日 07:00 毎日 本日 + 14 日先の race_lists
jra-baba-morning 05:00–09:00（10 分毎） 開催日・前日朝 クッション値 / 含水率 / 馬場状態
CRON 前日 18:00 raceday-eve — 前日準備

翌開催日の出馬表・馬柱・追い切りを一括取得し、追走難度の事前計算まで行います。

18:00 起動
翌開催日の全レース一覧を取得
出馬表
race_shutuba を全レース分取得（前日時点の最新枠順・騎手）
馬柱・調教
race_shutuba_past + race_oikiri を全レース分取得
事前計算
追走難度キャッシュを事前計算（出馬表のみ・result 非使用）
CRON 07:30 (常駐) raceday-runner — 当日発走前

開催日に常駐し、各レース発走 15 分前に出馬表・オッズ・SmartRC・JRA 馬場情報をスクレイプします。

07:30 起動
レース一覧取得 → 発走時刻リスト生成
各R −15分前
race_shutuba（最終枠順・騎手確定版）+ race_odds + smartrc + jra_baba_live
※ 馬柱・追切は前日取得済みならスキップ
取得完了後
AI 予測トリガー（KEIBA_PRE_RACE_PREDICT_ENABLED=1 のとき）
CRON 07:30 (常駐) raceday-result-runner — 速報結果

各レース発走後 15 分に速報結果を取得します。

  • 各 R 発走 +15 分後に race_result をスクレイプ
  • raceday-runner と並行して同日に常駐
CRON 開催日 17:30 raceday-evening — 当日夕方

全レース完了後に確定データを取得します。

  • race_result（確定着順・確定タイム）
  • 確定オッズ（払い戻し）
  • SmartRC 最終値
CRON 毎週金曜 17:30 weekly-update — 週次更新

netkeiba の DB が金曜 17:00 に先週分更新されるため、直後に一括取得します。

カテゴリ 内容
race_result 先週分の確定着順・タイム・払い戻し
horse_result / horse_ped 馬情報（プロフィール・戦績・血統）
race_index 指数（速度指数）
race_oikiri 調教タイム・追い切り内容
CRON 毎日 07:00 daily-race-lists — レース一覧更新

本日 + 14 日先のレース一覧（race_lists）をローカルに保存します。

CRON 5:00–9:00 10分毎 jra-baba-morning — 馬場情報

開催日・前日の朝に JRA 公式から馬場情報を定期取得します。

  • クッション値（cushion_value）
  • 含水率
  • 馬場状態

スクレイパー実行コマンド

# 前日準備（通常は cron から）
python -m src.scraper.auto_scrape --task raceday-eve

# 開催日常駐（発走 −15 分取得）
python -m src.scraper.auto_scrape --task raceday-runner

# 開催日常駐（発走 +15 分速報結果）
python -m src.scraper.auto_scrape --task raceday-result-runner

# 当日夕方まとめ取得
python -m src.scraper.auto_scrape --task raceday-evening

# 週次 DB 更新取得（毎週金曜 17:30）
python -m src.scraper.auto_scrape --task weekly-update

# 毎日レース一覧更新（07:00）
python -m src.scraper.auto_scrape --task daily-race-lists

# ステータス確認
python -m src.scraper.auto_scrape --status

HybridStorage — データ保存レイヤー

src/scraper/storage.py の HybridStorage が全カテゴリのデータ保存・読み込みを管理します。

ストレージ階層

レイヤー 場所 / 仕様
GCS primary（全データ）
ローカル disk L2 data/cache/（週次アクセス頻度が高いキーのみ）
メモリ LRU 直近アクセスを TTL 付きでキャッシュ
race_lists のみ data/page_reference/ にローカル保存

鮮度管理

  • 保存時に _meta.scraped_at タイムスタンプを自動付与
  • is_fresh() は StructureMonitor の versions.json と照合
  • ページ構造変更があった場合、それ以前のデータは stale 扱いとなる

キュー型スクレイピング

バックフィルや大量取得には src/scraper/job_queue.py + queue_worker.py を使います。 API 経由で /api/scrape-queue/add にジョブを積み、バックグラウンドワーカーが順次処理します。 

POST /api/scrape-queue/add             # ジョブ追加
POST /api/scrape-queue/add-batch       # バッチ追加
GET  /api/scrape-queue/status          # キュー状態確認
GET  /api/scrape-queue/progress        # 進捗確認
GET  /api/scrape-queue/worker-logs     # ワーカーログ取得
POST /api/scrape-queue/resume          # 一時停止後に再開
POST /api/scrape-queue/kick            # スタック時の再起動

レートリミット遵守

キューワーカーは config/settings.yaml の data_sources.netkeiba.rate_limit と data_sources.smartrc.rate_limit に従ってリクエスト間隔を制御します。 日次上限（netkeiba: 5,000 / SmartRC: 1,000）を超えた場合は自動停止します。

HTTP 400 対策

src/scraper/http400_strategy.py に 400 エラー時の再試行戦略が実装されています。 連続 400 が発生した場合はセッション初期化・アクセス停止などに自動エスカレーションします。
