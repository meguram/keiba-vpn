1. エグゼクティブサマリ

本ドキュメントは keiba-vpn の現運用 SLA と 今後 (レース 10 分前 AI 予測) を見据えた要件 を一枚にまとめたものです。 現状はサーバ常駐スレッド (FastAPI lifespan) とローカル cron / systemd の二本立てで動作し、 APScheduler のような統一スケジューラは使用していません。

カテゴリ取得時の smart_skip 仕様 (重要)

HybridStorage.is_fresh() は ページ構造バージョンと既存 JSON の構造ハッシュを比較して鮮度判定します。 時刻ベースの鮮度ではないため、同じ race_id を T-2 時間で取得済みでも、 smart_skip=True のまま T-15 で再投入すると 古いオッズ / 馬体重がそのまま採用される恐れがあります。

auto_scrape の raceday-runner は scrape_race_card(..., skip_existing=False) で 強制再取得しているため通常は問題ありませんが、API 経由 (POST /api/race/{rid}/predict 内の scrape_race_all(..., smart_skip=True)) では「直前性」を保証しません。 → §9 の対応を参照。

scrape_race_all がまとめて呼ぶカテゴリ

ScraperRunner.scrape_race_all(race_id, smart_skip=True) は内部で ThreadPoolExecutor を使い、以下を並列取得します:

  • scrape_race_card (出馬表)
  • scrape_odds + scrape_pair_odds (オッズ)
  • scrape_shutuba_past (馬柱)
  • scrape_oikiri (追切)
  • scrape_paddock (パドック ★)
  • scrape_smartrc (SmartRC)
  • 取得した馬の scrape_horse + scrape_horse_pedigree_5gen

★ パドックは scrape_race_all には含まれるが、 auto_scrape.py の raceday-runner では除外されています。 AI 予測パイプラインは scrape_race_all を使うため、 API 経由の予測時はパドックも取得される点は把握しておくべきです。

9. 「レース 10 分前 AI 予測」運用設計（統合版）

目標は 発走 10 分前時点の予測物を出すことですが、 別途 T-10 再取得ジョブは置かない方針とします。 raceday-runner が各 R の 発走 T-15 分に netkeiba + JRA 馬場を一括取得し、 そのバンドル完了をトリガに AI 予測パイプラインを起動します（取得完了時点はおおよそ T-15〜T-12）。

実装（モック） src/scraper/raceday_pre_race_pipeline.py（T-15 バンドル）と src/pipeline/inference/pre_race_predict_trigger.py（予測トリガ）を追加済み。 auto_scrape._scrape_single_race から呼び出します。 本番予測は KEIBA_PRE_RACE_PREDICT_ENABLED=1 で有効化（最終版は後補完）。

9.1 1 レースあたりのフロー

9.2 JRA 馬場ライブの統合

  • 旧: setup_baba_cron.sh の systemd で開催窓内 5 分ポーリング（jra_baba_live --watch）
  • 新: 各 R の T-15 バンドル内で JRABabaLiveScraper().scrape() を呼び、netkeiba 取得と同タイミングで最新馬場をキャッチアップ
  • 同一バッチで複数 R を処理するため、KEIBA_BABA_REFRESH_MIN_SEC（既定 120 秒）で重複取得を抑制

10. 既知のリスクと注意点

カテゴリリスク内容緩和策
高 smart_skip HybridStorage.is_fresh は構造ベース。時刻ベースではないため、API 経由の予測で古いオッズ / 馬体重を採用する恐れ 予測パスでは smart_skip=False 強制。raceday-runner は skip_existing=False 既定なので問題なし
高 auto_scrape の cron 未自動セットアップ auto_scrape.py 用の setup_*.sh が無く、運用者が手動で crontab 登録する必要 本ドキュメントの §4.1 を参照に手動登録 + 確認用 --status サブコマンドを定期実行
中 uvicorn ワーカー多重起動 main.py --prod で 4 workers 起動すると、scheduler 群が 4 重に走る ワーカー単位で起動するロガーを確認。ファイルロックや singleton 化を検討
中 TZ 依存 scripts/cron/setup_*.sh 群はサーバ TZ に依存。海外 VPS だと意図しない時刻で実行される CRON_TZ=Asia/Tokyo を crontab 先頭に書く。OS TZ も Asia/Tokyo に設定推奨
中 netkeiba ログインセッション切れ パドック / 指数 / バロメータ / 厩舎コメントが要ログイン。セッション切れで自動取得失敗 NetkeibaClient の再ログインリトライ + watchdog でセッション健全性監視
中 ページ構造変更 netkeiba 側の HTML 改修で全パーサー停止 scheduler.structure (06:00 JST 毎日) で構造ハッシュ差分を即座に検知
低 5 世代血統ローカル不整合 data/local/horse_pedigree_5gen と 10gen の 2 系統。10gen 解析後の整合性は人手確認 10gen は研究用途専用。本番特徴量は 5gen を使う設計を維持
低 watchdog のログ肥大化 3 分ごとに走るため logs/watchdog.log が肥大化 0 0 * * 1 で 50MB 超を truncate (週次) 設定済み
