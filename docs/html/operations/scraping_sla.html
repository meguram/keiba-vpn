1. エグゼクティブサマリ

本ドキュメントは keiba-vpn の現運用 SLA（品質保証基準） と 今後の目標（レース 10 分前 AI 予測）に向けた要件と現状ギャップ を一枚にまとめたものです。

現状はサーバ常駐スレッド（FastAPI lifespan）とローカル cron / systemd の二系統で動作しています。 APScheduler のような統一スケジューラは使用していません。

2. システム全体の構成

データ取得処理は大きく 2 系統に分かれています。

  • 外部 cron / systemd で起動する auto_scrape: 開催日の朝・夕方・週次など時刻ベースで自動実行します。
  • FastAPI 常駐の queue.worker / scheduler 群: UI や API からの操作、および定期メンテナンス処理を担当します。

両系統は HybridStorage（Google Cloud Storage（GCS）を一次ソースとし、ローカルをキャッシュとして使う仕組み）を共有します。

smart_skip の仕様と注意点（重要）

HybridStorage.is_fresh() は ページ構造バージョンと既存 JSON の構造ハッシュを比較して鮮度を判定します。 時刻ベースの鮮度判定ではない点に注意が必要です。

同じ race_id を T-2 時間前に取得済みでも、smart_skip=True のまま T-15 で再投入すると 古いオッズ・馬体重がそのまま採用される恐れがあります。

auto_scrape の raceday-runner は scrape_race_card(..., skip_existing=False) で 強制再取得しているため通常は問題ありません。 ただし、API 経由（POST /api/race/{rid}/predict 内の scrape_race_all(..., smart_skip=True)）では 「直前性」が保証されません。詳細は §9 を参照してください。

5. 開催日のタイムライン

時刻はすべて JST です。各レースの発走時刻が異なるため、「R」は第 1R（例: 10:00）から順次繰り返されます（例: 第 2R 10:30, …）。

8. キューシステムと API トリガ

UI / API から投入されたジョブは JSON ファイルキュー（job_queue.json）に積まれ、 queue.worker が順次処理します。tasks 配列の順序がそのまま実行順になります（DAG ではありません）。

9. 「レース 10 分前 AI 予測」の運用設計（統合版）

目標は 発走 10 分前時点の予測結果を出すことですが、 別途 T-10 再取得ジョブは置かない方針です。

raceday-runner が各レースの 発走 T-15 分 に netkeiba + JRA 馬場を一括取得し、 そのバンドル完了をトリガ に AI 予測処理を起動します。 取得が完了する時点はおおよそ T-15〜T-12 です。

実装状況（現在はモック） src/scraper/raceday_pre_race_pipeline.py（T-15 バンドル）と src/pipeline/inference/pre_race_predict_trigger.py（予測トリガ）を追加済みです。 auto_scrape._scrape_single_race から呼び出します。 本番予測は KEIBA_PRE_RACE_PREDICT_ENABLED=1 を設定することで有効化できます（最終実装は後補完）。

9.2 JRA 馬場ライブの統合について

  • 旧方式: setup_baba_cron.sh の systemd サービスで開催窓内 5 分ごとにポーリング（jra_baba_live --watch）していました。
  • 新方式: 各レースの T-15 バンドル内で JRABabaLiveScraper().scrape() を呼び出し、netkeiba 取得と同タイミングで最新馬場情報をキャッチアップします。
  • 同一バッチで複数レースを処理する際、KEIBA_BABA_REFRESH_MIN_SEC（既定 120 秒）で重複取得を自動的に抑制します。

10. 既知のリスクと注意点

カテゴリリスク内容対策
高 smart_skip HybridStorage.is_fresh は構造ベースの判定です。時刻ベースではないため、API 経由の予測で古いオッズ・馬体重が採用される恐れがあります。 予測パスでは smart_skip=False を強制します。raceday-runner は skip_existing=False が既定なので問題ありません。
高 auto_scrape の cron 未自動セットアップ auto_scrape.py 用の setup_*.sh が存在しないため、運用者が手動で crontab 登録する必要があります。 本ドキュメントの §4.1 を参考に手動登録し、--status サブコマンドで定期確認してください。
中 uvicorn ワーカーの多重起動 main.py --prod で 4 workers を起動すると、scheduler 群が 4 重に動作します。 ワーカーごとのロガーを確認し、ファイルロックや singleton 化を検討してください。
中 タイムゾーン依存 scripts/cron/setup_*.sh 群はサーバのタイムゾーン設定に依存します。海外 VPS では意図しない時刻で実行される恐れがあります。 crontab の先頭に CRON_TZ=Asia/Tokyo を記載し、OS のタイムゾーンも Asia/Tokyo に設定することを推奨します。
中 netkeiba ログインセッション切れ パドック・指数・バロメータ・厩舎コメントは要ログインです。セッション切れで自動取得が失敗します。 NetkeibaClient の再ログインリトライと watchdog によるセッション健全性監視を行っています。
中 ページ構造変更 netkeiba 側の HTML が改修されると全パーサーが停止する恐れがあります。 scheduler.structure（毎日 06:00 JST）が構造ハッシュの差分を即座に検知します。
低 5 世代血統のローカル不整合 data/local/horse_pedigree_5gen と 10gen の 2 系統が存在します。10gen 解析後の整合性は人手での確認が必要です。 10gen は研究用途専用とし、本番の特徴量には 5gen を使う設計を維持してください。
低 watchdog のログ肥大化 3 分ごとに実行されるため logs/watchdog.log が肥大化します。 0 0 * * 1 の週次 cron で 50MB 超のファイルを truncate する設定を済みです。
