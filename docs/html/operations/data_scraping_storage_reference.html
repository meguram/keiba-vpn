§1 スクレイピングスケジュール

自動スクレイピングは src/scraper/auto_scrape.py が管理する 6 つのタスクと、常駐するキューワーカーで構成される。

1-1. 自動タスク (auto_scrape)

毎日 07:00
daily-race-lists
今日から 14 日先のレース一覧を取得し race_lists に保存。
保存先: data/local/race_lists/{YYYYMMDD}.json (ローカル専用・GCS なし)
未開催日は書かない (開催があった日のみファイル作成)。
開催日 07:30 起動 → レース発走 T-15 分ごと
raceday-runner
開催日のみ常駐。発走時刻を取得後、各レース T-15 分前に T-15 バンドル を実行。
T-15 バンドル取得データ:
  • race_shutuba (出馬表)
  • race_odds (単複オッズ)
  • race_pair_odds (2連系オッズ)
  • smartrc_race (SmartRC)
  • jra_baba_live (馬場速度 — jra_cushion カテゴリ)
T-15 バンドル完了後に AI 予測 (race_predictions) をトリガ。
src/scraper/raceday_pre_race_pipeline.py 参照。
開催日 17:30
raceday-evening
当日全レースの結果を速報取得。
取得データ: race_result_on_time, race_odds, race_pair_odds, smartrc_race
※ race_result_on_time は race.netkeiba.com/race/result.html からの当日速報専用カテゴリ。 AI 予測の再現用に別パスへ保持する。
race_result（db.netkeiba 確定版）は金曜 weekly-update で取得される。
終了後 _trigger_post_evening_baba_update() で race_result_flat.parquet を増分更新し馬場速度指数を再計算。
毎週金曜 17:30
weekly-update
先週分 (月〜日) のデータベース更新。JRA は金曜夕に前週データが確定する。
取得データ: race_result, race_index, race_barometer, horse_result
※ race_result は db.netkeiba.com/race/{race_id}/ 確定版。 ラップタイム・タイム指数の確定値はこのタイミングで揃う。 race_result_on_time（当日速報）とは別カテゴリのため上書き競合は生じない。
毎日 09:00 (非開催日のみ実質動作)
catchup-missing
カレンダーと race_lists の差分から取りこぼし日を検出し、キューへ date_all ジョブを投入するバックフィル処理。

1-2. キューワーカー (常駐)

FastAPI サーバー起動と同時に起動する非同期ワーカーが data/queue/scrape_queue.json を監視し、 デフォルト 6 並列でジョブを処理する (SCRAPE_QUEUE_PARALLEL 環境変数で変更可)。

1-3. jra_baba_live (systemd デーモン — legacy)

scripts/cron/setup_baba_cron.sh で登録する jra-baba-watch.service は JRA 馬場情報の 5 分ポーリングデーモン。
2026-05 以降は raceday-runner の T-15 バンドルに統合済みのため、二重取得回避のため停止推奨。

§2 scrape_race_all フェーズ詳細

ScraperRunner.scrape_race_all(race_id) (src/scraper/run.py) は 1 レースのすべてのデータを段階的に取得する。 date_all キュージョブは全レースにわたってこれを順次呼ぶ。

Phase 1 — 並列
HTML + API 同時取得
  • HTML (逐次)
  • race_result (レース結果)
    db.netkeiba 版を使用
    (当日速報は race_result_on_time に別保存 → §3 参照)
  • race_shutuba (出馬表)
    horse_id リストを起点に horse_result を join → 馬柱相当データを構成
  • race_index (タイム指数)
  • race_paddock (パドック評価)
  • race_trainer_comment (厩舎コメント)
  • API (別セッション並列)
  • race_odds (単複オッズ)
  • race_pair_odds (2連系オッズ)
  • race_barometer (偏差値)
Phase 2a — 並列
SmartRC + 馬情報
  • smartrc_race (SmartRC データ)
  • 出走馬ごと:
  • horse_result (馬情報・レース歴)
  • is_horse_fresh() で 7 日以内はスキップ
Phase 2b — 逐次
出走馬 5 世代血統
  • horse_pedigree_5gen (出走馬全頭)
  • GCS に 5 世代以上あればスキップ
  • 失敗時は個別キュージョブを再投入
  • 保存成功後トリガー:
  • → local/horse_pedigree_5gen/ 書き込み
  • → local/horse_pedigree_10gen/ ビルド
  • → ped_tbl Parquet 生成
Phase 2c — 逐次
種牡馬 5 世代血統
  • Phase 2b で取得した父系 horse_id
  • horse_pedigree_5gen (種牡馬)
  • 出走馬 ID は除外
  • 失敗時は再キュー
smart_skip が有効なとき、各カテゴリは is_fresh() (構造バージョン照合) でスキップ判定される。 バージョン v1 カテゴリは常に fresh 扱い。

§4 ストレージアーキテクチャ

GCS をプライマリとし、ローカルに複数の階層キャッシュを持つ構成。

☁
GCS (プライマリストア)
バケット: $GCS_BUCKET
race 系: chuou/data/preprocessed/netkeiba/pc/{category}/{year}/{key}.json
horse 系: chuou/data/preprocessed/netkeiba/pc/{category}/{pfx}/{key}.json
other 系: chuou/data/others/{category}/{key}.json
障害後 60 秒バックオフ。バックオフ中の save() は False を返し GCS には書かない。
1
メモリ L1 キャッシュ (プロセス内 LRU)
storage._load_cache — TTL: 3600 秒、最大 8000 エントリ
save() 成功時に _cache_put() で即時更新。GCS バックオフ中は更新されない。
環境変数: LOAD_CACHE_TTL_SEC, LOAD_CACHE_MAX_ENTRIES
2
ディスク L2 キャッシュ (週次アクセス閾値付き)
data/cache/{category}/{prefix}/{key}.json
TTL: 過去年 172800 秒 (2 日) / 当年 43200 秒 (12 時間)
週次アクセス ≥2 回のエントリのみ L2 に書き込む (ホットデータのみ保持)。
環境変数: DISK_L2_TTL_PAST_SEC, DISK_L2_TTL_CURRENT_SEC, DISK_L2_MIN_WEEKLY_ACCESSES
3
ローカルミラー (常設・オプション)
data/local/mirror/{category}/{prefix}/{key}.json
data/meta/local_mirror_config.json の enabled=true かつカテゴリが categories リストに含まれる場合のみ書かれる。
オフライン研究パイプライン向けの永続コピー。

Blob リストキャッシュ (モニター高速化)

batch_check_keys() が内部で使う GCS list API 結果のキャッシュ。 スクレイピング活動の有無によって TTL を動的に変える設計 (_last_scrape_activity)。

年スクレイピング活動なし直近 5 分以内に GCS save あり
当年60 秒60 秒
過去年3600 秒 (1 時間)60 秒 (活動ウィンドウ内)
過去年の血統 (horse_id が 2023xxxxx 等) を新規取得した直後は、活動ウィンドウ (5 分) 内に限り 60 秒 TTL でモニターに反映される。 GCS バックオフ中に save がスキップされた場合は _last_scrape_activity が更新されないため、TTL は長いまま維持される。

§5 キュージョブ仕様

ジョブは job_kind × tasks[] の組み合わせで指定する。 API エンドポイント POST /api/scrape/enqueue またはコード内で ScrapeJobQueue.bulk_add_jobs() を呼ぶ。

job_kindtarget_id使用可能 tasks説明
race race_id (例: 202601010101) race_all, race_result, race_result_lap (★副次生成), race_shutuba, race_odds, race_pair_odds, race_index, race_paddock, race_barometer, race_trainer_comment, smartrc 1 レース単位の取得。race_all は scrape_race_all() を呼ぶ。
horse horse_id (例: 2023100001) horse_pedigree_5gen, horse_training horse_pedigree_5gen は 5gen JSON のみ取得。
成功時に local 5gen / 10gen / ped_tbl も自動更新。
date YYYYMMDD (例: 20260510) race_list, date_results, date_cards, date_all date_all = scrape_date_all() を呼び、当日の全レースを scrape_race_all() する。

ジョブ状態フロー

新規投入 → precheck (smart_skip かつ overwrite=false のとき GCS 充足チェック)
         ↓ 不充足
       pending (待機)
         ↓ ワーカーが取得
       running (実行中)
         ↓ 完了 or 失敗
   completed / failed
         ↓ 1 時間ごとのメンテ
   failed → pending に戻る (上限回数まで)
   completed → 削除

§6 ローカル研究用ファイル

以下のファイルは GCS には保存されず、ローカルディスク上にのみ存在する研究・パイプライン用データ。 5gen GCS 保存成功を起点とするトリガーで自動生成される。

ファイル パス 生成タイミング 生成元
5gen ローカル JSON data/local/horse_pedigree_5gen/{pfx}/{horse_id}.json horse_pedigree_5gen の GCS save 成功直後 _update_local_pedigree_10gen()
10gen ローカル JSON data/local/horse_pedigree_10gen/{pfx}/{horse_id}.json 5gen ローカル JSON 書き込み直後 build_10gen_record() + _update_local_pedigree_10gen()
ped_tbl Parquet data/features/horse/ped_tbl/{pfx}/{horse_id}.parquet 5gen ローカル JSON 書き込み直後 sync_ped_tbl_for_horses()

10gen 構築ロジック

GCS read なし。起動時点でローカルにある gen-5 祖先の 5gen JSON を参照して mini_idx を組み、 build_10gen_record(horse_id, mini_idx) で 10 世代分の ancestors を展開する。

  • gen 1–5 は取得済みの 5gen データをそのまま使用。
  • gen 6–10 は gen=5 の祖先馬の 5gen データを接木 (via フィールドで追跡可能)。
  • ローカルに存在しない gen-5 祖先は coverage.missing_5gen_data に記録され、10gen は部分的に生成される。
  • 一括バッチ補完: python -m src.research.pedigree.build_horse_pedigree_10gen --skip-existing

その他の一括生成スクリプト

スクリプト用途出力
python -m src.scraper.horse_pedigree_5gen_bulk mirror-local GCS から全 5gen を一括ダウンロード data/local/horse_pedigree_5gen/
python -m src.research.pedigree.build_horse_pedigree_10gen 全馬の 10gen 一括ビルド data/local/horse_pedigree_10gen/
