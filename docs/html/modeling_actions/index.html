次にやれること — 議論の整理

docs/modeling までの設計は主軸が揃っている。 ここでは「まだ文書・契約・運用で詰められる余地」と「実行可能なアクション」を優先度付きで並べ、 HTML に落とすのに向く範囲を明示する。

P0 文書整合 P1 データ契約 P2 運用・実験 本ページ = ダッシュボード チューニング SOP → md 正本

前提（すでに十分な層）

  • race 単位 split・purge/embargo・OOF TE・LayerA/B・多段 Stage0–7
  • 一次 split（2020–2023 / 2024 / 2025）と dataset_split_manifest.json
  • ランキング（LambdaRank 系）・深いアンサンブル・NN 探索軸（modeling_coding_strategy.html）
  • 馬券は safe_ev・二次評価（betting_backtest_dataset_spec.html）
以降は「モデル式を増やす」より、train–serve 一致・ラベル定義・多頭確率の一貫性・誤差相関など、 確率を本番と馬券に載せるときに効く層が中心になる。

モデルチューニング工程 — 議論は終わっているか

いいえ、専用ドキュメントまで含めて初めて輪郭が閉じる段階にした。 split・キャリブ・リークは別文書で厚い一方、探索手順・試行予算・凍結ルールは hyperparameter_tuning_workflow.html で初めて SOP 化した。

工程を最適化する主な打ち手（要約）

方針内容
データ境界チューニングは 2024 valid または inner purged CV のみ。2025 test は報告専用。
二層評価inner で多数 trial → 上位のみ outer（2024 全体）で sanity。過適合を減らす。
粗→細年サブセット／少 round でスクリーニング → 本番 budget。
多保真度early stopping・ASHA/Pruner・fold 行列のキャッシュで壁時計を削る。
凍結順母表・TE recipe 固定 → 単一モデル族 → 他族 → ランキング枝（別 study）→ メタ層。
指標1 study につき primary メトリック 1 つ（logloss か NDCG@3 等）を宣言。

詳細・チェックリスト・成果物パス規約はマークダウン正本: docs/html/modeling/hyperparameter_tuning_workflow.html（§10 現行実装とのギャップに pipeline/trainer.py / ensemble_trainer.py の対照表あり）。

実装を踏まえた注意（要約）

  • ModelTrainer.train は 行インデックスの割合分割。日付ソート・race 単位・2024/2025 分離は コード上は未保証。
  • LambdaRank ラベルは現状 複勝圏二値。ドキュメントの relevance ベース LTR と別ラインとして扱うか、列を追加して揃える。
  • n_optuna_trials はログ用が中心で、探索本体は未接続のため SOP の「study」とは別物。
  • EnsembleTrainer は GroupKFold(race_id) だが purge/embargo なし。メタは train 内 OOF 上で fit。

やれること（優先度イメージ）

P0 — すぐ着手しやすい（コスト小・効果大）

  • ドキュメントとコードの接続：advanced_modeling_techniques.html は valid/test の扱いを更新済み。次は pipeline/trainer.py の分割を manifest 準拠に寄せる（§10 マイルストーン）。
  • ラベル定義 1 枚：同着・降着・除外・取消後の field_size と is_win / is_top3、 LTR の relevance を horse_pre_race_dataset_spec.html または短い label_spec.html に集約。
  • T-10 とスナップショット：snapshot_phase・prediction_timestamp と学習母表の列の対応を Stage0 仕様に明記（train–serve skew 防止）。

P1 — 実装に直結（中コスト）

  • 多頭確率の一貫校正：p_win ≤ p_top2 ≤ p_top3 を壊さない校正方針（単一 logits＋累積、または制約付き校正）を advanced_modeling_techniques.html または betting 仕様に追記。
  • 多段の誤差伝播：前段の confidence / coverage を下流特徴に渡す、または単段 strong baseline を常時並走して監視（staged_probability_pipeline.html に「診断」節）。
  • ランキング OOF の保存規約：既に命名例あり → runner の契約（入力列・group・relevance）を JSON schema か README 1 本に固定。

P2 — 運用・評価の厚み（長期）

  • ROI の分散：同一レース内予測誤差の相関を前提に、ブートストラップやレース単位リサンプリングで信頼区間（betting_backtest_dataset_spec.html 注記）。
  • 実験トラッキング：run id・seed・ハイパラ・artifact パスを data/meta/modeling/runs.jsonl 等で統一（MLflow 不要でも可）。
  • プロダクト境界：自動投票・配信・スクレイピング利用範囲は技術ドキュメント外のポリシー文書に一行でも参照リンク。

HTML としてまとめられるもの（整理）

静的 HTML に向くのは 構造化されたチェックリスト・ロードマップ・参照表。生データの分布は Parquet から都度再生成する方が正なので、HTML には 指標名とファイルパス・優先度まで載せ、数値は必要時にスクリプトで更新するのがよい。

内容推奨の載せ方既存 / 新規
パイプライン全体・Stage・データ契約 1 枚のナビ付きリファレンス 既存 ml_pipeline/index.html
レースパフォーマンス算出・運用 指標カード・Mermaid・ベンチ結果 既存 race_performance/index.html
次アクション・議論の整理・優先度表（本ページ） チェックリスト・P0–P2・ロードマップ・参照リンク 新規 modeling_actions/index.html
年次 EDA 数値（行数・NDCG 等） HTML にハードコードせず、生成レポート or マークダウン + 定期ジョブ 例: train_valid_test_split_strategy.html ＋ CI/手動更新
全マークダウンの全文ミラー 重複保守になるため非推奨。差分は md を正とし HTML は索引・要約に限定 —
