レース直前確率モデルに使う実戦的 ML テクニック

ソース: docs/html/modeling/advanced_modeling_techniques.html · 生成: 2026-05-20 06:01 UTC

レース直前確率モデルに使う実戦的 ML テクニック

目的

このドキュメントは、競馬の着順予測モデル（tabular モデル）の次フェーズで活用する機械学習テクニックを、実戦投入を前提に整理したものです。

Kaggle 系の tabular 開発で一般的な手法を、競馬データへ適用する際の具体的なルールまで含めて定義します。

基本方針

  • テクニックは積極的に使うが、リーク防止（過去の正解データが学習に混入しないこと）を最優先にする
  • まず BaselineV1（最初の安定版モデル）を完成させてから、テクニックを段階的に追加する
  • 追加したテクニックは必ず train 内 OOF と 2024 valid（検証データ） で比較・選定する。設定を凍結したあとに限り、2025 test（最終評価データ） で最終報告する（test の結果を見ながら設定を変えない）。data/meta/modeling/dataset_split_manifest.json の一次プロトコルに合わせる
  • 実装基盤は polars 中心で組み、OOF / stage 出力は fold 単位の Parquet に保存する

実装時のデータ処理・メモリ戦略は docs/html/modeling/modeling_coding_strategy.html を参照してください。

モデルの設定値（ハイパーパラメータ）探索・inner/outer 評価・試行予算の配分は docs/html/modeling/hyperparameter_tuning_workflow.html を参照してください。本書はテクニック集であり、運用手順（SOP）ではありません。

新規特徴量の探索候補は docs/html/modeling/feature_engineering_exploration.html（LayerA 前提・検証プロトコル付き）を参照してください。

1. カテゴリ変数エンコーディング戦略

1.1 基本カテゴリ処理

まずはリスクが低く効果が安定した定番手法を常設します。

手法名内容
frequency encoding出現頻度への置き換え
count encoding出現回数への置き換え
rare label grouping出現頻度が低いカテゴリをまとめる
label encodingカテゴリを整数 ID に変換

主な対象列:

列名内容
jockey_name騎手名
trainer_name調教師名
sire父馬名
dam_sire母父馬名
venue_surface_distance_key競馬場・馬場・距離の複合キー
venue_era競馬場の時代区分

1.2 target encoding

カテゴリごとの的中率などを数値に変換する手法で、種類が多いカテゴリに特に有効です。

ただし、全件の平均値をそのまま使うのはリーク（正解データの混入）となるため禁止します。

ルール

  • 必ず OOF（交差検証） の枠組みで作成する
  • fold ごとに、train 側のデータだけでカテゴリ平均を計算する
  • valid 側には、その train 側の統計値だけを適用する
  • test 側は全 train データの統計値のみを使う
  • 出現回数が少ないカテゴリは全体平均に引き寄せる（shrink）

主な候補

カテゴリターゲット意味
jockey_nameis_win騎手ごとの勝率
trainer_nameis_top3調教師ごとの複勝率
sireis_win父馬ごとの勝率
dam_sireis_top3母父馬ごとの複勝率
jockey_name x venue_codeis_top3騎手×競馬場の複勝率
trainer_name x surface x distance_bucketis_win調教師×馬場×距離帯の勝率

保存する派生列の例

列名説明
te_jockey_win騎手の勝率エンコード
te_trainer_top3調教師の複勝率エンコード
te_sire_win父馬の勝率エンコード
te_jockey_venue_top3騎手×競馬場の複勝率エンコード

1.3 CatBoost-style ordered target encoding

Kaggle や tabular 実戦で安定しやすい、時系列を考慮した target encoding の代表的な手法です。

過去のレースデータを順番に積み上げながら統計を計算するため、未来の情報が混入しません。

使いどころ

  • 出走順・開催日順に累積できるカテゴリに適用する
  • 対象列: jockey_name, trainer_name, sire, horse_id

ルール

  • race_date の昇順で並べてから計算する
  • 同一サンプルよりも未来の正解ラベルを使わない
  • レース単位のグループをまたがないよう注意する
  • OOF TE と ordered TE の両方を試し、valid スコアで比較する

1.4 複数統計量 target encoding

平均（mean）だけでなく、ばらつき（std）・件数（count）・偏り（skew）を組み合わせると、モデルに渡せる情報が増えます。

候補となる統計量

統計量説明
target mean平均（標準的なターゲットエンコード）
target std標準偏差：ばらつき
target count件数（サンプル数）
top3 rate複勝圏率
multi-target encodingwin/top2/top3 の複数の正解ラベルで同時にエンコード

列名の例

列名説明
te_jockey_win_mean騎手の勝率エンコード（平均）
te_jockey_win_std騎手の勝率エンコード（標準偏差）
te_trainer_top3_mean調教師の複勝率エンコード（平均）
te_sire_top2_mean父馬の連対率エンコード（平均）

2. 数値特徴量の実戦加工

2.1 log / clipping / winsorization

競馬データには極端に大きな値が含まれる列が多いため、そのままモデルに入れず変換してから比較します。

具体的には、対数変換（log）・上限カット（clipping）・外れ値の圧縮（winsorization）が有効です。

変換対象の候補列:

列名内容
career_runs通算出走回数
days_since_last前走からの間隔日数
avg_interval平均出走間隔
training_time_4f追い切りの4ハロンタイム
oikiri_best_lap追い切りのベストラップ

2.2 レース内相対特徴

競馬の着順予測では、各馬の絶対値より同一レース内での相対的な位置が重要です。

レース内で相対化すると、コースや馬場の影響を除いた純粋な比較ができます。

相対化の方法

手法説明
rankレース内での順位
percentileパーセンタイル（相対的な位置）
z-score平均・標準偏差で正規化した値
gap to bestレース内最良馬との差
gap to medianレース内中央値との差

相対化対象の候補列:

列名内容
speed_max最高速度
speed_avg平均速度
speed_courseコース指数
body_weight馬体重
oikiri_best_evaluation追い切り評価
paddock_scoreパドックスコア

2.3 交差特徴（interaction / crosses）

2つの列を組み合わせた交差特徴を、競馬のドメイン知識に基づいて限定的に追加します。

たとえば「この騎手はこの競馬場で特に強い」という関係を数値で表現できます。

優先する組み合わせ

組み合わせ意味
jockey x venue騎手×競馬場
trainer x venue調教師×競馬場
sire x surface父馬×馬場種別
distance_bucket x surface距離帯×馬場種別
venue_era x surface競馬場時代×馬場種別
bracket_number x venue_surface_distance_key枠番×コース条件

3. 学習・検証テクニック

3.1 Purged Group Time Series Split（時系列対応の交差検証）

通常の KFold（ランダム分割）では、近い日付のレース情報が train と valid に混在してリークが起こりやすい問題があります。

そのため、時系列順 + グループ分割（race 単位）を前提とします。

ルール

  • 分割の単位は race_id（1レースを丸ごと1つのグループとして扱う）
  • 時系列の基準は race_date
  • purge window（直近のレースを学習から除外する期間）を設ける
  • embargo（検証直後の期間も学習から除外）を設ける

目的

  • 近い日付のレース間での統計量リークを防ぐ
  • OOF target encoding が過大評価されるのを防ぐ

3.2 adversarial validation（分布シフトの検出）

学習データと検証・評価データの分布がずれていないかを確認するための手法です。

train と valid（または test）を「どちらのデータか」で分類し、AUC（予測順位の精度指標、1に近いほど良い）が高いほど分布がずれています。

dataset_split_manifest.json の一次プロトコルに合わせ、次の 2系統を使い分けます。

  1. train(2020-2023) vs valid(2024) — 特徴選抜・校正前の健全性確認（sanity check）として使う。
  2. train(2020-2023) vs test(2025) — 将来のホールドアウトデータが過去とどれだけ離れているかを早期に把握する。
    ※ 2020-2024 を train に含めて 2025 と比較すると、2024 が valid 設計と混線するため、上記の切り方を推奨する。

使いどころ

  • 2024 / 2025 が過去の train データとどれだけ異なるかを確認する
  • venue_era（競馬場の時代区分）、cushion（クッション値）、paddock（パドックスコア）などの分布変化（drift）を発見する

期待する出力

  • adversarial_auc（分布の乖離度：0.5 に近いほど分布がそろっている）
  • drift が大きい特徴量の上位一覧
  • drift が強い segment（馬場・距離帯など）の一覧

3.3 calibration（確率の補正）

モデルが出力する確率値を、実際の的中率に近づける後処理です。

着順予測モデルでは最重要の工程です。予測確率が正確でないと、期待値計算や馬券判断の精度が下がります。

候補手法

手法名説明
temperature scaling出力を一定係数でスケール調整する最もシンプルな方法
isotonic regression単調な非線形変換で補正
Platt scalingロジスティック回帰で確率を再校正

ルール

  • calibration は必ず OOF または valid の予測に対してのみ fit する
  • test データに対しては fit しない
  • p_win（勝率）、p_top2（連対率）、p_top3（複勝率）それぞれで補正・比較する

4. モデル戦略

4.1 単体モデル

いずれも GBDT（木を積み上げた機械学習モデル）です。まず1つを安定させてから比較します。

モデル特徴
LightGBM高速・省メモリで大規模 tabular に強い GBDT
CatBoost高カーディナリティカテゴリに強い GBDT
XGBoostGBDT の別実装。多様なランキング目的関数を持つ

推奨する順番

  1. LightGBM baseline（最初の基準モデル）
  2. CatBoost baseline（2番目の基準モデル）
  3. 2つの OOF blending（両者の予測を組み合わせて精度を上げる）

4.2 マルチモデル アンサンブル（複数モデルの組み合わせ）

単一モデルより、複数モデルの予測を OOF ベースで組み合わせる アンサンブル（複数モデルの組み合わせ）の方が着順予測の精度が安定しやすいです。

最初の組み合わせ案

  • blend_p_win = 0.5 * lgbm + 0.5 * catboost（勝率予測の平均）
  • blend_p_top2 = weighted average（連対率予測の加重平均）
  • blend_p_top3 = weighted average（複勝率予測の加重平均）

次のステップ

  • スタッキング（モデルの予測を別モデルで再学習）を導入する場合も、スタッカーの学習は必ず OOF 予測だけで行う

4.3 確率モデルとランキングモデルの二層構成

競馬ではレース内の着順が直接の評価軸であるため、確率モデルとランキングモデルを組み合わせる設計が有効です。

構成案

モデル役割出力
ModelA（主）確率モデルis_win、is_top2、is_top3 の確率
ModelB（補助）ランキングモデルレース内のランキングスコア。タイブレークや予測の安定化に利用

4.4 多段階パイプライン

すべての特徴量を1つの学習器に入れるのではなく、「中間予測モデル → 最終確率モデル」の順で段階的に学習します。

各ステージが専門的な情報を担当するため、最終モデルが学習する内容をシンプルに保てます。

推奨ステージ構成

  1. HorseLatentState（馬の潜在状態：体調や能力の推定）
  2. PositioningModel（脚質・位置取りの予測）
  3. PaceScenarioModel（レースのペース展開の予測）
  4. DayTrackBiasModel（当日の馬場バイアスの予測）
  5. ContextualBloodlineFitModel（血統とコース条件の適性スコア）
  6. FinalProbabilityModel（最終的な着順確率の出力）

ルール

  • 各ステージの出力は必ず OOF（交差検証外のデータで生成）して次のステージに渡す
  • 次ステージの学習時に、train fold の中に valid fold の出力を混ぜない
  • スタッキング（モデルの予測を別モデルで再学習）時も OOF only のルールを守る

4.5 補助タスク（疑似ターゲット学習）

最終的な着順だけを学習するのではなく、途中の状態（コーナー順位やペースクラスなど）を補助的に学習すると、モデルの安定性が上がります。

補助ターゲットの候補

列名内容
first_corner_rank_norm1コーナー通過順位
position_deviation位置取りのばらつき
lap_time_1f1ハロンのラップタイム
lap_time_3f3ハロンのラップタイム
pace_classペース区分（先行・差し・追い込みなど）
day_bias_front_like当日の先行馬有利バイアス
day_bias_closing_like当日の差し・追い込み有利バイアス

目的

  • 着順確率の前段にある中間的な構造をモデルに学ばせる
  • 特徴量ブロックごとの寄与を分解しやすくする

5. 特徴量の重要度（どの数値が予測に効いているか）の確認・解釈

5.1 SHAP（各特徴量の予測への寄与度を示す手法）

各特徴量が個々の予測にどれだけ影響しているかを定量的に確認できます。

  • 着順予測に効いている特徴量の上位確認
  • 分布シフト（drift）が起きたときの原因分析
  • 競馬場ごとに特徴量の効き方がどう変わるかの差分確認

5.2 permutation importance（順番入れ替え重要度）

特徴量の値をシャッフルして予測精度の低下幅を測ることで重要度を測る手法です。

  • SHAP の順位とずれる列を見つける（両者で順位が違う場合は要注意）
  • ノイズが多く予測に貢献しない特徴量（noisy feature）の削除判断に使う

5.3 ablation study（特徴量ブロックの ON/OFF 比較）

特徴量を意味のあるグループ（ブロック）単位で有効/無効を切り替え、各ブロックの貢献度を測ります。

比較対象のブロック:

ブロック名内容
base race info基本レース情報
history aggregate過去成績の集計
race_indexレース指数
pedigree血統情報
oikiri追い切りデータ
paddockパドック評価
trainer_comment調教師コメント
cushion / seasonal contextクッション値・季節情報
target encoding blockカテゴリのターゲットエンコード

6. 期待値最適化向けの追加テクニック

6.1 safe expected value（安全な期待値）

高オッズ帯では予測誤差の影響が大きくなるため、バッファを引いた保守的な期待値（safe_ev）を使います。

  • raw_ev = p * odds - 1（生の期待値：勝率×オッズ − 1）
  • safe_ev = calibrated_ev - buffer（補正後の期待値からバッファを差し引いた安全な期待値）

6.2 policy search（馬券購入ルールの最適化）

予測モデルと馬券購入ルールを切り分け、ルール側のパラメータを別途探索します。

モデルの再学習なしに馬券戦略だけを調整できるため、過学習のリスクを減らせます。

探索するルールパラメータ

パラメータ名説明
min_safe_ev購入する最低期待値
max_odds購入するオッズの上限
min_model_confidence最低モデル確信度
min_rank_gap1位と2位の確率差の下限
max_tickets_per_race1レースあたりの最大購入点数

最初に比較する馬券種別

  • 単勝（1着を的中させる）
  • 複勝（3着以内を的中させる）

7. 実装時の優先順

安定した基盤から順番に積み上げます。スキップせず上から順に進めてください。

  1. frequency/count encoding（出現頻度・件数によるエンコード）
  2. OOF target encoding（交差検証を使った的中率エンコード）
  3. CatBoost-style ordered encoding（時系列順の累積エンコード）
  4. purged group time series split（時系列対応の交差検証）
  5. calibration（確率の補正）
  6. positioning / pace の補助モデル（脚質・ペース展開の中間予測）
  7. day bias / contextual bloodline の補助モデル（当日バイアス・血統適性の中間予測）
  8. LightGBM + CatBoost OOF ensemble（複数モデルの OOF アンサンブル）
  9. safe_ev と policy search（期待値ベースの馬券ルール最適化）

8. やらないこと（禁止事項）

禁止事項理由
train/valid の split 前に target encoding を作るリーク（正解データの混入）が発生する
test の結果を見ながらエンコーディング設計を変える評価が汚染され再現性が失われる
市場オッズを LayerA（レース直前の特徴量層）に混ぜるモデルがオッズに依存してしまう
高次交差特徴を検証なしに無制限に増やす過学習のリスクが高まる
ROI（投資回収率）だけを基準にモデルを採択する再現性がなく過学習しやすい
