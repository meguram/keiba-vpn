1. ベースラインを先に固定する

Speaker Deck と法政大の資料では、まずシンプルな指数中心のベースモデルを作り、そこから特徴量（予測に使う数値）を段階的に追加・比較する流れが有効であることが示されていました。

採用内容説明
BaselineV1 の定義最初の比較基準となるシンプルなモデル
ExtendedV1 を別出力として用意追加特徴量を積み上げたモデル
差分評価追加特徴量の効果は BaselineV1 との差分で評価

2. 確率の信頼性を ROI より先に評価する

Speaker Deck では、期待値が 1 を超えていても、高オッズ帯ではわずかな確率誤差で簡単に損失になることが強調されていました。logloss や Brier score のような確率の信頼性指標が特に重要とされていました。

評価優先度指標位置づけ
一次log_loss, Brier score, ECE確率の信頼性評価
二次ROI収益評価（後段）
安全域safe_ev（安全域付き期待値）, buffer高オッズ帯での判定バッファ

3. 時系列順での分割（TimeSeriesSplit）を採用する

法政大資料と Speaker Deck の両方で、競馬モデルでは時間順での検証が不可欠であることが示されていました。

設定内容
期間分割2020-2023 train / 2024 valid / 2025 test
fold 方式rolling-origin fold
禁止事項行単位のランダム split は禁止
encodingカテゴリ encoding や集計は train fold 内のデータのみで作成

4. 1着・2着以内・3着以内を分けて持つ

Qiita の 140% 記事では、1着指数・2着以内指数・3着以内指数を並行して扱う設計が示されており、本プロジェクトの目標とも合致していました。

種別ラベル確率出力
1着is_winp_win
2着以内is_top2p_top2
3着以内is_top3p_top3

5. 継続開発を前提に成果物をバージョン管理する

Qiita の dijzpeb 記事では、単発の notebook で終わらせず、ソースコード・データ・モデル・実験結果を整理して継続的に改善できる形にする重要性が強調されていました。

採用内容説明
data_version, code_version, source_table_versions成果物ファイルにバージョン情報を保存
dataset_profile.json, feature_missingness.csv, label_balance.json監査成果物として出力
bet_policy_thresholds.json閾値ルールを外部設定ファイルで管理

6. EDA と欠損・外れ値の点検を必須にする

法政大資料と Speaker Deck では、特徴量を設計する前に分布・欠損・相関・外れ値を点検することが重要だと示されていました。

必須成果物内容
dataset_profile.jsonデータ全体の分布・基本統計
feature_missingness.csv特徴量ごとの欠損率
segment_coverage.csvセグメント別カバレッジ
outlier_watchlist.csv外れ値の監視リスト

7. 市場情報はモデル本体と意思決定層で分離する

Qiita の 140% 記事は「オッズを特徴量に使わない能力モデル」を示し、Speaker Deck は「期待値計算には市場価格が必要」と示していました。この相反する要求を整理するため、以下のように 2 層に分けました。

レイヤ役割オッズの扱い
LayerA能力・状態推定モデルオッズに依存しない
LayerB意思決定・馬券最適化レイヤオッズを使う

8. OOF target encoding を正式採用する

Kaggle の実践では、target encoding は強力である一方、データリークを起こしやすいため、OOF（Out-of-Fold）方式を前提にすることが定石です。また、出現が少ないカテゴリは全体平均に縮小（shrink）し、未知カテゴリにはフォールバックを持たせることが一般的です。

採用内容説明
OOF target encodingfold 外データのみで encoding を計算
CatBoost-style ordered target encoding時系列順序を保持した encoding
OOF aggregate features集計特徴量も OOF 方式で生成
encoding_recipe_version再現性のためのバージョン管理

9. Purge / embargo 付き CV を採用する

Kaggle の時系列・非 IID タスクでは、通常の KFold より PurgedGroupTimeSeriesSplit の方が安全であるという知見が多数あります。近い時点のデータが学習・検証に同時に入ることで生じるリークを防ぐための仕組みです。

設定内容
cv_policy.primary_schemepurged_group_time_series_split
purge_window_days学習・検証に同時利用しない期間（除外ウィンドウ）
embargo_days学習後の待機期間
OOF 特徴量purge / embargo のルールに従う

10. CatBoost / LightGBM の 2 モデル比較を行う

Kaggle のテーブルデータ実践では、カテゴリ変数の処理に強い CatBoost と、高速で拡張しやすい LightGBM（いずれも GBDT、木を積み上げた機械学習モデル）の比較とアンサンブルが定番とされています。

採用内容説明
LightGBM baseline高速で拡張しやすい GBDT
CatBoost baselineカテゴリ変数の処理に強い GBDT
OOF blending複数モデルのアンサンブル

11. 期待値の安全域をルール化する

Speaker Deck と Qiita の実践例では、高オッズ帯での期待値判定はわずかな確率誤差に弱く、そのまま買い続けると破綻しやすいことが繰り返し示されていました。

採用内容説明
safe_ev（安全域付き期待値）確率誤差を考慮した安全域付き期待値
buffer（判定バッファ）高オッズ帯での判定バッファ
bet_policy_thresholds.json閾値の外部ファイル管理

12. 追加特徴量は必ず差分比較する

Kaggle の実践では、「手法を追加したら必ず OOF / holdout で差分比較する」流れが徹底されています。思い込みによる過剰な特徴量追加を防ぐために重要なプロセスです。

採用内容説明
BaselineV1 / ExtendedV1 の 2 本立て追加特徴量の効果を差分で測定
ablation study特徴量ブロック単位の寄与測定
feature block ごとの ON/OFF 比較各特徴グループの単独寄与を検証
adversarial validation学習・テスト分布のずれ検出

13. 位置取りを前段モデルで予測する

既存の src/pipeline/tracking_difficulty.py には、脚質プロファイル・隣枠影響・頭数・コース条件から position_deviation（位置取りのずれ）を学習する設計があります。

追加特徴量説明
expected_first_corner_pos予測される第1コーナー通過順
tracking_difficulty_score位置取りの難しさスコア
front_run_probability / stalker_probability / closer_probability脚質確率（逃げ / 番手 / 差し）

位置取りモデルを最終確率モデルの前段に組み込む。

14. ペースを前段モデルで予測する

既存の src/pipeline/pace_predictor.py には、距離・馬場・頭数・前に行きたい馬の比率からラップタイムを予測する実装があります。

追加特徴量説明
pred_lap_1f / pred_lap_3f予測ラップタイム
pred_pace_class予測ペース分類（ハイ / ミドル / スロー）
pred_energy_distributionエネルギー分配の予測

ペースシナリオを Stage3 として独立したモジュールに分ける。

15. 血統適性を当日のコースバイアスと組み合わせる

既存の src/research/bloodline_distance.py は父・母父と距離適性の関係を整理しており、src/research/race_quality_model.py は「そのレースでどの脚質タイプが好走したか」をベイズ推定で計算しています。この 2 つを組み合わせると、血統適性（静的な距離適性）とコースバイアス（当日の馬場タイプ）の相互作用として血統を扱う設計になります。

追加した考え方説明
day prior + same-day posterior事前分布（血統） + 当日更新（馬場）のベイズ的組み合わせ
contextual_bloodline_fit文脈に応じた血統フィットスコア
same_day_sire_cluster_bias当日の父系クラスターバイアス
bloodline_day_fit_speed/stamina/mud/burstスピード / スタミナ / 道悪 / 瞬発別の血統適合度

16. Polars 中心の省メモリ実装を前提にする

現在のデータ量と WSL 環境では、全年・全列を pandas で一括処理するとメモリ不足や不安定な動作が起きやすくなります。

採用内容説明
polars scan_parquet() を第一選択遅延評価でメモリ効率が高い
fold 単位で materializefold 単位でメモリ上に展開する
OOF / stage 出力は都度 Parquet に保存中間成果物をファイルで管理
float32 / int8 への型削減メモリ使用量を削減

まとめ

参考資料の知見を反映した結果、設計は「何を作るか」だけでなく、以下まで含む形に強化されました。

強化された観点具体的な内容
どう比較・検証するかBaselineV1 / ExtendedV1 の 2 本立て、ablation study
どの指標で良し悪しを判断するかlogloss / Brier score / ECE を一次評価、ROI は二次評価
どこでリークや過大評価が起きやすいかtarget encoding の OOF 化、time series split、purge / embargo
継続的に改善できる構造にするにはどうするかバージョン管理、監査成果物の出力、外部設定ファイル
