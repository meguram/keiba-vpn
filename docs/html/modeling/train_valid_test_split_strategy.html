Train / Valid / Test 分割戦略（レース直前確率モデル）

1. 目的

このドキュメントは、通常の i.i.d.（各サンプルが独立・同分布）想定の機械学習とは異なる競馬データに対して、以下を整理する。

  • どの期間を train / valid / test に割り当てるか
  • データリーク（未来の情報を誤って使うこと）・同一開催週の相関・改修・制度変更をどう扱うか
  • 分割を決める前に、母集団の分布をどう把握するか

dataset_split_manifest.json および既存のモデリング設計書と整合する形でまとめている。

2. なぜ「ランダム split」が通用しないか

課題 説明
時間の因果 未来のレース・成績が過去の行に混入するとデータリーク（未来の情報を誤って使うこと）になる。
グループ相関 同一 race_id 内の馬行は互いに独立ではない。分割単位は必ずレース単位にする。
開催週の連続性 日付が近いレースは馬場・天候・メンバー強度が似る。train と valid が「接しすぎ」ないよう purge / embargo（除外期間）が必要。
分布シフト コロナ以降の開催密度、データ取得カバレッジ、指数・パドックの提供範囲、京都競馬場改修後の venue_era など、年によって特徴量の意味が変わる。
希少カテゴリ 重賞・特定コースはサンプル数が少ない。ランダム分割では valid に偶然偏りが生じやすい。

以上の理由から、行単位のランダム split は禁止（マニフェスト constraints.no_random_row_split と一致）。

3. 一次プロトコル（モデル選定・本番前評価）

data/meta/modeling/dataset_split_manifest.json の protocols.model_selection_primary を採用する。

役割 年（暦年・race_date 基準） 用途
Train 2020, 2021, 2022, 2023 学習本体・（fold 内）ターゲットエンコーディング（カテゴリを予測対象の平均値に変換）/ 集約特徴量の学習
Valid 2024 モデルの設定値（ハイパーパラメータ）・特徴量選択・確率補正の fit（test は見ない）
Test 2025 一度きりの最終報告・本番前の汎化チェック
  • train / valid / test は race_id 単位で完全分離する。
  • 確率補正（Calibration）（温度スケール・アイソトニック等）は、交差検証での予測（OOF）または valid のみに fit する。test では fit しない。

3.1 本番再学習（モデル確定後のみ）

protocols.production_refit_after_selection に従い、選定完了後の本番用再学習では 2020〜2025 を train に含められる。これは「test で評価を報告した後」に行う手順であり、test 指標を報告する前に実行しない。

4. 交差検証（CV）— モデル開発の主戦場

一次プロトコルの valid は 2024 年の 1 本だけのため、開発中の安定した精度推定には rolling-origin + purged group time series の交差検証（CV）を併用する。

CV ポリシー（マニフェスト cv_policy）

パラメータ値説明
group_keyrace_idレース単位でグループ化
time_keyrace_date時系列の基準
purge_window_days7 日同一開催週付近の情報漏れを抑える保守的初期値
embargo_days3 日validation 直前の train を除外し、ターゲットエンコーディング・集約特徴量のデータリークを抑える

rolling_origin_folds は 2020-01-01 起点で valid を半年ずつずらす定義になっている。実装時は母表の race_date と照合し、fold ごとに各レースが片側にのみ属することを自動テストする。

5. 補助評価セット（セグメント・ドリフト分析）

マニフェストの special_sets および segment_evaluations を活用する。

  • 京都改修後 (kyoto_post_renovation_*): 現行コース条件への追従性を検証
  • クッション値観測あり (cushion_observed_test): 数値馬場特徴量の効き具合を検証
  • パドック情報の有無 (with_paddock_test / without_paddock_test): ExtendedV1 の依存度を検証

全体の logloss / Brier / ECE に加え、セグメントごとに性能と確率補正（キャリブレーション）の状態を確認する。

6. 他ドキュメントの「2020–2024 vs 2025」との関係

race_performance_rating_system.html では、レースパフォーマンス特徴量（予測に使う数値）単体の寄与検証として、以下の切り方を採用している。

  • 履歴構築・学習側: 2020–2024
  • ホールドアウト（テスト用として取り分ける）: 2025（かつ 2025 は online-safe 算出）

これはモジュール単体ベンチ用の切り方であり、一次の確率モデル分割（2020–2023 / 2024 / 2025）とは目的が異なる。両方を併存させる。

  • 確率モデル本線: 本書 §3 のマニフェスト準拠
  • パフォーマンス信号の感度分析: 上記ベンチ（長い train で信号の安定性を確認する用途）

混同しないよう、実験ログに protocol_name を必ず記録する。

7. 分布把握（EDA）— 実データに基づくサマリ

horse_pre_race_dataset が完成する前でも、同一カレンダー範囲・JRA 中央・レース単位の派生表として data/features/race_performance/20xx.parquet（1行 = 1出走頭）が揃っている。ここから 年次の規模・ラベルバランス・主要カテゴリ・数値列のドリフトを把握した（集計日: リポジトリ上の Parquet 更新時点）。

クラス比率は年間を通じてほぼ安定しており、時系列データの分割（未来のデータでテストする）でも極端なクラスバランスの変化は起きにくい。一方、レース内相関は依然として最大の設計上の制約である。

7.3 カテゴリ分布（全期間合算の上位）

全期間で観測された主なカテゴリのシェア（分割前の母集団把握用）:

カテゴリ分布の傾向
surfaceダート > 芝 >> 障害
distance_bandmile > sprint >> intermediate > long > extended
pace_shape_classbalanced（平均ペース）が大半、次いで grind（消耗ペース）。unknown / burst は少数
class_group未勝利・1勝クラスが中心。G・OP は少数だが重要なセグメント
track_condition良が大半。稍重・重・不良は少数だが馬場補正の検証に必要

重賞などサンプル数が少ないセグメントは、全体指標だけに依存せず segment_evaluations で個別に評価する。

7.4 数値列の年次ドリフト（例: パフォーマンス派生列）

run_performance_final の平均は年によって数ポイント程度変化する（例: 2020 平均 約−21 → 2024〜2025 平均 約−18 付近）。標準偏差は年とともにやや縮小しており、分布が少し締まっている可能性がある。race_level_pre_race の平均も年次で緩やかに低下傾向を示す。

こうしたドリフトは adversarial validation（train vs test の分離容易度を測る手法）で確認する（advanced_modeling_techniques.html と整合）。

track_variant_online は平均は 0 付近だが、最大値側に外れ値（極端に速い日の補正値）が存在する。学習時は winsorize / robust scaling を valid で調整するのが安全。

2025 年の run_performance_final の最小値付近に極端値が見られる。母表の結合後も、結合キー誤り・欠損の埋め方・単位崩れの監査対象として dataset_profile / outlier_watchlist に載せる。

8. 推奨ワークフロー（実装順）

  1. race_date でソートした学習母表を構築する（horse_pre_race_dataset_spec.html 参照）。
  2. マニフェストの primary split でホールドアウト（テスト用として取り分ける）を固定し、2025 年 test には触れない。
  3. purge + embargo 付き rolling 交差検証（CV）で開発する。ターゲットエンコーディング・交差検証での予測（OOF）の中間特徴量すべてに同一ポリシーを適用する。
  4. 2024 valid でモデル・特徴量・確率補正（キャリブレーション）を確定する。
  5. 2025 test で最終報告を行い、special_sets / segment で補助診断を加える。
  6. 成果物として dataset_profile.json, feature_missingness.csv, label_balance.json を保存する（reference_insights.html の監査リスト参照）。

9. まとめ

項目 方針
分割単位 race_id（レース単位）
一次 train / valid / test 2020–2023 / 2024 / 2025（マニフェスト準拠）
開発時の交差検証（CV） Purged group time series + rolling-origin
ランダム行分割 禁止
分布・ドリフト確認 年次 EDA + adversarial validation + セグメント別評価
他用途の split race_performance 単体ベンチは別プロトコルとして明示する

マニフェストのバージョンを更新する際は、本書の該当節と reference_insights.html の相互リンクも一緒に更新する。
