多段階レース直前確率パイプライン設計

目的

このドキュメントは、勝率・連対率・複勝率 を単一モデルで直接予測するだけでなく、位置取り、ペース、当日馬場タイプ、血統適性 といった中間状態を順に推定して最終確率へつなぐ、多段階の処理の流れ（パイプライン）の設計を定義する。

多段階化の目的は「モデルを複雑にすること」ではない。以下の理由で行う。

  • 競馬ドメインに沿った中間変数を明示し、解釈しやすくする
  • 予測が外れたときの原因を段階ごとに分解しやすくする
  • 当日更新できる情報（馬場・オッズ）と事前固定の情報（血統・戦績）を明確に切り分ける
  • 血統を”その日の馬場特性に応じて”使い分ける

基本思想

Stage役割
Stage0レース直前時点の学習母表を作る
Stage1馬単体の能力・状態を要約した潜在表現（latent state）を作る
Stage2位置取り・追走難度を予測する
Stage3レース全体のペースシナリオを予測する
Stage4当日馬場タイプ・レース当日バイアス（race-day bias）を推定する
Stage5血統・適性の当日文脈への適合度を推定する
Stage6Stage1〜5 の出力をまとめて最終確率 p_win / p_top2 / p_top3 を推定する
Stage7市場統合・馬券最適化へ渡す

全体像

flowchart TD stage0[Stage0_PreRaceDataset] --> stage1[Stage1_HorseLatentState] stage1 --> stage2[Stage2_Positioning] stage1 --> stage3[Stage3_PaceScenario] stage1 --> stage5[Stage5_BloodlineContextFit] stage2 --> stage3 stage3 --> stage4[Stage4_DayTrackBias] stage4 --> stage5 stage1 --> stage6[Stage6_FinalProbability] stage2 --> stage6 stage3 --> stage6 stage4 --> stage6 stage5 --> stage6 stage6 --> stage7[Stage7_MarketAndBetting]

Stage0: Pre-race 母表

役割

  • race_shutuba_flat を学習母表の起点にする
  • race_shutuba_past_flat, race_index_flat, race_oikiri_flat, race_paddock_flat, race_trainer_comment_flat を時点整合で結合する
  • GCS horse bundle とクッション値を統合する

出力候補

  • horse_labels.parquet

Stage1: Horse latent state（馬の潜在表現）

役割

馬単体の「現在の地力・適性・仕上がり」を意味のある固まりに要約する。生の特徴量（予測に使う数値）を大量に最終確率モデルへ直接渡すのではなく、解釈しやすい潜在ブロック（latent block）に整理する。

主な潜在ブロック（latent block）

ブロック名意味主な入力情報
ability_latent能力近5走戦績、同条件成績、speed index、通算成績
form_latent仕上がり休み明け、叩き何戦目、馬体重変化、近走トレンド
surface_distance_latent芝・ダート適性芝/ダート・距離帯ごとの成績傾向
pedigree_latent血統sire / dam_sire / myostatin / note aptitude
trainer_jockey_latent陣営交差検証での予測（OOF）によるターゲットエンコーディングや CatBoost ordered encoding で生成した人馬・陣営系の特徴量

出力候補

  • horse_latent_state.parquet

Stage2: 位置取り / 追走難度モデル

目的

対象レースで「その馬がいつもの位置を取りやすいか」を推定する。

既存コードの活用先

  • src/pipeline/tracking_difficulty.py

入力特徴量 / 出力 / 学習ターゲット

区分項目
入力特徴量直近走の通過順傾向、脚質のばらつき、枠順・馬番、隣枠の馬の傾向、出走頭数、コース・距離、馬場状態
出力expected_first_corner_pos, expected_first_corner_pos_norm, tracking_difficulty_score, front_run_probability, stalker_probability, closer_probability
学習ターゲット既存 position_deviation、または first_corner_rank_norm

このステージの価値

  • 位置取りは最終着順の前段変数として解釈しやすい
  • 後段のペースモデルと自然につながる

Stage4: 当日馬場タイプ / race-day bias モデル

目的

「今日はどんな馬が走りやすい日か」を推定する。レース条件の事前情報と、同日先行レースの結果を組み合わせて動的に更新する。

その日によって好走しやすい馬のタイプが変わる、という性質を扱うための中心ステージ。

二層構造

Stage4A: 事前推定（day prior）

対象レースが始まる前に、以下の情報から当日バイアスの事前分布を作る。

  • venue（開催場）
  • venue_era（改修・時代区分）
  • surface（芝/ダート）
  • distance（距離）
  • track_condition（馬場状態）
  • weather（天気）
  • cushion_value（クッション値）
  • moisture（含水率）
  • season（季節）

Stage4B: 同日更新（day posterior）

同日にすでに終わったレースがある場合、その結果から推定を更新する。

  • 早いレースの上位馬の特徴
  • 脚質（前残り・差し）の偏り
  • 内外枠のバイアス
  • 上がり偏重か前残りか
  • 血統クラスタの偏り

出力

カラム名説明
day_bias_front前残り有利度
day_bias_closing差し・追い込み有利度
day_bias_inner内枠有利度
day_bias_outer外枠有利度
day_bias_staminaスタミナ型有利度
day_bias_burst瞬発力型有利度
day_bias_mud道悪適性有利度
day_bias_speed高速馬場適性有利度

実装方針

先行レース数方針
0 本事前推定（prior）のみを使う
1〜2 本prior を重視しつつ、同日結果（posterior）を弱めに反映する
3 本以上posterior の重みを増やす

既存コードの活用先

  • src/research/race_quality_model.py
  • src/research/tune_race_quality_priors.py

Stage5: 血統 × 当日文脈適合モデル

目的

血統を固定した特徴量として使うだけでなく、Stage4 で推定した「今日の好走タイプ」に合わせて血統の適合度を動的に評価する。

中核アイデア

同じ sire（父馬）でも、効果が出やすい条件は日によって異なる。

  • 高速馬場の瞬発戦で強い日
  • 消耗戦で強い日
  • 道悪で強い日

そのため、血統特徴量は「静的な距離適性」だけでなく「当日の race-day bias との相性」で再評価する。

既存コードの活用先

  • src/research/bloodline_distance.py

  • src/research/course_bloodline.py
  • src/research/sire_factor_stats.py
  • src/research/sire_factor_race_map.py
  • src/research/sire_factor_aptitude.py
  • src/research/note_aptitude_5gen.py
  • src/research/sire_aptitude_note.py
  • src/research/myostatin.py
  • src/research/race_quality_model.py

血統特徴量の二層構造

5A: 静的血統適性（固定値）

カラム名説明
sire_distance_fit父馬の距離適性
dam_sire_distance_fit母父の距離適性
surface_fit芝/ダート適性
mud_fit道悪適性
speed_fit高速馬場適性
stamina_fitスタミナ適性
burst_fit瞬発力適性

5B: 当日文脈適合（動的スコア）

Stage4 の出力との掛け合わせで生成する。

  • bloodline_day_fit_speed = sire_speed_fit * day_bias_speed
  • bloodline_day_fit_stamina = sire_stamina_fit * day_bias_stamina
  • bloodline_day_fit_mud = mud_fit * day_bias_mud
  • bloodline_day_fit_burst = burst_fit * day_bias_burst

日単位で血統を使う具体策

1. 同日先行レースの血統クラスタ集計

同日・同場・同 surface の先行終了レースから以下を集計し、当日のクラスタバイアスを作る。

  • どの sire cluster が上位に入ったか
  • どの適性軸（aptitude axis）が強く出たか

2. segment prior + same-day posterior

src/research/race_quality_model.py の segment prior を day prior として利用し、同日先行レースの結果で posterior 補正する。

3. race-wise contextual fit（レース単位の文脈適合スコア）

対象馬について以下の類似度を計算し、contextual_bloodline_fit とする。

  • その馬の血統ベクトル
  • 当日 posterior bias ベクトル

出力

カラム名説明
contextual_bloodline_fit血統の当日文脈総合適合スコア
contextual_bloodline_speed_fit高速馬場文脈適合
contextual_bloodline_stamina_fitスタミナ文脈適合
contextual_bloodline_mud_fit道悪文脈適合
contextual_bloodline_burst_fit瞬発力文脈適合

Stage6: 最終確率モデル

目的

Stage1〜5 を統合し、最終的な p_win（勝率）、p_top2（連対率）、p_top3（複勝率）を推定する。

入力ブロック

入力ブロック由来 Stage
基礎母表Stage0
潜在表現（latent state）Stage1
位置取り出力Stage2
ペース出力Stage3
当日バイアス出力Stage4
血統文脈適合出力Stage5
LayerB 市場特徴量（任意）—

推奨モデル構成

モデル予測ターゲット
ModelAp_win（勝率）
ModelBp_top2（連対率）
ModelCp_top3（複勝率）
ModelD（任意）race-wise ranking score（レース内順位スコア）

推奨アンサンブル

  • GBDT（木を積み上げた機械学習モデル）として LightGBM / CatBoost の 2 本柱
  • 交差検証での予測（OOF）を使った blending
  • 後段で確率の信頼性補正（calibration）

出力

カラム名説明
p_win_raw, p_win_calibrated勝率（補正前/補正後）
p_top2_raw, p_top2_calibrated連対率（補正前/補正後）
p_top3_raw, p_top3_calibrated複勝率（補正前/補正後）
rank_by_p_win勝率によるレース内順位

Stage7: 市場統合 / ベッティング

役割

以下の 4 つのテーブルを統合し、safe_ev（安全期待値）を計算する。

  • horse_probability_table（馬別確率テーブル）
  • horse_market_snapshot（馬別市場スナップショット）
  • pair_market_snapshot（馬連・馬単等の市場スナップショット）
  • payout_table（払い戻しテーブル）

推奨レイヤ（教師あり ML とは分離）

券種・投資配分・予測印と推奨券の結びつけなど、期待値を土台にした系列意思決定は、確率モデルの勾配学習とは切り分ける。強化学習（方策） を第一候補とする設計議論は別紙にまとめた。

  • docs/html/modeling/betting_policy_reinforcement_learning.html
  • HTML 要約: docs/html/appendix/betting_rl_policy.html

設計ルール

ルール内容
市場独立Stage6 までで市場情報に依存しないモデルを完成させる
市場結合の分離市場との結合は Stage7 でのみ行う
高オッズ対応高オッズ帯は buffer（安全マージン）を厚くする

優先実装順

Phase目標対象 Stage
Phase1単段ベースラインを成立させるStage0（母表構築）、Stage1（馬の潜在表現）、Stage6（最終確率）
Phase2位置取り・ペースを追加し改善幅を測るStage2（位置取り）、Stage3（ペース）
Phase3当日バイアスと血統文脈適合を追加するStage4（当日馬場バイアス）、Stage5（血統文脈適合）
Phase4safe_ev ベースで馬券最適化へ進むStage7（市場統合・馬券最適化）

この設計の利点

  • なぜ当たった・外れたかを stage ごとに説明できる
  • 血統を「固定特徴量」ではなく「その日の馬場文脈への適合度」として使える
  • 先行レース結果を使ったオンライン更新に自然につながる
  • 将来的に stage ごとの単独改善が可能
