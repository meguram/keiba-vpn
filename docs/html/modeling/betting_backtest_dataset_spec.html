betting_backtest_dataset 仕様

目的

このドキュメントは、レース直前確率モデルの出力を馬券最適化・過去データを使った検証（バックテスト）に接続するための入力テーブルを定義します。

モデリングの実装は別フェーズで行いますが、次のテーブルを共通契約として先に確定します。

  • 馬単位確率テーブル
  • 市場価格テーブル
  • 結果・払戻テーブル
  • レース制約テーブル
  • バックテスト結合テーブル

設計の基本原則

原則内容
過小な期待値への依存を避ける期待値が 1 をわずかに超えた程度の券だけを機械的に買うと、確率推定誤差により損失に転ぶことがある
高オッズ帯のリスク高オッズ帯では、確率をわずかに過大評価するだけで期待値が大きくずれる
安全域の確保期待値 > 1 だけを条件にせず、校正済み確率 と 安全域 の両方を前提にベット判断を行う
テスト期間での評価ROI の評価は、モデルが学習に使っていない期間（テスト期間）だけで実施する

テーブルの全体構造

flowchart TD horseProb[HorseProbabilityTable] --> backtest[BettingBacktestDataset] marketHorse[HorseMarketTable] --> backtest marketPair[PairMarketTable] --> backtest raceResult[RaceOutcomeTable] --> backtest payout[PayoutTable] --> backtest raceConstraint[RaceConstraintTable] --> backtest

1. horse_probability_table.parquet

確率モデルが出力した各馬の勝率・連帯率・複勝率を格納するテーブルです。

粒度

1行 = 1レース × 1頭

主キー

race_id + horse_number

必須列

列名説明
race_idレース識別子
race_date開催日
venue_code競馬場コード
horse_number馬番
horse_id馬識別子
horse_name馬名
field_size出走頭数
model_name使用モデル名
model_versionモデルバージョン
data_versionデータバージョン
prediction_timestamp予測実行時刻
p_win勝率（単勝確率）
p_top2連帯率（2着以内確率）
p_top3複勝率（3着以内確率）

推奨追加列

列名説明
score_rawモデルの生スコア
score_calibrated校正後スコア
rank_by_p_win勝率順位（1始まり）
rank_by_p_top2連帯率順位
rank_by_p_top3複勝率順位
uncertainty_proxy不確実性の指標
feature_coverage_score特徴量カバレッジスコア

制約

制約内容
確率の合計同一レース内で sum(p_win) ~= 1.0
確率の順序p_win <= p_top2 <= p_top3 <= 1.0
順位列rank_by_p_win は 1 から始まる連番

用途

  • 単勝・複勝・馬単・ワイド・馬連の期待値計算の基礎データ
  • モデル間の比較と確率校正の評価

7.5 期待値安全域ルール

期待値の判定は次の 3 段階で行います。段階を踏むことで、確率推定のブレを安全域として吸収します。

段階指標名計算式説明
1生期待値raw_ev = predicted_prob * odds - 1モデルの生出力ベース
2校正後期待値calibrated_ev = calibrated_prob * odds - 1確率を補正した後
3safe_evsafe_ev = calibrated_ev - buffer安全域込みの最終判定値

初期ルール

  • ベット候補は safe_ev > 0 を最低条件とする
  • buffer は固定値ではなく、odds bucket（オッズ帯）と uncertainty_proxy（不確実性の指標）に応じて変動させる
  • 高オッズ帯ほど buffer を大きく取る

7.6 閾値管理テーブル

将来の戦略比較に備えて、判定に使う閾値は本テーブルとは別に管理します。

bet_policy_thresholds.json

フィールド説明
policy_nameポリシー名（識別子）
snapshot_phase適用するオッズスナップショット時刻
bet_type対象券種
min_safe_evベット判断の最低 safe_ev 閾値

目的

  • 「モデルは同じで閾値だけ変えた」という実験を手軽に行えるようにする

券種別最適化の前提

券種主な入力列備考

複勝p_top3、place_odds_min、place_odds_max初期版は place_odds_avg を使い、後でオッズ幅を考慮した評価へ拡張する
馬連・ワイド・馬単pair_probability_input テーブル（別途 join）単体の勝率・複勝率だけでは期待値を計算できない。初期段階は市場比較と回収率検証に必要な入力のみ整備する

評価のルール

ルール内容
ROI 至上主義の禁止ROI（回収率）だけをモデル採択の指標にしない
校正精度の優先確認まず p_win、p_top2、p_top3 の校正精度を確認する
テスト期間の分離検証は必ずモデルが学習に使っていないテスト期間だけで行い、学習期間と完全に分離する
シミュレーション専用の明示レース確定後の最終オッズでしか評価できない指標は、simulation_only と明示する
オッズ時刻の混同禁止発走10分前のオッズ（T-10）と最終確定オッズを混同しない

配分最適化・予測印と強化学習（関連設計）

券種選択・予測印と推奨券の結びつけ・資金配分は、教師あり確率モデルとは切り分けて、強化学習（報酬を最大化するよう行動を学習する手法）によるポリシー学習で扱う設計方針を採用しています。

詳細な設計議論は次のドキュメントを参照してください。

  • docs/html/modeling/betting_policy_reinforcement_learning.html（行動サイクル・報酬設計・過去データ学習・印のパターン）
  • HTML 要約: docs/html/appendix/betting_rl_policy.html

本書（betting_backtest_dataset_spec）の役割はデータ契約・safe_ev（安全な期待値の閾値）・検証指標の定義です。強化学習の環境は、本書のテーブルをシミュレーターの入出力として使います。

実装の順序

  1. horse_probability_table（確率テーブル）の仕様を確定する
  2. horse_market_snapshot を snapshot_phase=t_minus_10（発走10分前）で整備する
  3. race_outcome_table（結果テーブル）と payout_table（払戻テーブル）を整形する
  4. betting_backtest_dataset（結合テーブル）を生成する
  5. 券種別に expected_value（期待値）列を追加する
  6. safe_ev と閾値設定（policy thresholds）を別ファイルで管理する
  7. 上記が整ってから配分最適化モデルへ進む
