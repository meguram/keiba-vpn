モデリング実装時のコーディング戦略

目的

このドキュメントは、2020-2025 の年次ローカルテーブルと GCS 由来補助データを使って、メモリを極力抑えながら再現性のあるモデリング基盤を実装するためのコーディング戦略を定義します。

特に次の点を重視します。

  • WSL（Windows Subsystem for Linux）上でも安定するメモリ使用量
  • OOF（交差検証外データ）・purged CV（時系列対応交差検証）・多段ステージに対応できる中間成果物の管理
  • polars を中心に据えた lazy 処理（必要な列・行だけを遅延評価で読み込む）
  • 速度より「落ちない」「再現できる」「比較できる」実装を優先する

競馬の「レース内での相対順位」予測には ランキング学習（Learning-to-Rank, LTR） がよく採用されます。

本書では従来の 点推定（pointwise: 各馬を独立した行として対数損失（LogLoss）：確率予測の誤差指標 などで学習する方式） を主軸にしながら、LambdaRank 系・最新の LTR・深いアンサンブル（複数モデルの組み合わせ）・NN を精度探索の軸として明示します（実装は段階的にブランチを切る）。

基本方針

1. DataFrame の第一選択は polars

以下の処理はすべて polars で行います。

  • 学習母表の組み立て
  • 特徴量結合
  • 集約
  • fold ごとの OOF 特徴量生成

2. pandas は境界面だけで使う

以下の最小限の場面のみに限定します。

  • LightGBM / CatBoost / XGBoost へ最終入力する直前
  • SHAP（各特徴量の予測への寄与度を示す手法）や一部可視化ライブラリが pandas を要求する場面
  • 小サイズの CSV を出力する場面

3. 入力は原則 Parquet + scan

全列を一括で読む read_parquet() ではなく、まず scan_parquet()（遅延読み込み）を使います。

必要な列だけ select() し、必要な行だけ filter() してから collect()（実際の読み込み）を実行します。

4. 中間成果物を積極的に保存する

OOF や stage 出力は毎回再計算せず、Parquet に保存して使い回します。

保存対象のファイル:

  • horse_latent_state.parquet
  • positioning_features.parquet
  • pace_scenario_features.parquet
  • day_track_bias_features.parquet
  • contextual_bloodline_features.parquet
  • horse_probability_table.parquet

メモリ節約の原則

1. 必要な列だけ読む

全列を読み込むと不要なメモリを大量に消費します。必要列のみ指定してから読み込みます。

悪い例（全列読み込み）:

df = pl.read_parquet("race_shutuba_flat.parquet")

良い例（列を絞ってから読み込む）:

df = (
    pl.scan_parquet("race_shutuba_flat.parquet")
    .select([
        "race_id", "date", "venue_code", "surface", "distance",
        "horse_number", "horse_id", "jockey_id", "trainer_id",
        "weight", "weight_change"
    ])
    .collect()
)

2. 必要な年だけ読む

複数年にまたがる処理でも、全件を一度に collect() しません。年ごとの scan を lazy で結合します。

lf = pl.concat([
    pl.scan_parquet(f"data/local/tables/{y}/race_shutuba_flat.parquet")
    for y in years
])

3. 必要な fold だけ materialize する

全 fold 分の OOF を一度にメモリ上に作らず、1 fold ずつ処理します。

  • fold_1 を作って Parquet に保存する
  • メモリを解放する
  • fold_2 を作る（同じ手順を繰り返す）

4. 型を落とす

不必要に精度の高い型を使うとメモリが増えます。用途に合わせて型を下げます。

用途推奨型理由
確率・連続量Float3232bit 浮動小数点で十分な精度
count・フラグInt8 / Int16 / UInt16小さい整数型でメモリ節約
順位UInt16符号なし16bit整数で十分
カテゴリ IDInt32符号あり32bit整数
長い文字列辞書型または ID可能な限り変換してメモリを節約

5. Python ループを避ける

行ごとに Python ループを回すと処理が遅くなります。polars の組み込み関数を使います。

  • groupby().agg()（グループ集計）を優先する
  • with_columns() で複数列をまとめて一括作成する
  • map_elements()（Python 関数をセルごとに適用）は最後の手段

6. 巨大な join を避ける

母表に全テーブルを一度に join するとメモリが爆発します。段階的に join して都度保存します。

推奨する手順:

  1. 母表の最小列（識別子と基本情報のみ）を作る
  2. race_index（レース指数）を join する
  3. 保存する
  4. past aggregate（過去成績の集計）を join する
  5. 保存する

7. 生テキストを学習行列に展開しない

  • comment や paddock_comment（パドックコメント）を全文保持したままの巨大 DataFrame を学習に使わない
  • スコア化（数値変換）または presence flag（存在フラグ）化してからテキスト列を削除する

polars 実装パターン

1. 母表生成（全期間の基本特徴量を結合する）

base = (
    pl.concat([
        pl.scan_parquet(f"data/local/tables/{y}/race_shutuba_flat.parquet")
        for y in years
    ])
    .select([
        "race_id", "date", "venue_code", "venue", "surface", "distance",
        "direction", "grade", "race_class", "field_size",
        "horse_number", "horse_id", "horse_name",
        "jockey_id", "trainer_id", "weight", "weight_change",
        "sire", "dam_sire"
    ])
)

2. sparse テーブルの結合（存在しない行をスキップして left join する）

paddock = (
    pl.concat([
        pl.scan_parquet(f"data/local/tables/{y}/race_paddock_flat.parquet")
        for y in years
    ])
    .select(["race_id", "horse_number", "paddock_rank", "paddock_comment"])
    .with_columns([
        pl.lit(1).alias("has_paddock"),
    ])
)

joined = base.join(
    paddock,
    on=["race_id", "horse_number"],
    how="left",
)

3. レース内 rank / z-score（同一レース内での相対値を計算する）

df = df.with_columns([
    pl.col("speed_max")
      .rank("ordinal")
      .over("race_id")
      .alias("speed_rank_in_race"),
    (
        (pl.col("speed_max") - pl.col("speed_max").mean().over("race_id")) /
        (pl.col("speed_max").std().over("race_id") + 1e-6)
    ).alias("speed_z_in_race"),
])

OOF 特徴量の生成戦略

1. target encoding（カテゴリの的中率エンコード）

方針:

  • split manifest に従って fold ごとに train / valid を分ける
  • train 側だけでカテゴリ平均（的中率など）を計算する
  • valid 側へその統計値を map する（valid 側の情報は使わない）
  • 結果を oof_fold_x_*.parquet として保存する

2. ステージモデルの出力

以下のモデル出力は、必ず OOF（交差検証外データ）で生成して Parquet に保存します。

モデル名役割保存パス例
PositioningModel脚質・位置取り予測data/modeling/oof/fold_1_positioning.parquet
PaceScenarioModelペース展開予測data/modeling/oof/fold_1_pace.parquet
DayTrackBiasModel当日バイアス予測—
ContextualBloodlineFitModel血統適性スコア—
最終確率モデル最終確率予測data/modeling/oof/fold_1_final_prob.parquet

3. calibration（確率補正）

  • 全 fold の OOF 予測を集めた 1 本の calibration 用テーブルを作る
  • この calibration 用テーブルだけ pandas / numpy に変換してもよい

学習器ごとの実装戦略

LightGBM

  • 最終入力は float32 / int32（小さい型でメモリを節約）
  • categorical_feature（カテゴリ特徴量）を活かせる列は integer id に変換してから渡す
  • OOF target encoding 後の列は数値列として扱う
  • 学習用 matrix は fold 単位で生成し、不要になったら即解放する
  • ランキングモード: objective="lambdarank" に label_gain（relevance レベルごとのゲイン）と group（各レースの行数）を指定する。監視指標に eval_at=[3,5] などで NDCG@k を追加すると比較しやすい

CatBoost

  • 種類が多いカテゴリ（高カーディナリティ）の扱いが CatBoost の強み
  • 文字列カテゴリをそのまま渡すか、辞書 ID に変換して渡す
  • 巨大な DataFrame を一気に pandas に変換しない
  • fold 単位で Pool（CatBoost の入力フォーマット）を組み立てる
  • ランキングモード: loss_function に YetiRank / StochasticRank / QueryCrossEntropy 等を指定し、group_id=race_id でクエリ（1レース）を指定する

XGBoost

  • QuantileDMatrix（メモリ効率の良い入力形式）を優先して使う
  • dense float64（全セル展開の倍精度）行列は避ける
  • ランキングモード: objective=rank:ndcg（または rank:pairwise / rank:map）と、クエリごとの行数 group を必ず指定する

ランキング学習（LTR）と競馬タスクの相性

競馬の着順予測を「検索順位付け」と同じ問題として捉えたのがランキング学習（Learning-to-Rank）です。

  • クエリ（query） = 1レース（race_id に属する出走馬の集合、頭数は可変）
  • ドキュメント = 各馬の特徴ベクトル（レース発走 10 分前までに観測できる列のみ）
  • 関連度（relevance） = 確定着順や複勝圏から定義した非負整数・実数（例: 1着を最高値、失格・除外は 0 など）

Pointwise（各馬に is_win を当てる分類）は実装が単純で馬券用の確率補正とも相性が良いです。

一方で Listwise / Pairwise は「同一レース内の順序」を直接損失に組み込むため、上位馬の捕捉精度（NDCG@3）や着順の並び方の再現に強いことが多いです。

本プロジェクトでは両方を 同一の purged CV / race 単位 split の下で比較できる設計にします（train_valid_test_split_strategy.html と整合）。

古典〜標準: LambdaRank と GBDT ランキング

LambdaRank / LambdaMART

  • LambdaRank: ペアを入れ替えたときの NDCG の変化量を勾配に反映する近似手法。「このペアの順位を直すとどれだけ評価が改善するか」を損失に組み込む。
  • LambdaMART: LambdaRank の勾配を使って GBDT（木を積み上げた機械学習モデル）を学習する枠組み。
  • LightGBM での使い方: objective="lambdarank" と label_gain（relevance レベルごとのゲイン配列）、group（各レースの行数）を指定することで listwise 学習が可能。
  • メモリの注意: group は numpy 配列で渡すことが多い。レース単位で fold を materialize し、学習直前だけ行列化する。

XGBoost のランキング目的関数

  • rank:ndcg — NDCG を意識した pairwise 近似（結果の報告がしやすい）
  • rank:pairwise — シンプルなペアワイズ損失（高速・ベースライン比較向き）
  • rank:map — 平均精度（上位重視の別解釈）

CatBoost のランキング

  • YetiRank、StochasticRank、QueryCrossEntropy、LambdaMart などを loss_function に指定できる（CatBoost 公式ドキュメントの Ranking: objectives and metrics を参照）。高カーディナリティなカテゴリと組み合わせやすい。
  • group_id=race_id でクエリ（1レース）を指定し、同一レース内でペア／リスト損失を構成する。

relevance 設計の例（実装メモ）

  • 着順ベース: relevance = max_rank + 1 - finish_position（1着が最高値、着外は下限値にクリップ）
  • 複勝重視: 3着以内を同じ値に近づける、または top3 にボーナスを付けた多段 relevance
  • 将来情報の混入禁止: relevance は 確定した結果ラベルのみで定義する。特徴量側に結果由来の列が入っていないことを二重チェックする。

拡張: 目的関数・学習枠の幅を広げる試み

着順予測の精度をさらに上げるための探索候補を、同一 CV プロトコルで ablation しやすい順に並べます。

  1. ListNet / ListMLE 型（NN または近似）
    レース内スコア s_i に対し、P(順序) ∝ softmax の負の対数尤度（ListMLE）や確率分布の KL ダイバージェンス（ListNet）を損失に使う手法。
    小規模なら レース単位バッチ（batch_size=1 race、馬数可変）の MLP（多層パーセプトロン）で試せる。

  2. Pairwise 深層（RankNet 系）
    同一レース内の馬ペア (i,j) を生成し、σ(s_i - s_j) と真の勝敗ラベルの cross-entropy を損失にする。
    メモリ注意: ペア数は頭数の 2 乗（O(field^2)）になるため、上位馬同士に限定・サンプリングして削る。

  3. Multi-task（複数タスクの同時学習）
    同一バックボーンから pointwise head（確率予測：BCE / focal 損失）と ranking head（順位予測：pairwise hinge 損失）を同時に学習し、損失を L = L_point + λ L_rank で合成する。
    馬券用確率は pointwise 側が補正しやすい。

  4. Tabular 深層学習（NN の可能性を残す）

    • 埋め込み（Embedding）: horse_id / jockey_id / trainer_id を hash embedding（出現頻度に応じて次元を調整）に変換する
    • FT-Transformer / TabTransformer: 列をトークン化して self-attention で学習する。精度ポテンシャルは高いが fold ごとの学習コストが大きいため、extended_v1 確定後の「精度を伸ばす」候補として位置づける
    • 知識蒸留: 強いランキングモデルや大きい NN のスコアを、LightGBM pointwise / lambdarank への追加ターゲットまたはソフトラベルとして与える

  5. 確率とランキングの橋渡し
    ランキングモデルの出力スコアをレース内 softmax で疑似確率に変換し、温度スケールで補正する（valid のみ fit）。
    Plackett–Luce 等の順序モデルは理論的には listwise と整合するが実装コストが高い。まずは softmax 近似で十分なことが多い。

  6. 研究寄り・長期オプション（優先度は低め）

    • LambdaLoss の一般形やオークション理論に基づく損失関数は、文献を読んでから個別に検討する
    • Determinantal Point Process (DPP) で多様性を入れる手法は馬券最適化層に近く、当面は保留

アンサンブル（複数モデルの組み合わせ）の層を深くする設計

単純な平均ブレンドだけでなく、層状スタッキング（モデルの予測を別モデルで再学習）を許容する設計です。

層 中身 備考
L1a LightGBM / CatBoost / XGBoost pointwise（is_win / is_top3 の二値分類） 現行方針の主戦力
L1b LightGBM lambdarank、CatBoost YetiRank、XGBoost rank:ndcg 同一特徴量を使いながら「順位付け」の視点で別モデルを作る
L1c NN（listwise または score + softmax、multi-task） メモリ・再現性コストが高いため fold 単位で checkpoint を保存する
L2 OOF 上のメタ特徴（L1 各モデルの予測・順位・レース内 z-score）を入力にした 浅い GBDT またはロジスティック回帰 valid でのみ学習。test では L1 の OOF 生成手順を固定して使う
L3 ランキング再ブレンド（Borda / 順位平均 + L2 出力の凸結合）や セグメント別重み（芝・ダート・頭数帯ごとに異なる重みを適用） 過学習に注意し、重みは valid のみで決定する

実装原則:

  • 各層の OOF を Parquet に保存し、上の層は下の層の保存ファイルだけを読む（毎回の再計算を禁止に近い運用）
  • test データにはメタ学習を一切 fit しない（advanced_modeling_techniques.html と同じルール）

LTR 導入時の polars とメモリの両立

  • relevance（関連度）と group（各レースの行数）列は polars で母表に付与してから、学習直前に numpy に変換する。
  • 可変長クエリ（出走頭数が毎レース異なる）は、学習 API ごとに group 配列または group_id 列で表現する。巨大なペア展開テーブルはディスクに書かず、DataLoader 内で on-the-fly（その場で）生成（NN の場合）を推奨する。
  • 型の設定: relevance は Int16 程度で足りることが多い。label_gain（LightGBM の relevance ゲイン配列）は事前に固定長配列として定義しておくと LightGBM 側の設定が単純になる。

ランキング用の評価指標（ログに必ず残す）

  • NDCG@k（k=1, 3, 5 など）: 上位 k 頭の順位予測精度。高いほど上位馬を正しく並べられている。
  • MAP（平均精度）: 上位を正しく順位付けできているかの指標。
  • レース内 Kendall τ: 予測した順位と実際の着順の一致度。
  • 馬券・確率補正との接続のため、pointwise の対数損失（LogLoss）：確率予測の誤差指標 / Brier スコア / ECE（期待誤差） も一緒に記録する（LayerA 確率の品質監査）。

calibration（確率補正）の実装戦略

  • OOF predictions + labels だけを集めた小さな calibration テーブルを作る
  • isotonic regression は過学習（overfit）しやすいので注意する
  • 初期版は temperature scaling（最も単純）と Platt scaling（ロジスティック回帰）を優先する

ステージごとの実装指針

ステージ目的実装指針
Stage0 / Stage1 母表と基本特徴量の構築 polars で全期間の母表を組み立て、まず BaselineV1 を Parquet に保存し、次に ExtendedV1 を差分追加する
Stage2 / Stage3 中間モデルの学習と OOF 出力 学習データの構築は polars、モデルへの最終入力だけ pandas / numpy に変換する。モデル出力は fold 単位で Parquet に保存
Stage4 / Stage5 当日バイアスと血統適性の計算 同日先行レース集計は polars.groupby、posterior は 1 日単位の小テーブルで扱う。血統ベクトルは事前計算して Parquet に保存
Stage6 最終確率の出力 各ステージ出力を join した最終特徴量テーブルを入力とする。OOF final probabilities を正式成果物にする（pointwise 主系）。任意でランキング OOF も別 Parquet で保存

MLflow による学習記録（必須運用）

確率モデル・ランキング枝・アンサンブル（複数モデルの組み合わせ）・校正前後の比較まで、学習ジョブごとに MLflow に記録します。ローカル運用でも Tracking URI を固定し、実験名でモデル種別を分けます。

やらないこと（禁止事項）

禁止事項理由
全年・全列を pandas に一括ロードするメモリが枯渇する
read_parquet() だけで巨大テーブルを毎回 materialize するscan_parquet() を使い遅延読み込みにする
全 fold 分のデータを同時にメモリ上に保持するメモリ不足になる。fold ごとに逐次処理する
stage 出力を保存せず毎回再計算する時間・メモリの無駄。Parquet に保存して使い回す
生テキストや nested list をそのまま最終学習行列に入れるメモリが増大する。スコア化または presence flag 化する
L2/L3 メタ学習や温度スケールを test データで fit するtest の情報が補正器に漏れる。valid または OOF のみで fit する

推奨実装順

  1. polars ベースの母表 builder（基本データの結合）
  2. fold 単位 OOF runner（交差検証の枠組み）
  3. target encoding recipe runner（カテゴリのエンコード）
  4. stage 出力保存 runner（各ステージモデルの OOF 保存）
  5. final probability runner（pointwise 主系で安定版を確保）
  6. （任意） LTR ブランチ: relevance / group 生成 → LightGBM lambdarank と CatBoost YetiRank の 同一 CV 比較
  7. （任意） L2 メタ学習（L1 pointwise + L1 ランキングの OOF のみを入力として使う）
  8. （任意） NN listwise / multi-task（extended_v1 と purged CV が固まってから実施）
  9. calibration runner（確率補正）
  10. backtest dataset runner（バックテスト用データの生成）
