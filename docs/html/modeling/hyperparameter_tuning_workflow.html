ハイパーパラメータ・モデル選定のチューニング工程

ソース: docs/html/modeling/hyperparameter_tuning_workflow.html · 生成: 2026-05-20 06:01 UTC

モデルの設定値（ハイパーパラメータ）チューニングとモデル選定の工程

0. 現状認識：この議論は終わっているか

終わっていない（未整理の領域）です。

  • train_valid_test_split_strategy.html には「2024 valid でモデルの設定値（ハイパーパラメータ）・特徴選抜を行う」とあるが、探索の手順・打ち切り条件・再現性・計算予算の配分については別ドキュメント化されていなかった。
  • advanced_modeling_techniques.html はテクニック集であり、チューニング運用の手順書（SOP）ではない。
  • 本書は 「どのデータで・どの指標で・どこまで探索し、いつ設定を凍結するか」 と、工程そのものの最適化（試行回数・壁時計時間・メモリ使用量） を定義する。

関連ドキュメント: data/meta/modeling/dataset_split_manifest.json、train_valid_test_split_strategy.html、modeling_coding_strategy.html。

1. 絶対ルール（リーク防止と評価の透明性）

ルール 内容
test 禁止 2025 の評価データを見ながらモデルの設定値・特徴列・補正器を変えない。
選定に使うデータ 2024 valid（検証データ）または inner CV（内側の交差検証）の OOF 集約スコアだけでチューニングを判断する。
確率補正（calibration） 温度スケールなどの補正器は 2024 valid または train 内 OOF のみで fit する。2025 test では fit しない。
データ版の固定 1 つの tuning study（探索セッション）では data_version・母表の列集合・TE recipe を変更せず固定する。変えた場合は別 study として記録する。

2. 二層構造：inner CV と outer valid

2.1 Inner（開発の主戦場：交差検証で設定を絞る）

  • rolling_origin_folds（時系列ローリング分割）+ purge / embargo（manifest 準拠）で 複数の検証ウィンドウを回す。
  • 各 trial（試行） は「1 つのモデルの設定値」を 全 inner fold で学習し、集約スコア（平均対数損失（LogLoss）：確率予測の誤差指標 や平均 NDCG@3）を返す。
  • 目的: 2024 の 1 年だけに過適合した設定を早期に排除する。

2.2 Outer（最終ゲート：2024 valid で最終確認）

  • inner で上位 K 件（例: 3〜5 件）に絞ったうえで、同一パイプライン・同一特徴量のまま 2024 valid 全体で再学習・再評価する。
  • 2025 test は、設定が凍結されたあと 一度だけ報告用に実行する。

2.3 inner CV（nested CV）が必要な場面

  • 試行数が少ない・探索空間が狭い場合 → flat valid（2024 のみ）でも問題ない。
  • 試行数が多い・Optuna（自動ハイパーパラメータ探索ライブラリ）などで自動探索する場合 → inner CV スコアを主指標（primary）にし、2024 valid は 最終の健全性確認（sanity check）として使うと安全。

3. 最適化する「対象」の順序（凍結する順）

複数の要素を同時に探索すると要因分解ができなくなります。上流から順番に固定し、下流ほど探索空間を狭くします。

  1. 母表・TE recipe・欠損方針を最初に固定する（BaselineV1 または ExtendedV1 のどちらか一方を選ぶ）。
  2. 単一モデル族（例: LightGBM pointwise）で木の深さ・正則化・学習率をチューニングする。
  3. 別モデル族（例: CatBoost）を同じ考え方でチューニングする。アンサンブル（複数モデルの組み合わせ）の重みは valid のみで決定する。
  4. ランキング枝（lambdarank / YetiRank）は pointwise が安定してから 別 study（別の探索セッション）として開始する。
  5. 多段ステージのモデルの設定値はステージごとに区切ってチューニングする（前段を毎試行再学習しないよう、前段は checkpoint を再利用する）。
  6. L2 メタ学習は L1 の OOF が揃ってから、極小の探索空間（線形ブレンド係数など）で行う。

4. チューニング工程の最適化（計算コストと時間の節約）

4.1 粗から細へ（Coarse-to-fine）

最初から全データで精密に探索するのではなく、段階的に絞り込みます。

  • 第 0 相: ランダム探索を 年サブセット（例: 2022 のみ）または レース数のサンプリングで実施し、明らかに性能が悪い設定領域を除外する。
  • 第 1 相: 残った候補を 全 train 年（2020-2023）+ inner CV で再評価する。
  • 第 2 相: 上位の候補だけ 木の本数を増やす・early stopping ラウンドを伸ばすなど本番に近い設定で最終評価する。

4.2 多保真度（Multi-fidelity）：コストに応じた段階評価

  • 少ない boosting round や 縮小した特徴集合で候補の大まかな順位を付け、上位だけフルのリソースで評価する。
  • LightGBM / XGBoost / CatBoost のいずれも early stopping を主指標にし、同じ valid fold で比較する。

4.3 探索アルゴリズム

  • 探索次元が低〜中の場合: ベイズ最適化（賢い設定値の自動探索）（TPE / Gaussian Process） — Optuna（自動ハイパーパラメータ探索ライブラリ）等を使う。
  • ベースラインとの比較: RandomizedSearch（ランダム探索）で十分なことが多い（競馬はノイズが大きく、精密な探索が常に有効とは限らない）。
  • グリッド全探索は探索次元が増えると指数的にコストが爆発するため、重要な次元だけグリッド探索し、他はランダム探索を推奨。

4.4 並列化と I/O

  • trial（試行）間は独立しているため並列 worker で実行できる。ただし WSL のメモリ上限に合わせて同時 trial 数を制限する（modeling_coding_strategy.html を参照）。
  • fold の行列化（materialize）は 1 回だけキャッシュし、同一 fold で複数 trial が読み込むだけにする（polars で準備 → 学習直前に numpy へ変換）。

4.5 早期打ち切り（Study レベル）

  • ASHA / Successive Halving 的運用: 途中のエポック（または boosting round 10 のスコア）が下位分位以下なら trial を途中で打ち切る。
  • Optuna の Pruner と inner fold の平均スコアを組み合わせると、試行回数あたりの探索効率が上がる。

5. 目的関数の設計（指標の宣言）

  • 1 study（探索セッション）につき主指標（primary metric）を 1 つだけ決める（例: mean_logloss または mean_ndcg@3）。
  • 副次指標（ECE・重賞セグメントの精度など）は ログに記録するが primary にはしない（多目的最適化は設計が複雑になる割に効果が薄いことが多い）。
  • ランキング枝（NDCG を主指標とする）と pointwise 枝（LogLoss を主指標とする）は 別の primary メトリック・別 studyに分ける。

6. アンチパターン（やってはいけないこと）

アンチパターン問題点
train と valid を結合してから early stopping や探索に使うvalid が汚染される
test を見たあとに「もう一度だけ」グリッドを広げるtest の情報がモデル選定に漏れる
TE recipe と木の深さを同一の trial で同時に変えて比較するどちらが効いたか分からなくなる
inner fold を無視して 2024 のたまたま良い日付にだけ合う設定を採用する過学習と区別できない

7. 成果物（study ごとに保存）

study（探索セッション）ごとに以下を保存します。後から再現・比較できるようにするために必須です。

項目説明
study_id探索セッションの識別子
git_commitコードのバージョン
split_manifest_versionデータ分割の設定バージョン
feature_list_hash使用した特徴量一覧のハッシュ値
params（各 trial）試行ごとのモデル設定値
inner_fold_scores（各 trial）inner CV の各 fold スコア
aggregated_score（各 trial）集約スコア（fold 平均など）
wall_time_sec（各 trial）処理時間（秒）
gpu_mem_peak（各 trial、任意）GPU メモリのピーク使用量
採用した設定（YAML / JSON）本番の refit でそのまま読み込める形式で保存

推奨パス例: data/meta/modeling/tuning_studies/<study_id>/

7.1 MLflow での必須記録

各 study・各 trial（少なくとも 採用 trial と最終 refit）について MLflow Run を作成します。

種別記録する内容
Parametersモデルの設定値（ハイパーパラメータ）の全セット、split_manifest_version、feature_list_hash
Metricsinner fold 集約スコア、2024 valid 再評価スコア、処理時間、n_trees（木の本数）など
Tagsstudy_id, trial_number, git_commit, primary_metric
Artifacts採用した config.yaml、特徴量リスト、（任意）OOF 予測のサンプル Parquet

horse_pre_race_dataset_spec.html の必須メタデータと mlflow_run_id を相互参照できるようにします。

8. チェックリスト（2025 評価データで報告する前の凍結確認）

確認項目判定基準
primary metric（主指標）の文書化study の冒頭で明記されている
2025 評価データの未参照2025 のデータを一切参照していない

9. まとめ

チューニング工程は、データ分割・リーク防止・実装メモリ管理とは別の軸として、本書まで含めて初めて運用上の手順が完結します。

今後の変更は study 単位でバージョンを付け、runs.jsonl または tuning ディレクトリに追記してください。

10. 現行実装（pipeline/）とのギャップ

ドキュメント（dataset_split_manifest.json・本書・train_valid_test_split_strategy.html）と、リポジトリ内の学習コードは 2026 時点で完全には一致していません。

モデル選定の手順をコードに落とすときは、次の差分を前提にマイルストーンを設定してください。

10.1 src/pipeline/trainer.py（ModelTrainer）

ドキュメント側の意図 実装の現状 リスク
暦年・race 単位で train / valid / test を分離する split_idx = int(len(X) * (1 - test_ratio)) で 行の先頭からの割合分割（DataFrame の行順に依存） 行が日付順に並んでいないと 時系列 split にならない。同一レースが train と test に跨る可能性がある。
2024 valid / 2025 test など manifest 準拠で分割する years は _load_or_build の一部経路でしか効かず、分割は 年単位になっていない 報告用 test を「一度きり」に固定できない。
着順の強弱を反映した relevance で LambdaRank を学習する y は 複勝圏の二値（1<=着順<=3）をランキングラベルとして使用している 着順の強弱が損失に反映されず、listwise のランキング学習の意図からずれる（ベースラインとしては許容できるが、ドキュメントの LTR の節とは別物として明示すべき）。
Optuna（自動ハイパーパラメータ探索ライブラリ）でモデルの設定値を自動探索する n_optuna_trials は MLflow へのタグ記録が中心で、train() 本体に 探索ループが存在しない ドキュメントの「study・pruner」による探索機能が未接続。

次の実装アクション: race_date（または race_id 先頭年）でソートし、dataset_split_manifest.json を読んで 評価用 race_id の集合でマスク分割するモジュールを ModelTrainer の前段に挿入する。LambdaRank 用には relevance = f(finish_position)（着順から計算した関連度）を別列で渡すオプションを追加する。

10.2 src/pipeline/ensemble_trainer.py（EnsembleTrainer）

ドキュメント側の意図 実装の現状 リスク
purge + embargo 付きの時系列 CV を使う GroupKFold(race_id) は train 内で 同一レースを同一 fold に閉じるが、日付ベースの purge はない 近接開催のデータ間の情報リークが、ドキュメントの設計より緩い。
メタ学習は valid のみで fit する メタは train 区間の OOF に対して LogisticRegression で fit する（設計としては一般的） 最終報告用の 2024 ブロックを完全にホールドアウトする設計になっていない（train の一部の OOF でメタが学習されている）。

次の実装アクション: GroupKFold を維持しつつ、fold の境界を race_date + purge_window_days で切る PurgedGroupTimeSeriesSplit 相当のイテレータに差し替える。また、メタ学習用に 2024 行を学習から除外した OOF を別パスで生成するオプションを検討する。

10.3 src/pipeline/feature_store.py

  • 列指向・スナップショット・_registry.json（特徴量レジストリ）は modeling_coding_strategy.html の方針と整合しやすい。
  • 学習行列の 行順・重複 race の保証方法については、build_training_matrix の呼び出し規約で明示する必要がある（trainer の分割処理と連動させる）。

10.4 src/research/evaluate_race_performance_signal.py 等（ベンチ・調査用スクリプト）

  • これらのスクリプトは 独自のデータ結合・特徴量定義を持っており、本番の ModelTrainer と同じ split 設計とは限らない。
  • 指標の比較は「シグナル検証」としてログに残し、本番の選定スコアは manifest 準拠の runner だけに集約するのが望ましい。

10.5 推奨マイルストーン（実装順）

  1. Split モジュールの切り出し: manifest 読込 + race_id / race_date による train/valid/test マスク生成を trainer から独立したモジュールに分離する。
  2. trainer の修正: 上記マスクで分割するように差し替え、test_ratio ベース分割を deprecated（廃止予定）または「デバッグ専用」と明記する。
  3. relevance 列の追加（任意）と、二値 top3 との 切り替えフラグの実装。
  4. Optuna の接続: train() から study.optimize を呼ぶオプション、または CLI src/pipeline/cli.py に study 専用コマンドを追加する。
  5. ensemble の purge 対応: purge 対応の fold に変更し、メタ学習の fit 対象データをドキュメントの outer valid 方針に揃えるオプションを追加する。

これらを進めると、ドキュメントと実装の二重管理が減り、study_id と split_manifest_version を同一の run で記録できるようになります。
