horse_pre_race_dataset 仕様

目的

horse_pre_race_dataset は、レース直前確率モデルの学習母表である。粒度は 1行 = 1レース1頭 で、レース発走 10 分前（T-10分）までに観測できる情報だけを保持する。

本仕様では以下の 2 ファイルを定義する。

  • horse_pre_race_dataset.parquet — 学習・推論で共通利用する特徴量（予測に使う数値）の母表
  • horse_labels.parquet — 学習・評価用の結果ラベル表

設計思想

  • 最初から全特徴量を一括投入するのではなく、ベースライン母表 と 拡張母表 を分けて比較できるようにする
  • ベースライン段階では、既存テーブルでカバレッジが高く安定している列だけを使う
  • 拡張段階で、race_oikiri（追い切り）、race_paddock（パドック）、race_trainer_comment（厩舎コメント）、血統補助、クッション値 proxy を順次追加する
  • 予測精度だけでなく、確率の信頼性（キャリブレーション） と 後続のベット最適化で使える一貫性 を重視する
  • Kaggle の tabular 実戦テクニックを積極的に採用するが、必ず 交差検証での予測（OOF） と データリーク（未来の情報を誤って使うこと）の防止 を前提にする
  • 単段モデルだけでなく、位置取り → ペース → 当日馬場タイプ → 血統文脈適合 → 最終確率 の多段階の処理の流れに拡張できる形で保存する

機械学習テクニックの利用方針

1. カテゴリ変数エンコーディング（カテゴリを数値に変換する方法）

カテゴリ変数は 1 つの手法に固定せず、以下の 3 層で比較する。

Tier方針手法一覧
Tier1 低リスクで常時利用 frequency encoding（出現頻度に変換）、count encoding（出現回数に変換）、rare label grouping（希少カテゴリをまとめる）、ラベルエンコーディング（simple label encoding）
Tier2 交差検証での予測（OOF）が必須 ターゲットエンコーディング（target encoding）、CatBoost-style ordered target encoding（時系列順に累積平均）、expanding mean encoding（時点ごとに累積平均）、group target encoding（グループ内平均）
Tier3 比較実験用 target std / target skew encoding（標準偏差・歪度に変換）、weight of evidence 風 encoding、cross feature target encoding（交差特徴量のターゲットエンコーディング）

2. ターゲットエンコーディングの原則

  • 全データをまとめて平均計算しない（データリーク防止）
  • 必ず交差検証での予測（OOF）として作る
  • 未知のカテゴリは train fold 全体の平均にフォールバックする
  • 希少カテゴリは全体平均へ shrink（縮約）する
  • 主な対象: jockey_name, trainer_name, sire, dam_sire, venue_surface_distance_key

3. CatBoost-style encoding の原則

  • 時系列順に並べて、同一行より未来を参照しない ordered target mean を使う
  • fold 内の train 部分だけで cumulative（累積）に更新する
  • レース単位のグループをまたいでリークしない

4. 交差特徴量

Kaggle の tabular 競技で効果が出やすい交差特徴量を、過剰に増やさず上位から順に採用する。

交差特徴量キー説明
jockey x venue騎手 × 開催場
trainer x venue調教師 × 開催場
sire x surface父馬 × 芝/ダート
sire x distance_bucket父馬 × 距離帯
sex x distance_bucket性別 × 距離帯
bracket_number x venue_surface_distance_key枠番 × 開催条件
venue_era x surfaceコース時代区分 × 芝/ダート

5. 交差検証での予測（OOF）を使った集約特徴量

  • OOF でのみ作成し、train fold 内で集計して valid fold にマッピングする
  • 例:
    • jockey_win_rate_oof（騎手の勝率 OOF）
    • trainer_top3_rate_oof（調教師の複勝率 OOF）
    • sire_win_rate_oof（父馬の勝率 OOF）
    • jockey_venue_top3_rate_oof（騎手 × 開催場の複勝率 OOF）
    • trainer_surface_distance_win_rate_oof（調教師 × 芝/ダート・距離の勝率 OOF）

6. 数値列の実戦処理

  • log transform（対数変換）
  • rank-gauss / quantile transform（分位点変換）
  • clipping / winsorization（外れ値の上限・下限カット）
  • レース内での rank / percentile / z-score
  • 高カーディナリティのグループ内 robust scaling

7. 比較するモデル構成

  • GBDT（木を積み上げた機械学習モデル）として LightGBM / CatBoost / XGBoost
  • 線形モデル（ベースライン）

まず LightGBM / CatBoost の 2 本柱を本命とし、交差検証での予測（OOF）確率をアンサンブル可能な形で保存する。

ベースライン母表と拡張母表

BaselineV1: まず作る比較基準

最初に必ず構築する比較用母表。カバレッジの高さと保守性を優先する。

採用するテーブル

  • race_shutuba_flat（出馬表）
  • race_shutuba_past_flat（過去走）
  • race_index_flat（指数）
  • GCS 由来 horse bundle の履歴・血統基礎集約

ベースライン段階で除外するもの

  • race_oikiri_flat（追い切り）
  • race_paddock_flat（パドック）
  • race_trainer_comment_flat（厩舎コメント）
  • クッション値の高度な proxy

目的

  • 安定したベースラインを先に作り、追加特徴量の寄与を差分で検証する

ExtendedV1: 直前情報・馬場 proxy を追加した母表

BaselineV1 に直前状態・馬場 proxy を追加した母表。

追加するテーブル・特徴量

  • race_oikiri_flat の集約統計（追い切り評価・タイム）
  • race_paddock_flat（パドック評価）
  • race_trainer_comment_flat（厩舎コメントのスコア）
  • クッション値・含水率・proxy
  • venue_era_fit（コース時代適合）、seasonal_surface_fit（季節×馬場適合）、field_strength_proxy（出走馬のレベル代理変数）

目的

  • 直前情報を追加したときの log_loss、Brier、ECE（確率精度）、ROI の改善幅を測る

加工ルール

1. 結合（join）ルール

  • 基本の結合キーは race_id + horse_number
  • race_oikiri_flat は race_id + horse_number に集約してから結合する
  • horse_id は補助キーとして保持するが、race_odds_flat には存在しないため結合の主キーに使わない

1.5 集約ルール

  • race_oikiri_flat は 1 頭に複数行あるため、必ず race_id + horse_number に集約する
  • race_shutuba_past_flat.past_races は対象レース日以前の履歴に絞ってから集約する
  • horse bundle 由来の集計も、対象レース日以前のデータだけで計算する
  • レース内の相対指標（順位・偏差等）は、同一 race_id の母表が完成した後に最後に計算する

2. 時点ルール（データリーク防止）

  • 各特徴量は対象 race_date より前の情報だけで計算する
  • days_since_last や戦績集計は、当該レース当日以前の履歴だけに限定する
  • クッション値は対象開催日の朝時点の公表値を使う

3. テキスト列ルール

  • 生テキストは原則保存しない
  • 保存する場合は *_raw_text として分離し、学習投入対象から除外する
  • 初期版では、コメントはスコア化と presence flag（存在フラグ）のみを正式採用とする

4. データリーク防止ルール

ルール内容
結果テーブルの除外結果確定後のテーブルは特徴量母表に含めない
LayerA へのオッズ禁止LayerA（市場情報なし層）に当日オッズを入れない
エンコーディングの適用範囲ターゲットエンコーディングや集計特徴量は、split 後・fold 内でのみ作る
着順の逆参照禁止finish_position（着順）を説明変数に戻さない
race_id 横断分割禁止レース単位で train / valid / test を分け、同一 race_id をまたぐ分割を禁止する
未来マスター禁止推論時点に存在しない future master や future encoded mapping を読み込まない
エンコーディングの前提ordered / target encoding は purged_group_time_series_split を前提に作る
確率補正の適用対象確率補正（calibration）は train データではなく valid または交差検証での予測（OOF）のみで学習する

4.5 EDA・品質監査ルール

学習前に欠損・外れ値・分布の確認を明示的に行うことが重要である。母表生成時に以下の監査成果物を必須とする。

label_balance.jsonis_win, is_top2, is_top3 の陽性率

4.6 外れ値・ノイズ方針

  • 取消・除外・競走中止・失格は label 側で明示フラグを立て、Baseline の学習対象から除外する候補として管理する
  • 極端なタイム差・明らかな異常値は単純除外せず、outlier_flag を立てた上で比較検証する
  • race_oikiri のようにノイズが多いテーブルは、最初は集約統計だけを使い、生の列を直接展開しない

8. 実装順

  1. race_shutuba_flat を母表の起点として固定する
  2. race_shutuba_past_flat・race_index_flat を結合する
  3. race_oikiri_flat を馬単位に集約する
  4. race_paddock_flat・race_trainer_comment_flat を left join する
  5. GCS 由来 horse bundle の集約を追加する
  6. クッション値・開催文脈・venue_era を付与する
  7. LayerA（市場情報なし）と LayerB（市場情報あり）の列集合を分離する
  8. race_result_flat から horse_labels を生成する
  9. BaselineV1 と ExtendedV1 を同時に出力する
  10. 品質監査成果物を保存する
  11. encoding recipe ごとの交差検証での予測（OOF）特徴量を分離保存する
  12. calibration input（確率補正用の入力）を別出力する
  13. tracking_difficulty 出力を中間成果物として保存する
  14. pace_scenario 出力を中間成果物として保存する
  15. day_track_bias と contextual_bloodline を別成果物として保存する
  16. polars scan ベースの builder で fold 単位に materiailize する
