馬券推奨レイヤと強化学習 — 設計議論

ソース: docs/html/modeling/betting_policy_reinforcement_learning.html · 生成: 2026-05-20 06:01 UTC

馬券推奨レイヤと強化学習（報酬を最大化するよう行動を学習する手法） — 設計議論

1. このドキュメントの対象範囲

本ドキュメントは、予測システム全体における次の 2 つのレイヤの役割と境界 を明確にします。

レイヤ 役割 主な手法（想定）
確率モデル（Stage6 まで） レース直前情報のみで 勝率・連帯率・複勝率（p_win, p_top2, p_top3）を推定する 従来の 教師あり学習（GBDT / ランキング / NN 等）。docs/html/modeling/ の母表・split・OOF 方針が主。
馬券推奨レイヤ（本書） オッズ・モデル確率・予測印を統合し、期待値と制約のバランスを取りながら 推奨券種・組合せ・印の結びつけ を出力する 強化学習によるポリシー学習を第一候補とする。「もう一段の勾配ブースティング」ではない。
  • 期待値計算（model_prob × オッズ から求めるエッジ、safe_ev（安全な期待値の閾値）など）は推奨レイヤの 構成要素 ですが、それだけで完結させません。印との一貫性・券種間のトレードオフ・資金制約下での連続した意思決定を 強化学習で統合します。

関連ドキュメント: betting_backtest_dataset_spec.html（データ契約・safe_ev）、pre_race_feature_whitelist.html（LayerA/LayerB）、staged_probability_pipeline.html（Stage7）

2. なぜ推奨部分を「機械学習の延長」と切り分けるか

  1. 目的関数が違う
    確率モデルは 校正精度（LogLoss / Brier / ECE）を最適化します。馬券推奨レイヤは 長期的なリターン・最大損失の抑制・的中体験（印の説得力） など、複数の目標と制約のバランスを取る必要があります。

  2. 買える組み合わせが膨大で選択が難しい
    馬単・三連系の組み合わせ数は非常に大きく、単純に期待値が最大の選択肢を選ぶ方法では分散が極端になりやすいです。資金制約や同一レース内の相関も同時に扱う必要があります。

  3. オッズ情報（LayerB）が状態の中心になる
    オッズは他の参加者の動向に応じて変化する非定常な情報です。確率モデルの学習では除外していたオッズ情報が、推奨レイヤでは意思決定の核心的な入力となります。

  4. 予測印は「人間が読める中間表現」
    印（◎○▲△など）は UI と説明責任のための記号です。確率ベクトルから一貫した印を学習するよりも、ポリシーが印と券を同時に出力する方が自然に一貫性を保てます。

3. 行動サイクル（状態→行動→報酬のサイクル、強化学習の基本構造）の骨格（概念）

強化学習では「エージェント（推奨エンジン）が環境（レースと市場）を観測し、行動（券の選択）を行い、報酬（回収結果）を受け取る」サイクルを繰り返します。

3.1 エージェントと環境

  • エージェント: 馬券推奨エンジン（1 ユーザーまたは 1 ポリシー単位で構成可能）。
  • 環境: 開催日・レースの時系列（過去データを使った検証では履歴の再生、本番ではリアルタイムのデータ）。各ステップで「そのレースの市場スナップショット＋確率テーブル」が観測されます。

3.2 状態（State）の典型的な構成要素

カテゴリ主な要素
モデル側p_win（勝率）、p_top2（連帯率）、p_top3（複勝率）、校正後確率、不確実性の指標、ランキングスコア、多段 stage の中間出力（任意）
市場側win_odds（単勝オッズ）、place_odds_*（複勝オッズ）、組合せオッズの要約、人気順、オッズ変化率（取得できる場合）
文脈情報venue_code（競馬場）、surface（芝/ダート）、field_size（頭数）、重賞フラグ、残高・当日累積損益（連続する行動の記録に使う場合）
印候補ルールベースまたは別モデルで生成した印ラベル候補（◎○▲△など）を multi-hot または embedding 形式で入力

3.3 行動（Action）

組み合わせが膨大なため、階層的な設計が必須です。

  1. レベル1（券種選択）: 単勝のみ / 複勝まで / 馬連を含むなど — 少数の離散選択。
  2. レベル2（組み合わせ・金額）: 各券種で 上位 K 組 または 金額付き離散バケット（100円×{0,1,2,5}）を選ぶ。
  3. レベル3（印）: 「今回の推奨セットに含める印の集合」と「券との対応」— 同一ポリシーが出力することで一貫性が保ちやすくなる。

行動空間を絞ることが学習の成否を左右します。初期段階では、1 レース = 1 回の意思決定として扱う 状況適応型選択学習（Contextual Bandit） に近い設計から始めるのが現実的です。

3.4 報酬（Reward）

「そのレースの払戻 − 投資」だけを報酬にすると、的中のノイズが大きすぎて学習が安定しません。以下の候補を組み合わせます。

報酬種別内容注意点
短期報酬safe_ev で重み付けした実現エッジbetting_backtest_dataset_spec.html の定義と整合させる
リスク調整報酬return − λ * drawdown_penalty や Sharpe 比に近い形レース単位ではノイズが大きいため、開催単位で集約する
印の一貫性ペナルティ印と買い推奨が矛盾する場合に負の報酬を与えるルールベースで実装可
スパース問題への対策「買わない」判断にも小さな報酬を与えることを検討「何も買わない」退化に陥るリスクがあるため慎重に設計する

4. 「バランスの良い期待値」と強化学習の関係

  • 期待値は 行動のスコアリング関数 として使うのが自然です（Q 値や advantage の形で f(p, odds, stake) を計算する）。
  • バランスを取る手段は、報酬設計と制約 に落とし込みます。具体的には次のようなものです。
    • 高オッズ偏重を safe_ev の buffer で抑える（安全域の考え方を強化学習の報酬に組み込む）。
    • 券種ごとに 上限 stake（最大賭け金額）や Kelly 基準に基づくクリップ を環境側の制約として設定する。
  • 強化学習は期待値を「置き換える」のではなく、「方策の学習信号として統合する」 ものです。
    たとえば、ルールベースの期待値フィルタで候補を絞り、残った候補に対して方策勾配を適用する二段構えの設計でも機能します。

5. 予測印との結びつけ（商品設計）

5.1 印の役割

  • 説明のための表現: ユーザーが「なぜこの券を推奨するのか」を直感的に理解するための圧縮記号（◎○▲△）。
  • 出力次元の削減: 印の種類は少ないため、方策の出力空間を絞るのに利用できる。

5.2 実装パターン

パターン 内容 メリット / デメリット
A. 印を状態のみ 別モジュールが印候補を生成し、RL は券と金額のみ選択 印と券のズレが出やすい → ペナルティで調整
B. 印を行動の一部 行動 = (印セット, 券種, 組) の joint 一貫性は高いが学習難易度上昇
C. 階層 RL 高レベルが「印ストーリー」、低レベルが具体券 解釈性と探索の分離

推奨: 初期はパターン A + ペナルティ で実装し、データが積み上がったら パターン B へ移行します。

6. 学習パラダイムの比較

手法概要適した状況・メリットリスク・注意点
6.1 オンライン強化学習 実際の馬券購入でデータを集めながら探索する リアルタイムの適応が可能 金銭的コストと評判リスクが非常に大きい。本番環境では採用しない方針
6.2 過去データから学ぶ強化学習（Offline RL） (状態, 行動, 報酬, 次の状態) のログを過去データ検証またはシャドウ運用から蓄積して学習する CQL / IQL など、学習ポリシーとログ生成ポリシーのずれに強い手法が使える オッズは仮説シナリオとして扱う必要がある。確定オッズでの近似に留まると宣言しておく
6.3 状況適応型選択学習（Contextual Bandit） 1 レース = 1 回の意思決定とし、次のレースとは独立とみなす簡略モデル LinUCB / Thompson Sampling から始められる。実装が最もシンプルで強化学習への橋渡しとして最優先 レース間の依存関係（残高の変化など）を無視する近似
6.4 行動クローン（BC）＋微調整 human の判断または safe_ev ルールの行動を模倣し、その後小規模なオフライン強化学習で改善する ルールベースの行動が「教師ポリシー」の下限となり、極端に悪い行動を取りにくくなる 模倣元の品質に依存する。BC だけでは改善の上限がある

7. 過去データを使った検証での行動サイクル（バックテスト環境）

  • betting_backtest_dataset が 遷移のシミュレーター となります。
  • 1 ステップ = 1 レース（または 1 開催日）で、報酬は payout_table（払戻）と stake（賭け金）から計算します。
  • 状態の変化: 主に 残高 と 累積指標 が変化します（レース特徴は次サンプルとは独立とみなす近似も許容されます）。

7.1 評価指標の役割分担

確率モデルの評価指標と推奨レイヤの評価指標を混同しないことが重要です。

種別 指標例 注意
確率品質 LogLoss, ECE 確率モデル（Stage6）の責務。強化学習で改善しようとしない（役割の混同を禁止）。
戦略品質 ROI, maxDD, avg_safe_ev, selection_rate 強化学習ポリシーの責務。テスト期間のみで報告する。

8. リスクと対策（設計レビュー用チェックリスト）

リスク項目確認内容
データリーク強化学習の学習データに、確率モデルのテスト期間の予測が混入していないか確認する
オッズの因果問題仮説シナリオ（実際と異なる条件での評価）の前提を文書化しているか確認する
探索の安全性本番では確率ポリシー＋閾値によって行動をクリップしているか確認する
分布のドリフトモデルやオッズ分布の変化を検知し、ポリシーの再学習トリガーが設定されているか確認する
説明責任印・券・期待値の追跡ログを同一 run_id で保存しているか確認する
規制・倫理自動投票の可否は製品ポリシーで判断する（この技術文書では参照のみ）

9. 実装ロードマップ（提案）

  1. Phase 0（強化学習なし）
    safe_ev（安全な期待値の閾値）＋ 閾値 ＋ 券種別ルールで ベースラインの推奨 と 印の機械的マッピング を実装する。このとき強化学習で使えるログ形式に記録を整えておく。
  2. Phase 1（状況適応型選択学習：Contextual Bandit）
    候補券を有限個に絞り込み、過去ログから 方策を改善（LinUCB またはニューラル bandit）する。
  3. Phase 2（過去データから学ぶ強化学習：Offline RL）
    Q 関数または 制約付き方策最適化 を使い、開催単位の報酬を導入する。
  4. Phase 3（階層構造・印との joint 出力）
    十分なデータ量と安全性の担保が揃ってから着手する。

10. 成果物（契約）

成果物内容・要件
betting_policy_manifest.jsonポリシー種別（rule_baseline / bandit_v1 / offline_rl_v1）、参照する確率モデルのバージョン、オッズのスナップショット時刻を記録する（既存案を拡張）
バックテスト出力policy_action_id（ポリシーの行動 ID）と mark_binding（印と券の対応）を含める
MLflow Run方策学習・検証再現のジョブごとに Run を作成し、manifest の policy_version と mlflow_run_id を相互参照する。確率モデルとは別の experiment 名を使うことを推奨

11. まとめ

対象領域方針
勝率・連帯率・複勝率の推定従来の教師あり機械学習で進める
馬券推奨オッズ・確率・印・資金制約を統合した意思決定問題として扱い、強化学習（まず状況適応型選択学習、次に過去データ学習）で取り組む
期待値の位置づけ核心的な情報だがそれだけで完結させない。バランスは報酬設計・制約・階層的な行動設計で表現する
ログ整合性既存の safe_ev と過去データ検証の仕様にログのフォーマットを合わせることで、研究と製品両方での成功確率を高める
