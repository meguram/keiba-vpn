特徴量エンジニアリング — 探索メモ（新要素の発見）

ソース: docs/html/modeling/feature_engineering_exploration.html · 生成: 2026-05-20 06:01 UTC

特徴量の作成・加工（予測に使う数値を整える工程） — 探索メモ（新要素の発見）

目的

既存の BaselineV1 / ExtendedV1・horse_pre_race_dataset_spec.html・pre_race_feature_whitelist.html を前提として、追加で検証する価値のある特徴量の候補を列挙します。

記載する候補はすべて以下の条件を満たすものです。

  • LayerA（発走10分前の時点で観測可能な情報） を優先する
  • 結果ラベルからの情報漏えい（リーク）にならないものだけを対象とする（オッズ由来は LayerB として別管理）

実装時は src/pipeline/feature_store.py の列登録と data/features/_registry.json の方針に従い、OOF + purged CV（データリークを防いだ交差検証）でのみ寄与を測定します。

HTML 要約（ナビ用）: docs/html/feature_engineering/index.html

1. レース内「能力の分布」を数値化する（出走メンバー依存の特徴）

候補 定義イメージ データソース例
field_speed_spread 同一レース内の speed_max / 指数の 標準偏差・IQR race_index_flat, shutuba past
field_form_dispersion 近走好走率や着順の 分散（メンバーが揃いか荒れか） past flat
favorite_clustering 人気が集中しているか（分散が小さい＝堅い）※LayerB なら T-10 オッズ LayerB のみ
ability_entropy レース内の能力指標を rank 化し エントロピー（混戦度） 指数・past 集約

仮説: 混戦レースでは、絶対的な能力スコアよりも 出走メンバー内での相対順位特徴が予測力を持ちやすい。race_rank（着順）や z-score（標準化スコア）との 交互作用 を試す価値がある。

2. 仕上げのリズムを数値化する（馬・厩舎・騎手の時系列特徴）

候補 定義イメージ
days_since_last_same_surface 前走同馬場からの日数（芝/ダ別）
rotation_pressure 短期間での連闘回数（開催単位でのカウント）
trainer_recent_volatility 厩舎の直近 N 走の着順分散（同一 trainer_id）
jockey_venue_momentum 騎手×場の直近 K 走の勝率変化率（OOF TE と別軸の raw 統計）

仮説: 馬の仕上がり周期や陣営の調子の波は、距離帯と季節の組み合わせと交差することが多く、その交差項を特徴として使うと効果が出やすい。

3. コース・馬場の状態を特徴にする（venue_era 以外の軸）

候補 定義イメージ
rail_side_bucket 直線の内外 / 向きと枠の交差（取得できる場合のみ）
weather_transition 前日〜当日の天候変化カテゴリ（同開催内）
cushion_z_in_meet 同一開催内でクッション値の Z 化（Extended）
heavy_track_sire_rate 父系統の重・不良別成績（血統 flat から集約）

仮説: venue_era（競馬場の改修フラグ）と 馬場状態が極端な日（良馬場・重馬場の端の値）との交差項が、重賞レースの予測で有効になる可能性がある。

4. ペース・位置取りを事前に合成する特徴

候補 定義イメージ
expected_pace_pressure 全馬の逃げ・先行指標の和／最大（レース全体の前圧）
draw_bias_x_running_style 枠 × 脚質の one-hot 交差（過去統計で埋める）
late_charge_density 差し・追込馬の頭数と距離バンドの組

仮説: Stage2/3 のモデル出力がなくても、ルールベースの合成特徴で一部を近似できる。後段モデルへの橋渡しとして機能する可能性がある。

5. レースパフォーマンス信号を「文脈付き」で取り込む

race_performance（走破パフォーマンス）系は 別モジュールで管理します。LayerA に組み込む場合は、当日より前の履歴データのみを使います。

候補 定義イメージ
rp_final_ma3_same_distance_band 同馬の run_performance_final の移動平均（過去のみ）
rp_pct_vs_field_hist 過去レースでの run_performance_final_pct の平均
race_level_pre_race_ma_field 当該馬が過去に出たレースの race_level_pre_race 平均（対戦強度履歴）

仮説: 単発のパフォーマンス値より、距離帯やコースとの組み合わせで集計した値の方が安定した予測力を持つ。

6. データの欠損・カバレッジ自体を特徴にする（メタ特徴）

候補 定義イメージ
n_past_races_used past から参照できた本数
has_speed_index_any 指数欠損の反転フラグ
feature_coverage_score 既に betting 仕様にあるが LayerA でも 学習に入れ欠損パターンを学習させる

仮説: 2020年前半など 特定の年でデータ欠損が構造的に発生している場合、メタ特徴がモデルのドリフト検知に貢献する可能性がある。

7. 検証プロトコル（新要素を「有効」と判断できる条件）

  1. 同一の split_manifest_version（データ分割バージョン）を使い、Baseline との差は 追加する特徴群のみにする。
  2. 主要評価指標（例: valid logloss）が統計的に改善するか、複数のセグメントで一貫して改善することを確認する。
  3. 改善が確認されたら feature_dictionary.html と _registry.json に登録し、mlflow_run_id を紐付ける。
  4. 有効でなかった候補は本書に 却下理由 を 1 行追記する（「何が効かなかったか」の記録も重要な知見）。

8. 次のアクション（優先順位順）

  1. 出走メンバー内の分散系（セクション1）— 実装コストが低く、結果の解釈もしやすい。
  2. メタカバレッジ特徴（セクション6）— データ品質が変化した年との交互作用を確認する。
  3. race_performance の履歴特徴（セクション5）— 既存パイプラインとの join 設計が必要なため、後回しにする。

9. 実測メモ（自動探索・2026-04）

実行スクリプト: notebooks/feature_engineering/run_10_fe_cycles.py（10 サイクル）、run_extended_fe_analysis.py（拡張分析）
成果物: notebooks/feature_engineering/_run_output/cycles_report.html / extended_analysis.html

10 サイクル探索の要約

特徴量・グループ観測結果採用判断・推奨アクション
混戦度系（field_speed_std / field_speed_iqr / CV クリップ後） 相互の相関が Spearman ρ≈0.91–0.98（レース単位）と非常に高い。speed_max/distance のレース内 std も距離が一定のためほぼ同じ情報 1 種類を採用して ablation（除外実験）で確認するのが現実的
ability_entropy（粗いビン区切り） 欠損と飽和が多く、現在の定義では判別力が低い ビン数と最小頭数の設計を見直す必要がある
field_weight_std（斤量の分散） 指数系との重複が少なく使いやすい 採用候補として有効
trainer_top_share（厩舎の成績占有率） speed_pct と弱い正の相関があり、交絡（他の要因との混在）の疑いがある 単独採用は慎重に。交互作用項での利用を検討
field_bracket_std（枠番の分散）と trainer_top_share 両者の間に強い負の相関がある 主効果として両方を独立に入れるのは避け、交互作用項または合成した 1 つの特徴を使うことを推奨

拡張分析で浮上した新規候補（数値はローカル flat Parquet 時点）

候補 解釈
field_std_minus_iqr std−IQR。ρ(特徴, field_std)≈0.25 で「外れ値混戦」成分として残りやすい。
bracket_trader_interaction = bracket_std * (1 - trainer_share) 枠×厩舎の合成。ρ(特徴, field_std)≈0.21。
field_weight_cv 斤量のレース内変動係数。ρ(特徴, field_std)≈0.15 で指数系と最も直交に近い。
field_speed_gap_std speed_max−speed_avg のレース内 std。ρ(特徴, field_std)≈0.32 — 付加価値は限定的になりやすい。
field_std_surface_z 年×surface 内 z 化。ρ(特徴, field_std)≈0.92 — 依然として混戦度と強く重複。
field_speed_std_per_sqrt_n 頭数補正。ρ ほぼ 1 — 実質 field_speed_std の単調変換。
