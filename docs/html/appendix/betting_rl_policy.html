Appendix · 馬券推奨レイヤ

馬券推奨レイヤと強化学習

このページが伝えたいことは1つです。
「着順を予測すること」と「どの馬券を買うか決めること」は、まったく別の問題です。

着順予測は機械学習（ML）で解けます。しかし馬券選択は「予測確率・オッズ・手持ちの印」を 組み合わせて、リターンと損失リスクのバランスを取りながら連続して意思決定する問題です。 これは強化学習（RL）が得意とする問題構造であり、このシステムでも馬券推奨レイヤにRLを採用する方針です。

1. 「予測」と「馬券選択」は別の問題

システムを2つのレイヤに分けて考えます。

Layer A — 確率予測
着順・複勝の確率を出す
勝率・連対率・複勝率などを数値で予測。
GBDT（木を積み上げた機械学習モデル）やランキングモデルで学習する。
Layer B — 馬券推奨（本ページの主題）
どの馬券をいくら買うか決める
券種・金額・予測印を組み合わせて最終的な購入判断を出す。
長期的な収支のバランスを保ちながら選択する必要がある。

Layer Aの出力（予測確率）はLayer Bへの入力情報の一つに過ぎません。Layer Bでは「期待値が高いからといって全額賭ける」のではなく、連続負けのリスク・印の一貫性・資金管理を同時に考慮する必要があります。

2. なぜ馬券選択に強化学習（RL）を使うのか

理由 1

最適化の目標が複数ある

「期待値を最大化する」だけでは不十分です。長期リターン・連続負け幅の抑制・印との整合性など、複数の目標をバランスよく達成する必要があります。強化学習はこれを「報酬の設計」で表現できます。

理由 2

一度きりではなく「連続した」意思決定

1レースの結果が次のレースの判断に影響します（資金残高・調子など）。強化学習は複数ステップにまたがる意思決定の最適化が得意です。

理由 3

オッズは「状態」として扱える

オッズは着順予測とは独立した市場情報です。着順モデルの学習に混ぜるべきではなく、馬券選択時の状態情報として強化学習に自然に組み込めます。

理由 4

行動空間が大きい

単勝・複勝・馬連・ワイド…複数の券種を組み合わせると選択肢は膨大です。単純に「期待値が一番高いものを買う」では分散が大きすぎます。方策（ポリシー）として表現することで資金分配を賢く扱えます。

3. 設計の骨格（強化学習の構造）

強化学習は「状態 → 行動 → 報酬」のサイクルで学習します。競馬に当てはめると以下のようになります。

強化学習の概念競馬馬券での意味
状態 予測確率（勝率・複勝率）、現在のオッズ、予測印の候補、コース・距離・馬場状態
行動 買う券種と組み合わせ、投資金額のバケット（少額・中額・見送りなど）
報酬 期待値ベースの加重リターン × 連続負け幅ペナルティ × 印との一貫性ボーナス
エピソード 1レース or 1開催日（状況に応じて設定）
なぜ「払戻額だけ」を報酬にしないのか？
払戻額をそのまま報酬にすると、ノイズ（運）が大きすぎて学習が安定しません。 代わりに「期待値（safe_ev）」を核にしながら、損失幅ペナルティや印の整合性を組み合わせた複合報酬を設計します。

4. 予測印（◎○▲）との関係

印（◎本命・○対抗・▲穴）は「この馬を強く推す」という判断の表現です。馬券推奨と印の整合性を保つには3つのパターンが考えられます。

パターン説明採用しやすさ
A：印と券を分離 印は別モジュールが生成し、RLは券種・金額だけを決める。印と券のズレは報酬のペナルティで抑える。 導入しやすい
B：印も行動に含める 「印セット＋券種＋組み合わせ」をまとめて1つの行動として学習する。 一貫性は高い 探索空間が大きい
C：階層的に分ける 上位ポリシーが「印ストーリー（軸馬を決める）」、下位ポリシーが「具体的な券種」を決める2段構造。 データが多くなってから

初期実装ではパターンAが最も現実的です。

5. どうやって学習させるか

本番環境で試行錯誤しながら学習する「オンラインRL」は、資金リスクが高すぎるため採用しません。代わりに過去データを使います。

フェーズ1（最初の実装）

Contextual Bandit
状況に応じた選択学習

1レース = 1回の選択として単純化。候補馬券を事前に絞り込んで有限化すれば実装しやすい。

フェーズ2（本格実装）

Offline RL
過去データから学ぶ強化学習

過去の（状態・行動・報酬）のログから学習する。実際には買わなかった馬券の仮の結果を仮定する「反事実推論」が必要になる。

学習パラダイム特徴採用フェーズ採用しやすさ
オンラインRL 本番環境でリアルタイムに試行錯誤しながら学習する —（不採用） 資金リスクが高すぎるため採用しない
Contextual Bandit 1レース = 1回の選択として単純化。候補馬券を事前に絞り込んで有限化する フェーズ1 実装しやすい・小さくスタートできる
Offline RL（CQL / IQL等） 過去の（状態・行動・報酬）ログから学習する。反事実推論が必要 フェーズ2 過去ログが蓄積されてから着手
階層的RL 上位ポリシーが印ストーリー、下位ポリシーが券種を決める2段構造 フェーズ3（印との統合） データとログが十分に蓄積されてから

6. 実装ロードマップ

  1. Phase 0 — ルールベースの整備
    safe_ev（期待値の安全基準）と印のルールマッピングを確立する。ログを（状態・行動・報酬）の形式で記録できるよう整備する。
  2. Phase 1 — Contextual Bandit で最初のプロトタイプ
    1レース1選択に単純化して小さくスタート。ルールEVで候補を絞ってから方策で選ぶ「二段構え」でもよい。
  3. Phase 2 — Offline RL でセッション学習
    複数レースにまたがる連続判断を学習。CQL / IQL などのアルゴリズムで過去ログから改善する。
  4. Phase 3 — 印との統合（階層的RL）
    印付けと馬券選択を統一した学習へ発展させる。データとログが十分に蓄積されてから着手。

7. 実装上の注意点

注意点理由・対策
着順モデルのテスト予測を学習データに混ぜない Layer Aの汚染防止。強化学習（Layer B）のデータと学習スコープを厳密に分離する
方策の出力にハードクリップをかける 1レースの最大投資額・禁止券種リストを本番に必ず適用する
印・券・期待値を同じ run_id でトレース可能にする ログを一元管理し、後から判断の根拠を検証できるようにする
反事実オッズの近似の仮定範囲を明記する 「あの券を買っていたら払戻はいくらか」の仮定が学習の前提となるため、仮定の範囲と限界を文書化する
