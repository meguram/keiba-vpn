1. keiba-vpnの現在のデータ保存構成

scraper/storage.py の HybridStorage は、原則としてJSONオブジェクトをGoogle Cloud Storage（GCS）に保存します（category + key で管理）。race_lists など一部はローカルのみで扱います。

カテゴリの例:

分類カテゴリキー内容
レース系 race_shutuba 出馬表
race_result レース結果
race_odds オッズ
smartrc_race SmartRC レースデータ
race_result_lap ラップタイム
馬系 horse_result 馬の戦績
horse_training 調教データ
horse_pedigree_5gen 5代血統
その他 race_predictions AI予測結果

API・モニタリング画面は多数のキーを並列で読み出します（storage.load(category, id)）。欠損検知・学習用特徴量はローカルParquet層（scraper/local_tables.py）と組み合わせて使います。

2. コスト試算の前提条件

データ量の前提

GCS相当のデータ保存場所に合計500GiB（標準ストレージ）を保存する想定です。
（実際の総量は計測が必要です）

本番読み出しの前提

1回1オブジェクトのGET操作を10,000回/日実行すると想定（1回あたり約500KiB、1日約5GiBの転送）。月換算では30万GET・約150GiBの転送となります。

予測・学習（AIワークロード）の前提

月1回の日次/週次バッチ学習で2TiB相当をフルスキャンするクエリを仮定しています（特徴量再計算・広い期間の検証など）。オンライン推論は「数十ミリ秒以内に少数の特徴量を取得」するケースが多い想定です（キャッシュ併用）。

4. 各サービスの価格の考え方

Google Cloud Storage（単一リージョン・Standardクラス）

公式価格（東京を含む単一リージョン）: $0.000027397/GiB時間（Storage pricing | Google Cloud）。

月額換算（730時間/月として計算）:
単価 ≈ 0.000027397 × 730 ≈ $0.020/GiB/月

試算: 500GiB × $0.020 ≈ $10.0/月（保存コストのみ）。Class A/Bのオペレーション料金は別途計算が必要ですが、月30万GETでも保存コストより小さいケースが多いです。同一リージョン内のEgress無料帯の条件は要確認です。

Cloud SQL（PostgreSQLなど）— 小規模本番の「固定費」

CPU・メモリ・ディスクの常時起動分が費用の大半を占めます。1vCPU級+小ディスクで月$50〜$200程度（リージョン・バージョンで変動）。1万GET/日程度のアクセスはインスタンス費用に比べると小さいです。

BigQuery（オンデマンド分析）

公式: BigQuery pricing | Google Cloud — オンデマンドは処理したデータ量（TiB単位）あたりの課金です。公示例: $6.25/TiB（クエリ処理）。さらに毎月最初の1TiBはクエリ処理が無料です（「On-demand (per TiB)」節を参照）。

試算: 月に2TiBスキャンを1回行う場合、無料枠1TiBを差し引くと (2 − 1) × $6.25 ≈ $6.25/月。無料枠内に収まる月は$0に近いこともあります。列・パーティションを最適化してスキャン量を減らすのが定石です。

5. GCP内の構成パターンと特性

構成パターンレスポンス遅延主なコスト要因本プロジェクトへの適合度
GCS + 現行アプリ GETで数十ms程度 保存量＋転送費。小〜中規模では比較的安価 現行構成と整合 ◎。大規模な学習は帯域・I/O設計が重要
Cloud SQL 中心 低遅延にしやすい インスタンスの固定費が支配的。大きなJSONは正規化・設計が必要 行指向メタ・索引中心なら ◯。JSONをそのまま大量格納するには非効率
BigQuery 中心 高め（リアルタイム点読には不向き） 大量スキャンには ◯。少数行のGETには割高・遅延が大きい 集計・特徴量バッチ処理に ◎。本番のリアルタイム読み出しには不向き
GCS + BigQuery + Cloud SQL の組み合わせ 用途別に最適化 役割を分離することで最もバランスが取れる 実務でよく見られる構成。データの同期コストに注意が必要

6. GCP以外のサービス（マネージドDB / BaaS）

「GCPとは別のサービス」とは、AWS（RDS / S3 / Redshift）・マルチクラウドDB（Atlas / Supabase / Neon）・SaaS分析などを指します。データ転送費（Egress）とプロバイダをまたぐ通信がコスト・遅延に大きく影響します。

サービス / 主な用途料金の仕組み（要公式確認）試算メモ
Amazon S3（Standard） GB/月 + リクエスト回数 + Egress（外部転送費） us-east-1で約$0.023/GB/月（S3料金）。500GBで月$11〜$12程度。GET操作の費用は規模によっては小額に収まることが多い。
Amazon RDS（MySQLなど） DBインスタンス時間 + ストレージ + バックアップ db.t3.microで月$20〜$30程度から（RDS料金）。Cloud SQL同様、固定費が中心。
Supabase（Proプラン） 月額定額 + 超過課金 $25/月〜 + 超過分（supabase.com/pricing で確認要）。PostgreSQL一体型。外部転送量が多い場合は超過課金に注意。
Neon（Serverless Postgres） CU使用時間 + ストレージ ストレージは約$0.35/GB/月程度（neon.tech/pricing）。低負荷でスケールゼロになる環境向き。ただし500GBではストレージ代だけで$175程度になり、全データをRDBに移すと高コストになりやすい。
MongoDB Atlas（M10クラスタ目安） クラスタ時間 + ディスク AWSリージョン・クラスタサイズで大きく変わります（料金ページで要計算）。2GB RAM級のM10で月$50程度から。
Snowflake / その他DW クレジット制 + ストレージ 大規模分析向け。keiba-vpn規模では最小枠でどの程度になるか見積もりが必須。

8. 予測AIシステムとしての観点

ワークロード推奨アーキテクチャ本プロジェクトとの関係
学習・大規模特徴量の再計算 列指向のParquet/外部テーブル + 列指向DW/Spark 行指向RDBより転送効率が良い。export_tables の設計思想と整合しやすい
オンライン推論（リアルタイム予測） メモリキャッシュ（Redis等）+ OLTP/キーバリューストア、または事前に計算済みの少数列を参照 低遅延が求められる。キャッシュ層の設計が重要
推奨構成（3段構成） 正本データ: オブジェクトストア（GCS/S3）、メタ情報・索引: SQL/検索DB、学習: DW/バッチ処理 実務でよく見られる構成。役割の分離により管理しやすい
